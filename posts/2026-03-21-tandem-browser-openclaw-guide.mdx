# Tandem Browser 도입기

이번 글은 **Tandem Browser를 OpenClaw 환경에 도입한 이유와 실제 설치/운영 방식**을 정리한 기록입니다.

핵심은 단순합니다.

기존의 웹 자동화나 페이지 읽기 도구만으로는 해결이 잘 안 되는 작업들이 있습니다.
예를 들면 다음과 같습니다.

- 로그인된 실제 웹 세션이 필요한 작업
- SPA처럼 구조가 복잡한 웹앱 분석
- 사람이 현재 보고 있는 탭 문맥을 AI가 같이 이해해야 하는 작업
- 사람이 인증이나 캡차를 처리하고, 이후 흐름만 AI가 이어받는 작업

이런 경우에는 단순한 HTML 추출 도구보다,
**사람이 쓰는 실제 브라우저와 AI가 협업할 수 있는 계층**이 필요합니다.

Tandem Browser는 바로 그 지점을 노리는 도구입니다.

이 글은 단순 도입 메모가 아니라,
**왜 도입했는지, 어떤 상황에서 유용한지, 어떻게 설치했고 어떻게 운영할지까지** 누구나 이해할 수 있게 풀어쓴 가이드입니다.

참고로 원문 기록에 포함된 개인 이름은 모두 **User** 로 일반화해 정리했습니다.

---

# Tandem Browser가 필요한 이유

웹 작업 도구를 크게 나누면 대략 세 단계가 있습니다.

## 1. 단순 웹 읽기 도구
예:
- `web_fetch`
- `web_search`

이런 도구는 다음에 잘 맞습니다.

- 공개 웹페이지 읽기
- 검색 결과 요약
- 문서/기사/블로그 콘텐츠 확인
- 로그인 없는 저복잡도 작업

즉 읽는 작업에는 매우 효율적입니다.

---

## 2. 일반적인 브라우저 자동화
예:
- Playwright
- Puppeteer 계열

이런 도구는 클릭, 입력, 이동, 다운로드 같은 자동화에는 강합니다.
하지만 실제 사용자의 현재 브라우저 상태와 긴밀하게 협업하는 흐름에는 한계가 있습니다.

예를 들어:
- 이미 로그인된 세션을 자연스럽게 활용해야 하거나
- 사람이 보고 있는 현재 탭을 그대로 함께 이해해야 하거나
- 복잡한 SPA 상태를 조사해야 할 때는
단순 자동화만으로는 부족할 수 있습니다.

---

## 3. 실브라우저 협업 계층
Tandem Browser는 이 세 번째 층에 가깝습니다.

즉,
**사람이 실제로 쓰는 브라우저와 AI가 같은 로컬 머신에서 함께 웹을 다루도록 돕는 OpenClaw-first 브라우저**에 가깝습니다.

이게 중요한 이유는,
웹은 점점 더 단순 문서가 아니라 앱처럼 동작하기 때문입니다.

- 로그인 세션
- 동적 렌더링
- 클라이언트 상태
- XHR/Fetch 네트워크 요청
- 현재 선택된 탭 문맥

이런 것들은 일반 fetch 도구만으로는 충분히 다루기 어렵습니다.

---

# 어떤 상황에서 Tandem Browser를 써야 하나

도입 전에 먼저 역할 구분을 해두는 것이 중요합니다.

## 기존 도구로 충분한 경우
다음 상황은 굳이 Tandem Browser까지 필요하지 않습니다.

- 공개 문서나 기사 읽기
- 단순 검색
- 로그인 없는 페이지 접근
- HTML 텍스트 추출 중심 작업

이 경우는 여전히 `web_fetch` 나 `web_search` 류가 더 빠르고 가볍습니다.

---

## Tandem Browser를 써야 하는 경우
반대로 아래 상황은 Tandem Browser 도입 가치가 큽니다.

- 실제 로그인 세션이 필요한 작업
- 사람이 보고 있는 현재 브라우저 탭을 기준으로 보조해야 하는 작업
- SPA 구조 분석
- 네트워크/XHR 흐름 확인
- 사람이 인증/캡차만 처리하고, 이후 흐름을 AI가 이어받는 작업

즉 Tandem Browser는 기존 도구를 대체하는 것이 아니라,
**브라우저 작업의 상위 옵션**으로 두는 것이 가장 자연스럽습니다.

---

# 도입 목표를 한 문장으로 정리하면

이번 도입의 목적은 이렇습니다.

**OpenClaw 환경에 Tandem Browser를 붙여, 로그인된 실제 웹 세션과 현재 탭 문맥을 AI가 함께 이해하고 보조할 수 있는 실브라우저 협업 계층을 만든다.**

이 문장을 먼저 정해두면,
도입 이후 운영 원칙도 훨씬 선명해집니다.

# 실제 설치 구조

실제 구성은 아래처럼 잡았습니다.

## 코드 위치
- `external/tandem-browser`

즉, 메인 워크스페이스 안에 외부 프로젝트 형태로 가져와 관리했습니다.

## OpenClaw 로컬 skill 설치
- `skills/tandem-browser/SKILL.md`

공식 저장소의 skill 문서를 로컬 skill로 붙여,
OpenClaw가 Tandem Browser 사용법을 일관되게 따를 수 있도록 구성했습니다.

이렇게 하면 나중에 사람이 직접 다루든,
AI 에이전트가 활용하든 같은 규칙으로 접근하기 쉬워집니다.

# 의존성 설치

설치 자체는 비교적 단순합니다.

```bash
cd /Users/jamie/.openclaw/workspace/external/tandem-browser
npm install
```

Electron 기반 프로젝트는 설치 후 추가 빌드나 rebuild가 필요한 경우가 있는데,
이번 구성에서는 Electron rebuild까지 정상 완료되는 것을 확인했습니다.

즉 설치 단계에서의 목표는 두 가지입니다.

1. 의존성 설치가 끝나는지
2. Electron 앱이 실제 실행 가능한 상태인지

---

# 실행 확인

설치가 끝났다면 바로 실행을 확인해야 합니다.

```bash
cd /Users/jamie/.openclaw/workspace/external/tandem-browser
npm start
```

실행 후 확인 포인트는 다음과 같습니다.

- Tandem Browser 앱이 실제로 뜨는지
- 로컬 API가 열리는지
- 상태 확인 엔드포인트가 응답하는지
- API 토큰이 생성되는지

실제 확인 기준은 대략 아래와 같습니다.

- 로컬 API 주소: `http://127.0.0.1:8765`
- 상태 확인: `/status`
- API 토큰 파일: `~/.tandem/api-token`

이 단계에서 중요한 건,
“앱이 실행된다”와 “OpenClaw가 붙을 수 있는 API가 열린다”를 구분해서 확인하는 것입니다.

# 핵심 연결 정보 정리

실사용 관점에서는 아래 세 가지를 기억하면 됩니다.

## 1. API 주소
```text
http://127.0.0.1:8765
```

## 2. API 토큰 파일
```text
~/.tandem/api-token
```

## 3. 상태 확인 예시
```bash
TOKEN="$(cat ~/.tandem/api-token)"

curl -sS http://127.0.0.1:8765/status
curl -sS http://127.0.0.1:8765/tabs/list \
  -H "Authorization: Bearer $TOKEN"
```

이 세 가지가 확보되면,
“브라우저가 켜졌다” 수준을 넘어
**실제로 외부 도구(OpenClaw)가 현재 브라우저 상태를 읽고 협업할 준비가 된 상태**라고 볼 수 있습니다.

# OpenClaw 관점에서의 사용 원칙

실제 도입에서 중요한 건 설치보다 운영 원칙입니다.

Tandem Browser를 어떻게 쓸지 먼저 정의해야,
나중에 도구가 섞여도 작업 흐름이 안정적입니다.

## 기본 원칙
도입 후 OpenClaw 쪽 사용 원칙은 아래처럼 잡을 수 있습니다.

- 새 URL 작업은 무조건 단순 이동부터 하지 않고, 필요하면 새 탭 생성으로 시작한다
- snapshot/ref 같은 현재 상태 참조 체계를 우선 활용한다
- SPA는 네트워크/XHR, 자바스크립트 실행, DOM 텍스트 추출을 조합해 본다
- 로그인, 캡차, 애매한 인증 단계는 사람이 처리하고 이후를 AI가 보조한다

이 원칙이 중요한 이유는,
실브라우저 협업은 자동화보다 **문맥 공유**가 핵심이기 때문입니다.

즉 “브라우저를 대신 움직이는 것”보다,
**현재 브라우저 상태를 함께 이해하고 이어받는 것**에 초점을 두는 편이 더 잘 맞습니다.

---

# 기대 효과는 무엇인가

이 계층을 붙이면 기대할 수 있는 효과는 꽤 명확합니다.

## 1. 사람이 보는 페이지를 AI도 함께 이해하기 쉬워짐
기존에는 사용자가 “지금 이 페이지 보고 있어”라고 설명해야 했던 부분을,
브라우저 상태 공유를 통해 더 직접적으로 파악할 수 있습니다.

## 2. 웹앱 조사 품질 향상
단순 HTML만 보는 게 아니라,
현재 탭 상태나 네트워크 요청, SPA 동작 흐름까지 같이 볼 수 있으면 분석 품질이 올라갑니다.

## 3. 로그인 기반 작업 지원
일반 fetch 도구로는 안 되는,
실제 로그인 세션 기반 지원 작업이 가능해집니다.

## 4. 사람-에이전트 협업에 더 적합
자동화 도구는 종종 사람을 완전히 배제하려는 흐름으로 가는데,
실제 업무는 오히려 사람이 인증/판단을 하고 AI가 나머지를 보조하는 구조가 더 잘 맞는 경우가 많습니다.

Tandem Browser는 이 협업 패턴에 더 어울립니다.

# 자동 실행 등록

실사용하려면 수동 실행만으로는 부족할 수 있습니다.
그래서 리붓 후에도 자동으로 올라오도록 launchd 등록까지 붙이는 방법이 자연스럽습니다.

예를 들어 다음 요소를 둡니다.

## LaunchAgent
- `~/Library/LaunchAgents/com.example.tandem-browser.plist`

## 시작 스크립트
- 워크스페이스 내 별도 start script

## 로그 파일
- stdout 로그
- stderr 로그

이렇게 구성하면,
문제가 생겼을 때도 로그를 기준으로 다시 진단하기 쉬워집니다.

자동실행의 핵심은 단순히 “부팅 시 켜진다”가 아니라,
**재현 가능한 시작 방법과 추적 가능한 로그를 남기는 것**입니다.

---

# 현재 도입 수준을 어떻게 판단할까

도입이 끝났는지 여부는 아래처럼 나눠 보면 좋습니다.

## 완료된 것
- 저장소 클론
- `npm install`
- 실제 실행 성공
- 로컬 API 확인
- API 토큰 생성 확인
- OpenClaw local skill 설치
- 자동실행 등록

## 앞으로 남는 것
- 실제 업무 흐름에서 Tandem Browser를 적극 활용해보기
- 반복 패턴을 운영 규칙으로 정리하기
- 자주 쓰는 웹 협업 시나리오를 메모리/문서로 축적하기

즉 도입은 “설치 완료”로 끝나지 않고,
**어떤 상황에서 이 도구를 꺼내 쓸지 패턴을 정착시키는 단계까지 가야 진짜 완성**입니다.

# 이 도구를 어떻게 위치시켜야 하나

운영 관점에서는 Tandem Browser를 아래처럼 이해하는 편이 좋습니다.

- 기본 웹 읽기 도구의 대체재는 아님
- 로그인/SPA/실브라우저 협업이 필요한 고급 웹 작업용 계층
- 사람이 보고 있는 탭을 AI가 함께 이해하고 보조하는 협업 도구

즉 모든 웹 작업에 쓰는 기본 도구가 아니라,
**필요할 때 꺼내는 상위 옵션**으로 두는 것이 가장 현실적입니다.

---

# 따라하기용 요약

이 글만 보고 다시 구성하고 싶다면 순서는 아래처럼 잡으면 됩니다.

1. Tandem Browser 저장소를 워크스페이스에 클론한다
2. `npm install` 로 의존성을 설치한다
3. `npm start` 로 실제 앱 기동을 확인한다
4. `http://127.0.0.1:8765/status` 로 로컬 API를 확인한다
5. `~/.tandem/api-token` 생성 여부를 확인한다
6. OpenClaw local skill을 설치해 사용 원칙을 문서화한다
7. launchd로 자동실행을 붙인다
8. stdout/stderr 로그 경로를 정리한다
9. 실사용 시 로그인/캡차는 사람이 맡고, 이후 흐름은 AI가 보조하도록 운영한다

이 순서대로 가면,
단순 설치가 아니라 실제 협업 가능한 실브라우저 계층으로 자리 잡기 쉽습니다.

# 개인적 판단

개인적으로 Tandem Browser의 가장 큰 가치는 “자동화 브라우저 하나 더 생김”이 아닙니다.

진짜 가치는,
**사람이 이미 사용 중인 브라우저 문맥을 AI가 더 실제적으로 이해할 수 있게 해준다는 점**입니다.

특히 다음 같은 작업에서는 가치가 큽니다.

- 로그인된 페이지 분석
- 복잡한 웹앱 조사
- 현재 열린 탭을 기준으로 한 보조 작업
- 사람이 인증하고 AI가 이어받는 흐름

반대로 공개 문서 읽기나 단순 검색만 할 거라면,
굳이 Tandem Browser까지 쓸 필요는 없습니다.

즉 이 도구는 범용 기본 도구가 아니라,
**필요할 때 꺼내는 고급 협업 브라우저 계층**이라고 보는 편이 정확합니다.

---

# 마무리

정리하면 이번 도입의 핵심은 한 문장입니다.

**Tandem Browser를 OpenClaw 환경에 붙여, 로그인 세션과 현재 탭 문맥을 포함한 실브라우저 협업 계층을 구축했다.**

이 구조는 단순 웹 자동화보다 더 실제적이고,
단순 페이지 읽기보다 더 깊이 있으며,
사람과 AI가 같은 웹 문맥을 공유하는 쪽에 더 가깝습니다.

로그인 기반 웹 작업, SPA 분석, 실제 브라우저 협업이 필요한 환경이라면,
Tandem Browser는 꽤 가치 있는 확장 계층이 될 수 있습니다.
