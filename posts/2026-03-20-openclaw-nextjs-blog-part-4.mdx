# 초보가 OpenClaw로 Next.js 블로그 만들기 Part 4

Part 3에서는 다음과 같은 문제들을 다루었습니다.

- 텔레그램 글 잘림 사건
- MDX 콘텐츠 관리 방식 변경
- 카테고리 필터 버그
- Next.js 캐싱 문제

이 문제들을 해결하면서 블로그 구조는 점점 안정되기 시작했습니다.

하지만 또 다른 큰 과제가 남아 있었습니다.

배포 시스템

즉 코드가 실제 웹사이트로 어떻게 연결되는지에 대한 문제였습니다.

---

# Next.js 배포 방식

Next.js 프로젝트는 여러 방식으로 배포할 수 있습니다.

대표적인 방법은 다음과 같습니다.

- 자체 서버 배포
- Docker 배포
- Vercel 배포

이 중에서 가장 널리 사용되는 방법은 **Vercel**입니다.

---

# 왜 Vercel을 선택했는가

Vercel은 Next.js를 만든 회사에서 운영하는 배포 플랫폼입니다.

그래서 Next.js와 매우 잘 통합됩니다.

특히 다음 기능이 매우 편리합니다.

- GitHub 자동 배포
- Preview 배포
- 빠른 CDN
- 무료 플랜 제공

블로그 프로젝트에는 매우 적합한 환경입니다.

---

# 자동 배포 구조

Vercel을 GitHub 저장소와 연결하면 다음 과정이 자동으로 실행됩니다.

코드를 GitHub에 push  
→ Vercel이 자동으로 빌드  
→ 웹사이트 배포

즉 별도의 서버 관리 없이도 웹사이트가 운영됩니다.

---

# 첫 번째 배포

GitHub 저장소를 Vercel에 연결한 후 첫 배포를 진행했습니다.

과정은 매우 간단했습니다.

1. Vercel에 로그인
2. GitHub 저장소 연결
3. Deploy 버튼 클릭

몇 분 후 블로그가 실제 인터넷에 공개되었습니다.

---

# 하지만 또 문제가 발생했다

배포는 성공했지만 예상하지 못한 문제가 나타났습니다.

로컬에서는 정상 작동하던 기능이  
배포된 사이트에서는 작동하지 않았습니다.

대표적인 문제가 다음과 같았습니다.

- 카테고리 필터 미작동
- 일부 포스트 렌더링 오류
- 빌드 경고 메시지

---

# Preview와 Production의 차이

Vercel에는 두 가지 배포 환경이 있습니다.

Preview  
Production

Preview는 테스트 환경입니다.

GitHub에서 PR이 생성되면 자동으로 Preview 배포가 만들어집니다.

Production은 실제 서비스 환경입니다.

---

# 문제 분석

문제를 분석해 보니 다음과 같은 상황이었습니다.

로컬 환경에서는 정상 작동

Preview에서는 일부 기능 오류

Production에서는 또 다른 오류

즉 환경마다 결과가 달랐습니다.

---

# 왜 이런 문제가 발생하는가

이 문제의 주요 원인은 다음과 같습니다.

환경 변수 차이

빌드 캐싱

의존성 버전 차이

Next.js 캐싱

특히 Next.js에서는 캐싱 문제가 자주 발생합니다.

---

# 실제로 발생했던 문제

예를 들어 다음과 같은 상황이 있었습니다.

카테고리를 클릭하면 URL은 변경되었습니다.

예를 들어

`/posts?category=Foodie`

하지만 페이지 내용은 변경되지 않았습니다.

즉 필터가 적용되지 않았습니다.

---

# 문제 해결 과정

문제를 해결하기 위해 다음 과정을 진행했습니다.

1. 로컬 코드 확인
2. GitHub 커밋 확인
3. Vercel 빌드 로그 확인
4. 캐시 초기화

이 과정에서 중요한 도구가 하나 있었습니다.

OpenClaw

---

# OpenClaw를 통한 분석

OpenClaw에게 다음과 같은 요청을 했습니다.

> "카테고리 필터가 production에서 작동하지 않는 이유를 분석해 달라."

OpenClaw는 다음 사항을 확인했습니다.

- `searchParams` 처리 방식
- Next.js 캐싱 정책
- 서버 렌더링 여부

그리고 해결 방법을 제안했습니다.

---

# 해결 방법

문제의 원인은 Next.js의 정적 렌더링이었습니다.

페이지가 정적으로 생성되면서  
URL 쿼리가 반영되지 않았습니다.

그래서 페이지를 동적 렌더링 모드로 변경했습니다.

이 변경 이후 카테고리 필터가 정상 작동했습니다.

---

# GitHub PR 관리

프로젝트가 커지면서 또 하나 중요한 개념이 등장했습니다.

Pull Request

PR은 다음과 같은 역할을 합니다.

- 코드 변경 검토
- 변경 이력 관리
- 테스트 환경 생성

---

# PR 기반 개발 방식

PR 기반 개발 방식은 다음과 같은 흐름입니다.

새 브랜치 생성  
→ 코드 수정  
→ PR 생성  
→ Preview 테스트  
→ `main` 브랜치 병합

이 방식은 매우 안정적입니다.

---

# 실제로 PR이 중요한 이유

PR을 사용하면 다음 장점이 있습니다.

- 배포 전 테스트 가능
- 코드 변경 기록 유지
- 오류 발생 시 롤백 가능

특히 Vercel Preview와 결합하면 매우 강력합니다.

---

# Preview 환경의 장점

Preview 배포는 실제 서비스에 영향을 주지 않습니다.

즉 다음과 같은 테스트가 가능합니다.

- 새 기능 테스트
- 디자인 수정 확인
- 콘텐츠 확인

모든 테스트가 완료된 후에만 Production으로 배포할 수 있습니다.

---

# 개발 방식의 변화

이 프로젝트를 진행하면서 개발 방식이 크게 바뀌었습니다.

초기에는 다음 방식이었습니다.

코드 수정  
→ 바로 배포

하지만 나중에는 다음 방식으로 변경되었습니다.

브랜치 생성  
→ PR 생성  
→ Preview 테스트  
→ Production 배포

이 방식이 훨씬 안정적이었습니다.

---

# 또 하나의 중요한 도구

이 시점에서 새로운 도구가 등장합니다.

Working Copy

Working Copy는 모바일 Git 클라이언트입니다.

즉 스마트폰에서도 Git 작업이 가능합니다.

---

# 모바일 포스팅의 필요성

블로그를 운영하다 보면 다음 상황이 자주 발생합니다.

- 외부에서 글 작성
- 여행 중 포스팅
- 빠른 수정 필요

이때 컴퓨터가 없으면 작업이 어렵습니다.

그래서 모바일 환경이 필요했습니다.

---

# Working Copy의 역할

Working Copy를 사용하면 다음 작업이 가능합니다.

- Git clone
- 파일 수정
- commit
- push

즉 스마트폰에서도 블로그 콘텐츠를 관리할 수 있습니다.

---

# 프로젝트의 다음 단계

이제 블로그 프로젝트는 다음 단계로 넘어가게 됩니다.

- 모바일 포스팅 시스템 구축
- SEO 구조 개선
- RSS 피드 생성
- 블로그 자동화

특히 모바일에서 글을 작성하는 방식이 크게 발전하게 됩니다.
