# AI와 대화하며 Travel Planner를 만든 개발기

이번에는 내가 실제로 쓰고 싶어서 만든 **Travel Planner** 개발 과정을 정리해보려고 한다.

처음 생각은 단순했다.

여행지를 정하면 그 지역 정보를 AI가 조사해주고, 며칠 일정인지 설정한 다음, 방문 장소를 순서대로 넣으면 지도에 경로가 나오는 앱을 만들고 싶었다.

그런데 막상 만들기 시작하니 필요한 기능은 계속 늘어났다.

- 날짜별 세부 일정
- 장소 카테고리와 할 일
- 자동차·도보·자전거 이동 경로와 예상 시간
- 경비와 결제 여부
- 준비물 체크리스트
- 로그인과 사용자별 데이터
- AI 목적지 조사
- 모바일 화면
- 오프라인 지도
- 네이버 지도와 Google Maps에서 찾은 장소 가져오기

결국 이 프로젝트는 단순한 일정표가 아니라, **여행 준비부터 현지 이동까지 한곳에서 관리하는 개인용 여행 운영 도구**가 됐다.

그리고 이 앱은 한 번의 프롬프트로 완성되지 않았다.

내가 직접 사용해보고 불편한 점을 발견하면 AI에게 다시 설명하고, 수정된 결과를 확인한 뒤 또 다음 문제를 찾는 과정을 수없이 반복했다. 이번 글은 완성된 기능 자랑보다, 그 과정에서 무엇이 실제로 어려웠고 어떻게 풀었는지를 중심으로 적어본다.

## 왜 새로운 여행 플래너를 만들고 싶었나

여행을 준비할 때 나는 여러 앱을 동시에 사용한다.

지도에서 장소를 찾고, 메모 앱에 일정을 적고, 예상 경비는 계산기에 넣고, 준비물은 또 다른 체크리스트에 기록한다. 지역의 날씨나 교통패스는 검색하거나 AI에게 따로 물어본다.

정보는 많은데 서로 연결되지 않는다.

예를 들어 식당을 하나 추가해도 다음 장소까지 자동차로 몇 분인지 다시 지도에서 확인해야 한다. 방문 순서를 바꾸면 경로와 시간도 다시 계산해야 한다. 가족과 함께 움직이는 여행에서는 예약 확인, 준비물, 결제 예정 금액까지 같이 봐야 하는데 기존 여행 앱은 어느 한 부분만 잘하는 경우가 많았다.

그래서 목표를 이렇게 정했다.

**여행지 조사, 일정, 경로, 경비, 준비물을 하나의 여행 데이터로 연결한다.**

앱 이름은 `ATLAS`로 정했다. 여행지를 보여주는 지도라는 의미도 있지만, 흩어진 여행 정보를 한곳에 올려두는 작업대에 가깝다는 느낌이 마음에 들었다.

## 처음 잡은 핵심 기능

초기 기획에서 가장 중요하게 본 기능은 다음과 같았다.

### 1. 국가와 도시를 먼저 정한다

여행의 기본 지역을 한 번 선택하면 세부 장소를 추가할 때마다 국가와 도시를 다시 입력하지 않도록 했다. 다른 도시로 이동할 때만 기본 지역을 변경한다.

이 기능은 사소해 보이지만 모바일에서 장소를 여러 개 입력할 때 차이가 컸다. 반복 입력을 줄이는 것이 여행 플래너의 첫 번째 UX라고 생각했다.

### 2. 날짜별 장소를 순서대로 관리한다

각 장소에는 다음 정보를 저장한다.

- 장소명과 주소
- 위도와 경도
- 식당, 관광지, 호텔, 마트, 공항, 터미널 같은 카테고리
- 방문 날짜와 시간
- 체류 시간
- 할 일과 메모
- 예상 경비
- 완료 여부

장소 카드는 드래그해서 순서를 바꿀 수 있게 했다. 지도 마커의 번호와 일정 카드의 번호도 동일하게 맞췄다. 그래야 지도를 보다가도 몇 번째 일정인지 바로 알 수 있다.

### 3. 경비와 준비물을 여행 데이터에 포함한다

전체 예산, 결제 완료, 결제 예정, 남은 금액을 나눠 보여주고 숙박·식비·교통 같은 카테고리별 지출도 확인할 수 있게 했다.

준비물은 단순 목록이지만 실제 여행에서는 중요하다. 체크 표시가 원 안에서 조금만 어긋나도 계속 신경이 쓰였다. 이 프로젝트를 진행하면서 기능만큼 정렬과 가독성을 많이 수정한 이유다.

## 기술 구조: 로컬 PC와 맥미니를 하나의 시스템으로 묶기

처음 고민했던 부분은 AI와 지도 서버를 어디에서 실행할 것인가였다.

현재 작업하는 PC에는 Hermes가 설치되어 있지 않았지만, 같은 네트워크의 내 맥미니에는 이미 Hermes와 Docker, Tailscale이 동작하고 있었다. 앱을 맥미니에서 처음부터 다시 개발하고 싶지는 않았다.

그래서 역할을 나눴다.

- 개발 작업: 현재 사용하는 PC
- 웹앱과 PostgreSQL: 맥미니 Docker
- Hermes Web UI와 GLM 모델: 맥미니
- OSRM 경로 서버: 맥미니 Docker
- 두 장비 연결: Tailscale

이 구조 덕분에 개발 위치와 실행 위치를 분리할 수 있었다. 코드는 한곳에서 고치고, 실제 서비스와 무거운 지도 처리는 맥미니가 맡았다.

### 주요 기술 스택

- **Frontend / Backend:** Next.js 16, React 19, TypeScript
- **Database:** PostgreSQL 17, Drizzle ORM
- **Map UI:** MapLibre GL
- **Place Search:** Nominatim, Photon, 한국관광공사 고캠핑 API
- **Route Engine:** OSRM
- **AI Research:** Hermes Web UI, Z.AI GLM 계열 모델
- **Infrastructure:** Docker Compose, Tailscale, macOS launch service

처음에는 간단한 로컬 저장으로 시작했지만 로그인과 사용자별 여행 데이터를 넣으려면 결국 데이터베이스가 필요했다. 그래서 PostgreSQL로 전환했고, 일정·장소·경비·체크리스트·AI 조사 결과가 사용자 계정에 따라 분리되도록 구성했다.

## 가장 어려웠던 문제 1: 지도에 경로가 아니라 직선이 나왔다

장소 마커를 지도에 표시하는 것과 실제 도로 경로를 그리는 것은 완전히 다른 문제였다.

처음에는 두 지점의 좌표를 선으로 연결할 수 있었다. 하지만 그것은 길이 아니라 직선이었다. 자동차가 갈 수 없는 산이나 강을 그대로 통과했다.

정확한 경로를 만들기 위해 OSRM을 붙였다. 자동차, 도보, 자전거 프로필을 따로 준비하고 지역별 OpenStreetMap 데이터를 가공했다.

여기서부터 시행착오가 시작됐다.

### 지역 데이터는 생각보다 무겁다

런던처럼 작은 지역은 비교적 쉽게 처리됐지만 프랑스나 한국 전체 지도는 달랐다. 경로 데이터를 추출하는 과정에서 메모리를 많이 사용했고, 한국 자동차 경로 생성은 진행률 95% 부근에서 종료 코드 `137`로 멈추기도 했다.

원인은 메모리 부족이었다.

해결 방법은 단순히 다시 실행하는 것이 아니었다.

- 다른 OSRM 서버를 잠시 정지해 메모리 확보
- 경로 생성 스레드를 1개로 제한
- 자동차 프로필을 먼저 완성
- 도보와 자전거는 순차 처리
- 실패한 원본 지도 파일은 다시 받지 않고 재사용
- 설치 진행률과 현재 단계를 화면에 표시

이 과정을 거쳐 한국 자동차·도보·자전거 경로를 모두 만들었다.

### Docker 업데이트 후 다시 직선으로 돌아간 사건

어느 날 Docker 업데이트 후 지도 경로가 다시 직선으로 표시됐다. 처음에는 도로 데이터가 삭제된 줄 알았다.

확인해보니 약 9GB의 한국 지도 데이터는 그대로 남아 있었다. 문제는 OSRM 컨테이너가 정지된 것이었다. 앱은 경로 서버에 연결하지 못하면 사용을 막는 대신 직선 경로를 임시로 보여주도록 설계되어 있었다.

자동차 OSRM을 다시 실행하자 충주의 두 장소 사이 경로가 약 5.09km, 132개의 도로 좌표로 정상 복구됐다.

이후에는 단순 재시작에 그치지 않고 다음 로직을 추가했다.

- 지도 관리 서버가 시작될 때 등록된 경로 컨테이너 자동 복구
- 경로 요청 시 컨테이너가 꺼져 있으면 한 번 자동 시작
- 실패하면 직선 경로를 사용하되 사용자에게 이유 표시

이 경험을 통해 느낀 점은 하나다.

**데이터가 존재하는 것과 서비스가 사용할 수 있는 상태는 다르다.**

## 가장 어려웠던 문제 2: AI 조사는 되지만 너무 오래 걸렸다

Travel Planner에는 목적지와 여행 기간을 기준으로 날씨, 교통패스, 치안, 결제, 문화, 음식 정보를 조사하는 기능을 넣었다.

별도 OpenAI API 키를 연결하는 방식 대신, 맥미니에서 이미 실행 중인 Hermes Web UI를 활용했다. 메인 모델은 Z.AI의 GLM을 사용했다.

처음 연결했을 때는 다양한 오류가 발생했다.

- JSON 대신 “조사 에이전트를 출발시켰다”는 중간 답변 반환
- `fetch failed`
- 2분 시간 초과
- 진행률 72%에서 멈춘 것처럼 보이는 현상
- 지원되지 않는 보조 모델 호출
- Z.AI 서버의 HTTP 429 과부하 응답

특히 넓은 조사 범위를 한 번에 요청하면 Hermes가 브라우저 검색과 도구 호출을 반복했고, 최종 JSON 정리 전에 제한시간을 넘겼다.

### AI에게 자유를 많이 주는 것이 항상 좋은 것은 아니었다

처음에는 “날씨, 교통, 음식, 치안, 결제, 문화까지 모두 조사해줘”라고 넓게 요청했다. 결과는 풍부했지만 느리고 불안정했다.

그래서 실행 규칙을 구체적으로 바꿨다.

- 사용자가 선택한 조사 항목만 처리
- 하위 에이전트와 백그라운드 위임 금지
- 외부 도구 호출 횟수 제한
- 동일 검색 반복 금지
- 공식 출처 우선
- 결과는 하나의 JSON으로 즉시 마무리
- 429 과부하 발생 시 잠시 기다린 뒤 한 번 자동 재시도

Hermes의 도구 반복 횟수도 제한했다. 그 결과 기존에는 151~239초가 걸리거나 실패하던 요청이 실제 검증에서 약 77초 안에 유효한 JSON으로 완료됐다.

여기서 중요한 교훈을 얻었다.

**AI 기능의 품질은 모델 이름보다 실행 범위와 실패 처리에서 결정되는 경우가 많다.**

## 가장 어려웠던 문제 3: 장소 검색 결과가 실제 장소와 달랐다

지도 앱에서 가장 중요한 것은 결국 “내가 찾은 곳이 정확히 찍히는가”다.

Nominatim과 Photon을 이용해 전 세계 장소와 주소를 검색할 수 있게 했지만, 모든 상호명과 신축 장소가 OpenStreetMap에 정확히 등록되어 있는 것은 아니었다.

그래서 장소 입력 방식을 세 가지로 나눴다.

1. 장소명 검색
2. 도로명·지번 주소 검색
3. Google Maps·네이버 지도 링크 또는 좌표 붙여넣기

사용자가 다른 지도 앱에서 이미 정확한 장소를 찾았다면 그 결과를 다시 검색하게 할 필요가 없다고 생각했다. 공유 링크에서 위도와 경도를 읽고, 역지오코딩한 주소와 함께 일정에 저장하도록 했다.

### 네이버 지도 `c=` 값을 좌표로 착각한 문제

이 기능에서도 오류가 있었다.

네이버 지도 URL의 `c=15.00,0,0,0,dh` 값을 좌표로 잘못 읽어 전혀 다른 나라가 표시됐다. 이 값은 장소 좌표가 아니라 지도 화면 상태였다.

수정한 방식은 다음과 같다.

- 네이버 장소 URL에서는 `place ID`를 먼저 추출
- 네이버 공식 장소 요약 정보에서 명칭·업종·주소·좌표 확인
- URL의 `lng`와 `lat`가 있으면 명시적 좌표로 사용
- `c=` 화면 상태는 장소 좌표로 사용하지 않음
- 네이버 업종을 ATLAS 카테고리로 변환

실제 링크를 다시 검증했을 때 장소명, 식당 카테고리, 도로명 주소와 좌표가 모두 정확하게 들어왔다.

Google Maps도 공유 링크의 `/place/장소명`을 읽고, 해당 좌표 주변 장소 검색 결과와 대조해 카테고리를 자동 선택하도록 보완했다.

자동으로 채워진 명칭·주소·카테고리는 사용자가 언제든 수정할 수 있다. 다만 지도에는 정확한 위도와 경도가 유지된다.

이 기능을 완성하고 나니 검색 데이터가 부족한 지역에서도 앱을 계속 사용할 수 있게 됐다.

## 모바일 화면은 별도의 제품처럼 다시 봐야 했다

데스크톱 화면이 잘 나온다고 모바일도 자연스럽게 되는 것은 아니었다.

실제 휴대폰으로 접속해보니 다음 문제가 계속 보였다.

- 일정 카드와 AI 조사 결과가 화면 밖으로 잘림
- 날짜와 시간이 입력 박스 안에서 어긋남
- 버튼 안의 아이콘과 글자가 중앙에 맞지 않음
- 지도 영역이 본문을 너무 많이 차지함
- 데스크톱에서는 저장된 국가·도시가 모바일에서 다시 초기화됨
- 체크 아이콘이 원 중앙에서 벗어남

모바일 지도는 항상 화면에 고정하는 대신 필요할 때 오른쪽에서 슬라이드해 보는 방식으로 바꿨다. 카드 너비와 글자 크기는 고정값이 아니라 화면 폭에 맞춰 조절했다.

프로젝트를 진행하면서 가장 자주 수정한 것은 거대한 기능이 아니라 버튼 정렬, 여백, 글자 줄바꿈 같은 부분이었다.

사용자는 기술 스택을 보지 않는다. 버튼 글씨가 삐뚤어져 있으면 앱 전체가 덜 완성되어 보인다.

## AI와 함께 개발하면서 가장 크게 느낀 점

이번 프로젝트를 하면서 “AI로 앱을 만들었다”는 표현을 다시 생각하게 됐다.

AI가 코드를 빠르게 만들고 오류를 분석해준 것은 맞다. 하지만 앱의 방향을 결정한 것은 계속된 사용 경험이었다.

- 이 버튼이 왜 오른쪽으로 치우쳐 보이는지
- 장소를 추가할 때 왜 국가를 또 입력해야 하는지
- 경로가 왜 직선인지
- AI 조사가 왜 72%에서 멈춘 것처럼 보이는지
- Docker 업데이트 후 어떤 서비스가 자동 복구되지 않았는지
- 지도 링크가 왜 다른 나라를 가리키는지

이런 질문은 앱을 직접 써보지 않으면 나오지 않는다.

개발 과정은 대략 이런 흐름이었다.

1. 내가 필요한 기능을 설명한다.
2. AI가 구현한다.
3. 실제 PC와 모바일에서 사용한다.
4. 예상과 다른 부분을 구체적으로 알려준다.
5. 로그, 데이터베이스, 컨테이너, API 응답까지 확인한다.
6. 수정 후 실제 좌표와 실제 사용자 흐름으로 다시 검증한다.

결국 AI는 개발 속도를 크게 높여줬지만, 좋은 결과를 만든 것은 **구체적인 피드백과 반복 검증**이었다.

한 번에 완벽한 프롬프트를 쓰는 것보다, 실제 문제를 정확히 설명하고 끝까지 확인하는 것이 더 중요했다.

---

## 이 프로젝트에서 얻은 실전 교훈

### 1. 기능보다 실패했을 때의 동작을 먼저 생각해야 한다

OSRM이 꺼졌을 때 직선 경로를 보여줄지, 경고를 띄울지, 자동으로 다시 시작할지 결정해야 한다. AI 서버가 느릴 때도 마찬가지다.

### 2. 로컬 AI와 로컬 지도도 결국 운영 대상이다

무료로 실행한다고 관리가 필요 없는 것은 아니다. 포트, 메모리, 컨테이너 재시작, 모델 설정, 시간 초과를 계속 관리해야 한다.

### 3. 정확한 좌표는 검색 결과보다 강하다

장소 데이터베이스에 없는 곳도 좌표가 있으면 지도에 정확히 저장할 수 있다. 외부 지도 링크 가져오기를 만든 이유다.

### 4. 모바일은 마지막에 줄이는 화면이 아니다

입력 순서, 지도 노출 방식, 카드 너비까지 모바일 사용 흐름에 맞춰 다시 설계해야 했다.

### 5. AI 개발에서도 최종 책임은 사용자에게 있다

AI가 “완료했다”고 말해도 실제로 저장되는지, 새로고침 후 유지되는지, 다른 기기에서도 같은지 직접 확인해야 한다.

## 결론: 한 번에 만든 앱이 아니라, 계속 고쳐 쓰는 도구

Travel Planner는 처음부터 완성된 설계대로 만들어진 앱이 아니다.

여행지를 조사하고 일정을 정리하려는 작은 아이디어에서 시작해, 장소 검색과 경로 계산, 오프라인 지도, AI 조사, 사용자 데이터, 모바일 UX가 하나씩 붙었다.

개발 중에는 서버가 꺼지고, 지도 데이터 생성이 메모리 부족으로 실패하고, AI 응답이 시간 초과되고, 네이버 링크가 엉뚱한 나라를 가리키기도 했다.

그런데 그 실패가 오히려 앱의 구조를 더 단단하게 만들었다.

지금의 내 결론은 이렇다.

**AI는 아이디어를 빠르게 제품으로 바꿔준다. 하지만 실제로 쓸 수 있는 제품은 사람이 계속 사용하고 질문하고 고치는 과정에서 만들어진다.**

앞으로도 Travel Planner는 계속 개선할 생각이다. 예약 정보와 현지 일정 알림, 더 정교한 장소 데이터, 여행 중 오프라인 사용성까지 아직 할 일이 남아 있다.

그래도 이제는 여행 계획을 세울 때 여러 앱 사이를 오가기보다, 내가 만든 ATLAS를 먼저 열어보게 된다.

그것만으로도 이 긴 개발 과정은 충분히 가치가 있었다.
