# 봇 분리 이후 새로 발견한 문제와 해결 과정

봇을 분리한 뒤에는 모든 것이 한결 깔끔해질 것이라고 생각했습니다.  
메인 대화는 메인 봇이 맡고, 예약 작업은 마리 봇이 맡으면 역할이 명확해지고 유지보수도 쉬워질 것 같았기 때문입니다. 실제로 분리 자체는 성공적이었습니다.  
하지만 운영을 시작한 뒤, 예상하지 못했던 새로운 문제가 드러났습니다. 봇을 나누는 것만으로는 충분하지 않았고, **분리 이후의 작업 계약과 전달 형식을 끝까지 일관되게 유지하는 것**이 더 중요하다는 사실을 다시 확인했습니다.

기존 블로깅 기록을 보면, 저는 작업을 단순히 “했다”라고 쓰지 않고, **어떤 변화가 있었고, 무엇이 막혔고, 어떻게 풀었는지**를 중심으로 정리해 왔습니다. 이번 글도 같은 방식으로 정리하겠습니다.

## 새로 발견한 문제점

가장 먼저 나타난 증상은 예약 작업이 **되묻기 시작한 것**이었습니다.  
어떤 작업은 실행 전에 “진행할까요?”처럼 확인을 요구했고, 어떤 작업은 같은 내용을 반복했으며, 또 어떤 경우에는 `400 {"message":"","code":"invalid_request_body"}` 같은 오류가 랜덤하게 발생했습니다.

처음에는 Telegram 전송 문제라고 생각했습니다.  
그다음에는 메시지 포맷이 꼬인 것 같았고, 이후에는 각 스크립트 자체의 버그까지 의심했습니다. 하지만 점검을 거듭해 보니 원인은 하나가 아니었습니다.  
실제로는 다음 세 가지가 동시에 얽혀 있었습니다.

- **작업 정의가 완전히 고정되지 않았습니다.**  
  봇을 분리하면서 `agentId`만 바뀌고, 일부 job은 전달 채널 형식을 예전 값 그대로 유지하고 있었습니다.  
  또 일부 작업은 메시지 자체가 질문형 문구로 덮여 있어서, 예약 작업인데도 마치 사람이 확인해야 하는 흐름처럼 바뀌어 있었습니다.

- **Telegram 전달 형식이 일관되지 않았습니다.**  
  어떤 작업은 `telegram:59888835`처럼 정규화된 값을 쓰는 반면, 어떤 작업은 숫자 ID만 남아 있었습니다.  
  이 차이가 아주 작아 보일 수 있지만, 실제 운영에서는 `invalid_request_body` 같은 오류를 만드는 충분한 원인이었습니다.

- **예약 작업이 너무 무거웠습니다.**  
  아침 브리핑은 날씨, 메일, 일정, 뉴스까지 한 번에 처리하고 있었고, 일일 활동보고나 백업 작업도 외부 API와 로컬 스크립트를 직렬로 호출하고 있었습니다.  
  즉, 봇 분리 이후에는 단순히 역할을 나누는 것보다 **실행 안정성**이 더 중요해졌습니다.

## 어떻게 해결했는가

이번에는 전체를 한 번에 뒤엎지 않고, **실행 경로를 하나씩 고정하는 방식**으로 접근했습니다.

### 1. 캘린더 리마인드 작업 정규화

먼저 캘린더 리마인드 작업부터 정리했습니다.  
Telegram 전달 형식을 `telegram:59888835`로 통일했고, `bestEffort`도 유지해서 전송 실패가 전체 흐름을 망치지 않도록 했습니다.

그리고 가장 중요했던 것은, 작업 메시지에서 **사전 승인 문구를 없앤 것**입니다.  
“진행할까요?”, “실행하겠습니다” 같은 문구가 들어가면 예약 작업이 명령형이 아니라 대화형으로 바뀌어 버립니다.  
예약 작업은 조건이 맞으면 실행하고, 아니면 조용히 종료해야 합니다. 이번 수정으로 그 원칙을 다시 강하게 고정했습니다.

### 2. 아침 브리핑 타임아웃 개선

아침 브리핑은 직접 실행해 보니 정상적으로 끝났습니다.  
즉, 브리핑 내용 자체가 문제라기보다는 예약 실행 중 누적된 지연과 타임아웃이 원인이었습니다.

그래서 외부 호출이 오래 걸리지 않도록 정리했고,  
실패할 경우 전체가 오래 붙잡히지 않도록 시간 제한도 줄였습니다.  
수동 실행에서는 120초 내에 정상 완료되는 것을 확인했습니다.

### 3. 로또 자동구매 판별 보강

로또 자동구매는 실제로 구매는 완료되었는데, 결과 판별이 보수적이라 `확인 필요`로 떨어지는 경우가 있었습니다.  
그래서 화면 문구와 스크린샷을 기준으로 **구매 완료 여부를 더 넓게 판단**하도록 보강했습니다.

이 작업은 작아 보일 수 있지만, 운영 중에는 이런 작은 판정 실패가 사용자 입장에서는 훨씬 큰 혼란으로 이어집니다.  
따라서 “실제로 완료된 작업은 완료로 인식한다”는 방향으로 로직을 수정했습니다.

## 이번 작업에서 얻은 결론

이번 경험에서 가장 크게 느낀 점은, **봇을 분리하는 것보다 분리 후의 계약을 유지하는 것이 더 중요하다**는 점입니다.  
어떤 봇이 어떤 역할을 맡는지, 전달 형식은 무엇인지, 실패 시 어디까지 조용히 넘어가야 하는지, 중복 실행은 어떻게 막아야 하는지가 명확하지 않으면 분리의 장점은 금방 사라집니다.

정리하면 다음과 같습니다.

- 봇 분리는 구조 개선의 시작일 뿐입니다.
- 자동화에서는 **예측 가능성**이 가장 중요합니다.
- 질문형 메시지보다 **명령형 규칙**이 안정적입니다.
- 전달 형식과 timeout 설정은 작아 보여도 운영에서는 매우 중요합니다.

결국 메인 봇은 대화에 집중하고, 마리 봇은 예약 작업에 집중하도록 다시 정리했습니다.  
이번에는 단순히 “분리했다”가 아니라, **분리 이후에도 흔들리지 않도록 작업 계약을 고정했다**는 점이 핵심입니다.
