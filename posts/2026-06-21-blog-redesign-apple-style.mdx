# 블로그 디자인을 전면 교체한 이야기

이 글은 블로그 플랫폼을 하나의 디자인 시스템에서 완전히 다른 시스템으로 갈아엎은 과정을 기록한 글입니다.

단순히 색상을 바꾸거나 폰트를 교체한 수준이 아니라, CSS 변수 구조부터 컴포넌트 설계, 다크 모드 메커니즘, 코드 블록 렌더링, 댓글 시스템까지 전부 다시 만들었습니다.

특히 이 과정에서 겪은 문제와 해결 방법을 솔직하게 기록하려고 합니다. AI 개발 도구를 쓰면서 어떤 부분은 마법처럼 빠르게 해결되고, 어떤 부분은 한 줄의 import 누락 때문에 몇 시간을 헤맸는지까지 전부 적었습니다.

---

# 왜 디자인을 바꿨는가

기존 블로그는 나쁘지 않았습니다. Next.js 16 기반이었고, styled-components와 Tailwind v4를 혼용해서 쓰고 있었어요. 하지만 몇 가지 문제가 있었습니다.

첫째, 디자인 언어가 일관되지 않았습니다. Header는 8px 라운드, PostCard는 14px, 버튼은 또 다른 반경을 쓰고 있었습니다. 통일감이 없었죠.

둘째, 다크 모드의 시인성이 떨어졌습니다. 어두운 배경에 낮은 대비의 텍스트가 섞여서, 밤에 글을 읽으면 눈이 피곤했습니다.

셋째, 코드 블록이 이상하게 렌더링되고 있었습니다. 문법 하이라이팅 테마와 배경색이 충돌해서, 코드가 읽기 어려운 상태였어요.

이 세 가지 문제를 한 번에 해결하고 싶었습니다.

# design.md — 디자인 레퍼런스 문서

이번 작업의 핵심은 design.md라는 파일이었습니다. 이 파일에는 디자인 시스템의 모든 토큰이 정의되어 있었어요. 색상, 타이포그래피, 간격, 반경, 그림자, 컴포넌트 설계 원칙까지.

처음에는 design.md에 "Awesomic"이라는 스타일 레퍼런스를 사용했습니다.

## Awesomic 스타일의 핵심

Awesomic은 극단적으로 둥근 모서리와 단일 타입페이스가 특징인 디자인 시스템이었어요.

- 카드 반경: 36px
- 버튼: 필(pill) 형태, 10000px 반경
- 색상: 옵시디안(#09090b)과 미스트(#f4f4f5) 중심의 무채색 팔레트
- 폰트: Cosmica라는 커스텀 타입페이스 (DM Sans로 대체)
- 그림자 없음 — 서피스 색상 단계로 깊이를 표현

design.md에 정의된 토큰을 CSS 변수로 변환해서 globals.css에 적용했습니다. Tailwind v4의 `@theme` 블록과 일반 CSS 변수를 함께 사용했어요.

```css
:root {
  --color-obsidian: #09090b;
  --color-mist: #f4f4f5;
  --radius-card: 36px;
  --radius-pill: 10000px;
}
```

이 토큰을 바탕으로 Header, PostCard, Footer, CategoriesClient 같은 컴포넌트를 전면 재작성했습니다. 홈 페이지의 히어로 섹션은 48px 라운드 카드로 감싸고, Contact 섹션은 옵시디안 다크 패널로 만들었어요.

나쁘지 않은 결과물이었습니다. 하지만 한 가지 문제가 있었죠.

# Awesomic에서 Apple로의 전환

Awesomic 스타일을 적용한 후, 다크 모드에서 시인성이 여전히 떨어진다는 것을 깨달았습니다. 그리고 코드 블록의 가독성도 개선이 필요했어요.

그래서 design2.md라는 두 번째 레퍼런스를 작성했습니다. 이번에는 Apple의 제품 페이지 디자인 언어를 차용했어요.

## Apple 스타일의 핵심

Apple 디자인은 "갤러리의 자연광"이라는 키워드로 요약할 수 있었습니다.

- 캔버스 배경: #f5f5f7 (fog, 순백에서 한 단계 아래)
- 카드 서피스: #ffffff (snow)
- 카드 반경: 28px (Awesomic의 36px에서 축소)
- 그림자: 전면 제거 — 색상 값 차이만으로 깊이를 표현
- 포인트 컬러: Azure (#0071e3), 오직 CTA 버튼에만 사용
- 바디 텍스트: 17px, letter-spacing -0.006em
- 폰트: SF Pro Display/Text (Plus Jakarta Sans로 대체)

핵심 차이는 "제한"이었습니다. Apple 스타일은 하나의 액센트 컬러(Azure)를 CTA 버튼에만 허용하고, 나머지는 전부 무채색으로 처리합니다. 그림자도 금지입니다. 카드의 깊이는 오직 배경색의 미세한 차이(fog vs snow)로만 표현해요.

이 원칙들을 design2.md에 토큰으로 정의하고, 다시 전체 컴포넌트를 재작성했습니다.

# 가장 큰 문제: 다크 모드가 작동하지 않았다

이게 이번 작업에서 가장 긴 시간을 잡아먹은 문제였습니다.

## 증상

우측 상단의 토글 버튼을 누르면, HTML 태그에 `data-theme="dark"` 속성은 잘 들어갔습니다. 개발자 도구로 확인해보면 분명히 속성이 있었어요. 하지만 화면은 전혀 변하지 않았습니다.

라이트 모드에서 다크 모드로 전환하면 배경이 어두워지고 텍스트가 밝아져야 하는데, 아무 변화도 없었습니다.

## 첫 번째 시도: ThemeProviderClient 수정

처음에는 ThemeProviderClient라는 컴포넌트를 의심했습니다. 이 컴포넌트는 다크 모드 토글 시 JavaScript로 CSS 변수를 직접 덮어쓰고 있었어요.

```javascript
// 이전 코드 — JavaScript가 CSS 변수를 강제 덮어씀
if (m === 'dark') {
  document.documentElement.style.setProperty('--background', '#0b1220')
  document.documentElement.style.setProperty('--foreground', '#e6eef8')
  // ... Awesomic 시절의 색상값
}
```

문제는 이 JavaScript 인라인 스타일이 CSS 클래스보다 우선순위가 높다는 것이었습니다. globals.css에서 Apple 다크 토큰을 정확히 정의해놓아도, JS가 `--background`를 예전 색상으로 덮어쓰고 있었어요.

ThemeProviderClient가 CSS 변수를 건드리지 않고 `data-theme` 속성만 설정하도록 수정했습니다. 하지만 여전히 다크 모드는 작동하지 않았습니다.

## 두 번째 시도: 인라인 스크립트 추가

layout.tsx의 `<head>`에 페이지 로드 전에 localStorage를 읽고 `data-theme`을 설정하는 인라인 스크립트를 추가했습니다. FOUC(Flash of Unstyled Content)를 방지하기 위한 표준 기법이에요.

```javascript
(function() {
  try {
    var t = localStorage.getItem('theme');
    if (t === 'dark') {
      document.documentElement.setAttribute('data-theme', 'dark');
    }
  } catch(e) {}
})();
```

이것도 문제 해결에 직접적인 도움은 되지 않았습니다. 속성은 더 빨리 설정되었지만, 여전히 다크 모드 CSS가 적용되지 않았어요.

## 세 번째 시도: 빌드된 CSS 확인

여기서 근본 원인을 발견했습니다.

빌드된 CSS 파일을 열어서 `data-theme` 규칙이 있는지 검색해봤습니다.

```bash
grep -c 'data-theme' .next/static/chunks/*.css
# 결과: 0
```

**제로.** 다크 모드 CSS 규칙이 빌드 결과물에 아예 존재하지 않았습니다.

## 진짜 원인: globals.css가 import되지 않았다

Next.js App Router에서 글로벌 CSS는 최상위 레이아웃 파일에서 명시적으로 import 해야 합니다. 그런데 `src/app/layout.tsx`에서 `globals.css`가 import되고 있지 않았어요.

```typescript
// 이전 — globals.css import 누락
import { Providers } from './providers'
import Header from '@/components/Header'
// import './globals.css'  ← 이 줄이 없었다
```

라이트 모드가 우연히 작동했던 이유는, 컴포넌트들이 `var(--color-ink, #1d1d1f)` 같은 CSS 변수 폴백 값을 사용하고 있었기 때문입니다. 변수가 정의되지 않아도 폴백 색상으로 렌더링되니까 겉보기에는 정상이었어요.

하지만 다크 모드는 `:root[data-theme="dark"]` 선택자로 변수를 오버라이드하는 방식인데, 그 규칙 자체가 빌드에 포함되지 않으니 `data-theme`을 설정해도 아무 일도 일어나지 않았던 겁니다.

## 해결

```typescript
import './globals.css'
```

이 한 줄을 추가했더니, 빌드된 CSS에 `data-theme` 규칙이 정상적으로 포함되었습니다.

```bash
grep -c 'data-theme' .next/static/chunks/*.css
# 결과: 1
```

다크 모드가 마법처럼 작동하기 시작했습니다.

이 문제를 해결하는 데 얼마나 걸렸는지 모르겠습니다. 원인은 단 한 줄의 import 누락이었어요. 하지만 그 원인을 찾기까지 세 단계의 디버깅을 거쳐야 했습니다. 속성이 설정되는지 확인하고, JS 덮어쓰기를 의심하고, 빌드 결과물을 직접 검사하고 나서야 발견했거든요.

AI 코딩 에이전트도 같은 함정에 빠졌습니다. 코드는 논리적으로 맞았으니까요. ThemeToggle은 올바르게 속성을 설정했고, globals.css에는 올바른 규칙이 있었습니다. 다만 그 규칙이 빌드에 포함되지 않고 있다는, 코드 밖의 문제였어요.

# 코드 블록 시인성 문제

다크 모드 외에 두 번째로 신경 쓴 부분은 코드 블록이었습니다.

기존 CodeBlock 컴포넌트는 prism-react-renderer의 nightOwl 테마를 사용하고 있었어요. nightOwl은 어두운 배경에 맞춘 문법 하이라이팅 테마입니다. 그런데 컴포넌트에서 배경색만 `var(--code-bg)`로 덮어쓰고 있었어요.

이게 무슨 의미냐면, 라이트 모드에서는 밝은 회색 배경에 nightOwl의 어두운 톤 토큰 색상이 얹혀지는 겁니다. 문법 하이라이팅이 깨져 보이는 게 당연했죠.

## 수정 내용

CodeBlock을 전면 재작성했습니다. 핵심 원칙은 nightOwl 테마의 배경과 토큰 색상을 그대로 유지하는 것이었어요. 배경을 덮어쓰지 않고, 테마가 설계된 대로 어두운 배경에 밝은 토큰이 보이도록 했습니다.

추가로 세 가지 개선을 했습니다.

- 라인 넘버 표시 (회색, 선택 불가)
- 복사 버튼 (글래스모피즘 스타일)
- 28px 라운드 카드로 감싸기 (Apple 카드 반경과 통일)

```typescript
<pre style={{
  ...style,  // nightOwl 테마 스타일 그대로 사용
  borderRadius: 'var(--radius-card, 28px)',
  fontSize: '14px',
  lineHeight: 1.6,
}}>
```

인라인 코드도 별도로 처리했습니다. prose 스타일에서 인라인 `code`는 `--card-muted` 배경에 6px 반경으로, 블록 코드와 시각적으로 구분되도록 했어요.

# 댓글창 디자인: Giscus 테마 동기화

댓글 시스템으로 Giscus를 사용하고 있는데, 이것도 블로그 디자인과 맞추는 작업이 필요했습니다.

## 기존 문제

Giscus가 `preferred_color_scheme` 테마를 사용하고 있었어요. 이건 브라우저의 시스템 설정을 따라가는 거라서, 블로그에서 다크 모드를 켜도 Giscus는 별도로 놀았습니다. 블로그는 다크 모드인데 댓글창만 하얗게 떠 있는 어색한 상황이었죠.

## 해결 방법

Comments 컴포넌트가 블로그 토글의 `theme-change` 이벤트를 구독하도록 만들었습니다. 모드가 바뀔 때마다 Giscus를 `light` 또는 `dark_dimmed` 테마로 다시 로드합니다.

```javascript
// 블로그 테마 변경 이벤트 리스닝
window.addEventListener('theme-change', handler)

// Giscus 테마 동기화
const giscusTheme = mode === 'dark' ? 'dark_dimmed' : 'light'
script.setAttribute('data-theme', giscusTheme)
```

래퍼도 Apple 톤으로 감쌌습니다. 28px 라운드 카드에 `--card-bg` 배경을 줘서, 라이트 모드에서는 흰 카드가 fog 캔버스 위에 떠 있고 다크 모드에서는 #1d1d1f 카드가 검은 배경 위에 떠 있는 구조예요.

# 기존 106개 글의 포맷 정렬

새 디자인을 적용하고 나니, 기존 글들이 어색하게 보이는 문제가 있었습니다. 106개의 MDX 파일이 전부 이전 디자인의 간격과 포맷에 맞춰져 있었거든요.

## 공통 패턴 분석

기존 글들을 분석해보니 몇 가지 공통적인 포맷팅 패턴이 있었습니다.

- 문장마다 빈 줄 삽입 — 문단 간 간격이 과도하게 벌어짐
- `---` 수평선 과다 사용 — 모든 섹션마다 수평선
- 헤딩 직후의 간격이 일관되지 않음
- 이미지가 라운드 처리 없이 렌더링됨

## prose 스타일 재작성

글 본문의 포맷을 담당하는 prose.ts를 전면 재작성했습니다. 기존 글의 패턴을 새 디자인 언어로 끌어오는 작업이었어요.

과도한 빈 줄로 인해 문단 사이 간격이 벌어지던 문제는 `p + p` 마진 축소로 해결했습니다.

```css
.prose p {
  margin: 0 0 16px;
}
.prose p + p {
  margin-top: 0;
}
```

수평선은 얇은 1px 라인으로 톤 다운했습니다. 헤딩은 Apple의 타입 스케일에 맞춰서 크기와 자간을 재조정했고요.

이 작업의 핵심은 **글 자체를 수정하지 않고도 새 디자인에 맞춰지도록** CSS만으로 해결했다는 점입니다. 106개 파일을 하나씩 수정하는 게 아니라, prose 스타일 규칙으로 포맷을 정규화했어요.

# 디자인 토큰 시스템 구조

최종적으로 구축된 디자인 토큰 시스템을 정리해보겠습니다.

## 색상 계층

Apple 스타일의 색상은 4단계 서피스 구조를 가집니다.

- Canvas: #f5f5f7 — 페이지 배경
- Card: #ffffff — 카드 서피스
- Muted: #f5f5f7 — 카드 내부의 리세스 영역
- Dark Stage: #000000 — 최대 대비 다크 섹션

다크 모드에서는 이 계층이 반전됩니다.

- Canvas: #000000
- Card: #1d1d1f
- Muted: #2c2c2e
- 텍스트: #f5f5f7

# AI 개발 도구와의 협업

이번 작업은 AI 코딭 에이전트와 함께 진행했습니다. 몇 가지 흥미로운 관찰을 공유하고 싶습니다.

## AI가 잘한 부분

디자인 토큰을 CSS 변수로 변환하는 작업은 매우 빠르게 진행되었습니다. design.md에 정의된 토큰 테이블을 보고 globals.css의 `:root` 블록으로 변환하는 건, 패턴 매칭 작업이니까 AI가 빠르게 처리할 수 있었어요.

컴포넌트 재작성도 효율적이었습니다. "Header를 Apple 스타일로 재디자인해"라는 지시에 대해, design.md의 원칙(frosted blur, 12px nav text, azure CTA)을 정확히 반영한 styled-components 코드를 생성해냈습니다.

## AI가 어려워했던 부분

반면 디버깅은 인간의 직관이 필요한 영역이었습니다. 다크 모드가 작동하지 않는 문제에서, AI는 코드 내부의 논리만 검사했어요. ThemeToggle이 올바르게 속성을 설정하는지, CSS 선택자가 맞는지, 변수가 정의되어 있는지. 전부 맞았습니다.

하지만 문제는 코드 밖에 있었습니다. globals.css가 빌드에 포함되지 않고 있다는 건, 코드를 아무리 검사해도 나오지 않는 정보였어요. 빌드 결과물을 직접 검사하는 발상이 필요했고, 그건 "이게 정말 빌드에 들어가 있을까?"라는 회의적인 질문에서 출발했거든요.

## 배운 점

AI 코딭 에이전트는 코드 생성에는 탁월하지만, 시스템 수준의 문제(import 누락, 빌드 파이프라인, CSS 로드 순서)를 발견하는 데는 한계가 있습니다. 이런 문제는 여전히 개발자의 경험과 직관이 필요해요.

그래도 혼자서 했으면 일주일이 걸렸을 작업을 반나절 만에 끝낼 수 있었습니다. 디자인 토큰 변환, 컴포넌트 재작성, 스크린샷 검증 같은 반복 작업을 AI가 빠르게 처리해준 덕분에, 핵심적인 문제 해결에 집중할 수 있었거든요.

# 마무리하며

이번 리디자인을 통해 배운 것들을 정리해보겠습니다.

**디자인 시스템은 토큰에서 시작한다.** 색상 하나, 반경 하나를 바꿀 때마다 모든 컴포넌트를 수동으로 수정하는 게 아니라, 토큰만 바꾸면 전체가 따라오는 구조가 중요합니다. 이번에 CSS 변수 기반의 토큰 시스템을 구축해둔 덕분에, Awesomic에서 Apple로 전환할 때 globals.css 하나만 교체하면 기반이 바뀌었어요.

**다크 모드는 처음부터 설계해야 한다.** 나중에 덧붙이려고 하면 ThemeProviderClient처럼 이전 코드의 잔재가 남아서 충돌이 발생합니다. CSS 변수 오버라이드 방식으로 처음부터 설계했다면, JavaScript가 변수를 덮어쓰는 문제도, import 누락 문제도 더 빨리 발견했을 겁니다.

**한 줄의 import가 모든 것을 결정한다.** 가장 시간이 오래 걸린 문제의 원인이 가장 단순했다는 건, 소프트웨어 개발에서 늘 반복되는 교훈인 것 같습니다.

이 블로그는 앞으로도 계속 발전할 예정입니다. 다음에는 검색 기능과 포스트 시리즈 네비게이션을 추가해보려고 합니다. 디자인 시스템의 기반이 정리되었으니, 이제 기능 추가가 훨씬 수월해질 거예요.
