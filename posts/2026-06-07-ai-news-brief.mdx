## 한줄 요약
오늘의 AI 뉴스는 **에이전트가 단순히 더 똑똑한 모델을 호출하는 단계에서 벗어나, 기억·도구·워크플로·추론 비용을 운영하는 시스템으로 진화하고 있다**는 흐름으로 묶인다. OpenAI는 Codex를 개발자 전용 도구에서 직무별 플러그인, Sites, annotations를 갖춘 업무 생성 플랫폼으로 넓혔고, Dreaming 기반 ChatGPT memory는 장기 개인화 상태를 더 신선하게 유지하려 한다. Anthropic은 Claude Opus 4.8과 Claude Code dynamic workflows로 장기 코딩 작업의 병렬 실행과 검증을 강화했다. Microsoft는 Build 2026에서 Microsoft IQ, Work IQ, Foundry, MAI 모델군을 통해 기업형 에이전트 스택을 “모델 선택 + 조직 컨텍스트 + 안전한 실행 환경”으로 재정의했다. 연구 쪽에서는 SIRI, ALAR, Code2LoRA, TempoVLA가 모두 같은 질문을 다룬다. 긴 작업을 더 싸고 안정적으로 수행하려면 어떤 기억을 모델 안에 넣고, 어떤 reasoning은 줄이며, 어떤 지식은 어댑터로 압축하고, 어떤 행동은 속도까지 제어해야 하는가.

## 오늘의 핵심 포인트
- OpenAI Codex는 주간 사용자 500만 명 규모를 내세우며, 비개발자 업무까지 포괄하는 플러그인·공유 사이트·주석 기반 수정 흐름을 공개했다.
- ChatGPT Dreaming memory는 오래된 기억이 부정확해지는 문제를 줄이고, 시간에 따라 사용자 상태를 갱신하는 장기 개인화 기반으로 자리 잡고 있다.
- Anthropic Claude Opus 4.8은 effort control, fast mode 비용 절감, Claude Code dynamic workflows를 통해 “장기 작업을 병렬로 맡기고 검증하는” 방향을 강화했다.
- Microsoft Build 2026은 MAI 자체 모델, Microsoft IQ, Work IQ, agent sandbox, Foundry Agent Service를 하나의 기업형 에이전트 운영체계로 묶었다.
- arXiv 최신 연구는 self-internalizing skills, latent reasoning, repo-specific LoRA, speed-controllable VLA처럼 런타임 비용과 실패 가능성을 줄이는 방향으로 수렴한다.

## 공식 발표와 산업 뉴스

### 1) OpenAI: Codex를 개발자 도구에서 직무형 업무 플랫폼으로 확장
OpenAI는 6월 2일 Codex의 새 기능을 공개하며, Codex가 더 이상 코드 작성만을 위한 도구가 아니라 분석가, 마케터, 디자이너, 영업, 투자, 뱅킹 같은 지식 업무까지 흡수하는 작업 공간이 되고 있다고 설명했다. 핵심은 role-specific plugins, Sites, annotations다. 플러그인은 특정 직무에 맞는 앱, 지침, 스킬, 워크플로를 묶어 Codex가 처음부터 해당 업무의 문맥으로 움직이게 만든다. Sites는 대시보드, 플래너, 프로젝트 보드, 리뷰 공간 같은 인터랙티브 웹 앱을 만들어 워크스페이스 안에서 공유하게 해준다. Annotations는 문서, 슬라이드, 사이트의 특정 부분을 찍어 “여기만 고쳐줘”라고 말할 수 있게 하는 편집 인터페이스다.

이 발표의 의미는 분명하다. 2025년의 코딩 에이전트 경쟁이 “누가 더 좋은 코드를 쓰는가”였다면, 2026년의 경쟁은 “누가 조직의 실제 업무 표면을 더 많이 가져가는가”로 이동하고 있다. Codex가 non-developer 사용자를 강조한 것은, 앱 생성과 문서 생성, 데이터 분석, 의사결정 공간 생성이 하나의 연속된 작업으로 통합되고 있다는 신호다. 개발자가 만든 도구를 비개발자가 쓰는 구조가 아니라, 비개발자가 자기 업무용 작은 앱과 분석 공간을 직접 만들고 팀과 공유하는 구조에 가까워지고 있다.

### 2) OpenAI Dreaming memory: 장기 개인화의 핵심은 저장이 아니라 갱신
OpenAI는 ChatGPT memory를 Dreaming 기반 구조로 개선하며, 기억의 staleness, correctness, scalability 문제를 정면으로 다뤘다. 기존 saved memories는 사용자가 “기억해줘”라고 말한 내용을 저장하는 데 강했지만, 시간이 지나면서 정보가 낡거나 맥락이 바뀌는 문제를 피하기 어려웠다. Dreaming은 백그라운드에서 여러 대화의 맥락을 합성해 최신의 관련성 높은 memory state를 유지하는 방식이다. OpenAI는 이번 구조가 compute 효율을 개선해 Free 사용자까지 확장할 수 있는 기반이라고 설명한다.

개인화 에이전트에서 중요한 것은 기억의 양이 아니다. 여행 계획, 프로젝트 상태, 선호, 제약 조건은 시간이 흐르며 바뀐다. “사용자가 싱가포르에 갈 예정”이라는 기억은 여행 후에는 “싱가포르에 다녀왔다”로 바뀌어야 하고, 업무 프로젝트도 완료·보류·전환 상태를 반영해야 한다. Dreaming은 이런 시간성을 memory layer에 넣는 시도다. 앞으로의 AI 비서는 긴 대화 기록을 많이 들고 있는 시스템보다, 오래된 기억을 새 상태로 바꾸고 사용자가 검토할 수 있게 해주는 시스템이 더 신뢰받을 가능성이 크다.

### 3) Anthropic: Claude Opus 4.8과 dynamic workflows의 방향
Anthropic은 Claude Opus 4.8을 공개하며 더 나은 agentic task 수행, effort control, Claude Code dynamic workflows를 강조했다. Opus 4.8은 기존 가격을 유지하면서 품질과 판단력을 개선했고, fast mode는 이전 모델 대비 더 저렴해졌다. 특히 Claude Code dynamic workflows는 큰 작업을 계획한 뒤 수백 개의 병렬 subagents를 한 세션에서 실행하고, 결과를 검증한 뒤 보고하는 연구 프리뷰 기능이다. 코드베이스 규모의 마이그레이션처럼 긴 작업을 한 번에 맡기는 사용 사례를 겨냥한다.

이 발표에서 눈여겨볼 부분은 모델 성능 자체보다 “작업 제어면”이다. 사용자는 effort를 직접 조절하고, Claude Code는 병렬 실행과 검증을 내부 워크플로로 가져간다. 즉 에이전트 제품의 차별점이 프롬프트 창에서 IDE, 터미널, 작업 큐, 승인·검증 체계로 이동하고 있다. Anthropic이 Opus 4.8의 정직성, 불확실성 표시, tool use 효율을 강조한 것도 같은 맥락이다. 장기 작업에서는 정답률보다 “틀렸을 때 조용히 지나가지 않는 능력”이 더 중요하다.

### 4) Microsoft Build 2026: 기업형 에이전트 스택의 독립 선언
Microsoft Build 2026의 메시지는 모델 다양성과 조직 컨텍스트, 안전한 실행 환경이다. Microsoft IQ, Work IQ, Fabric IQ, Foundry IQ는 기업 데이터와 업무 흐름을 에이전트가 이해하게 만드는 컨텍스트 레이어로 제시됐다. MAI-Thinking-1을 포함한 자체 모델군은 Microsoft가 OpenAI와 Anthropic 모델을 공급받는 플랫폼에 머무르지 않고, 비용과 성능을 직접 통제하려는 의도를 보여준다. Windows 쪽에서는 agent sandbox와 Microsoft Execution Containers 같은 실행 격리 기능이 강조됐고, Foundry Agent Service는 클라우드에서 세션별 격리, 영속 메모리, 탄력적 실행을 제공하는 방향으로 소개됐다.

기업 관점에서 이는 매우 현실적인 변화다. 에이전트가 실제 업무를 맡으려면 모델 호출만으로는 부족하다. 사내 문서와 회의, 이메일, CRM, 데이터베이스를 안전하게 읽어야 하고, 실행 환경은 격리되어야 하며, 모델은 비용과 지연시간에 따라 선택되어야 한다. Microsoft가 Build에서 말한 “model diverse, open and heterogeneous”는 단순한 생태계 구호가 아니라, 기업 AI 운영 비용을 낮추기 위한 전략이다.

## 최신 연구 포커스

### 1) SIRI: 스킬을 런타임 검색하지 않고 모델 안으로 내재화
SIRI는 long-horizon LLM agent가 재사용 가능한 skills를 활용하되, 추론 시점에 별도 skill bank나 retrieval을 요구하지 않도록 만드는 연구다. 에이전트는 먼저 성공한 plain rollout에서 스스로 compact skill을 채굴하고, skill-augmented rollout과 skill-free rollout을 비교해 실제로 유용한지 검증한다. 이후 유용한 skill-guided action token만 advantage-weighted distillation으로 기본 정책에 흡수한다. 결과적으로 추론 시에는 원래 프롬프트만으로 동작한다.

이 접근은 에이전트 제품의 배포 문제와 직접 연결된다. 외부 skill 저장소를 매번 검색하면 context length, latency, 운영 복잡도가 늘어난다. SIRI가 말하는 방향은 훈련 중에는 skill을 활용하되, 배포 시에는 그 지식을 모델 파라미터에 녹여 런타임 의존성을 줄이는 것이다. ALFWorld와 WebShop에서 GiGPO 대비 성능을 높인 점도 중요하지만, 더 큰 의미는 agent memory와 skill system의 일부를 훈련 과정으로 이동시킨다는 데 있다.

### 2) ALAR: 모든 턴에서 긴 CoT를 쓰지 않는 에이전트
Adaptive Latent Agentic Reasoning, ALAR는 LLM 에이전트가 매 턴 긴 chain-of-thought를 생성하는 비효율을 줄이려는 연구다. 에이전트 작업은 여러 턴으로 이어지기 때문에, 매번 verbose reasoning을 출력하면 토큰이 누적되고 지연시간이 커진다. ALAR는 routine turn에서는 compact latent reasoning을 쓰고, 어려운 결정이 필요한 턴에서만 명시적 CoT로 올라가는 dual-mode 구조를 제안한다. 논문은 agentic search와 tool-use 벤치마크에서 정확도를 유지하거나 개선하면서 토큰을 크게 줄였다고 보고한다.

이 연구는 최근 제품 흐름과도 맞닿아 있다. Anthropic의 effort control, OpenAI Codex의 워크플로 주석, Microsoft의 모델 선택 전략은 모두 “항상 최대 추론을 쓰지 말자”는 방향이다. 좋은 에이전트는 모든 단계에서 오래 생각하는 시스템이 아니라, 어떤 단계가 routine이고 어떤 단계가 위험한지 구별하는 시스템이다.

### 3) Code2LoRA: 저장소 지식을 긴 컨텍스트 대신 어댑터로 압축
Code2LoRA는 코드 언어 모델이 repository-level context를 처리하는 방식을 바꾸려는 연구다. 현재 코딩 에이전트는 imports, API, 프로젝트 관례를 이해하기 위해 RAG나 긴 입력에 의존한다. 그러나 저장소가 커지고 계속 변하면 토큰 비용이 커지고, 매번 같은 정보를 다시 넣는 비효율이 생긴다. Code2LoRA는 hypernetwork가 repository-specific LoRA adapter를 만들게 해, 저장소 지식을 추론 시 토큰 오버헤드 없이 주입하는 구조를 제안한다. Static 모드는 안정된 저장소 스냅샷에, Evo 모드는 코드 diff를 따라 adapter 상태를 갱신하는 active development에 맞춰져 있다.

코딩 에이전트 시장의 핵심 병목은 이제 모델 점수만이 아니라 repo context 비용이다. Claude Code, Codex, Cursor, Copilot, Antigravity가 모두 큰 코드베이스와 긴 작업을 다루려 할수록, 저장소 지식을 어떻게 싸게 들고 갈지가 경쟁력이 된다. Code2LoRA는 이 비용을 context window가 아니라 파라미터 효율화 문제로 바꿔 보는 시도다.

### 4) TempoVLA: 로봇 정책도 “속도 제어”가 필요하다
TempoVLA는 Vision-Language-Action 모델이 하나의 고정 속도만 학습하는 문제를 다룬다. 실제 로봇 조작은 이동 구간에서는 빠르게 움직이고, 접촉이나 정밀 조작 구간에서는 느리게 움직여야 한다. TempoVLA는 action magnitude가 이동 속도에 직접 연결된다는 점을 이용해, 명시적 speed condition으로 실행 속도를 제어하는 VLA를 제안한다.

소프트웨어 에이전트와 물리 에이전트의 공통점은 분명하다. 모든 단계가 같은 위험과 같은 계산량을 요구하지 않는다. 코딩 에이전트는 위험한 변경 전에 더 많은 검증을 해야 하고, 로봇 에이전트는 접촉 단계에서 속도를 낮춰야 한다. TempoVLA는 에이전트 설계의 핵심 변수가 “무엇을 할지”에서 “얼마나 빠르게, 어느 정도 확신으로 할지”까지 확장되고 있음을 보여준다.

## 오늘의 해석
오늘의 공통 키워드는 **운영 가능한 에이전트**다. OpenAI는 Codex와 Dreaming memory로 업무 표면과 개인화 상태를 넓히고, Anthropic은 Opus 4.8과 dynamic workflows로 긴 작업의 병렬 실행과 검증을 제품화한다. Microsoft는 조직 컨텍스트, 자체 모델, sandbox, Foundry를 묶어 기업형 에이전트 운영체계를 만든다. arXiv 연구는 이런 제품 흐름의 저수준 문제를 파고든다. SIRI는 skill을 런타임에서 훈련으로 옮기고, ALAR는 불필요한 reasoning 토큰을 줄이며, Code2LoRA는 repo context를 adapter로 압축하고, TempoVLA는 행동 속도까지 제어한다.

2026년 AI 경쟁에서 “모델이 얼마나 똑똑한가”는 여전히 중요하지만 충분조건은 아니다. 실제 차이는 장기 기억이 stale하지 않은지, 코딩 에이전트가 수백 단계 작업을 안전하게 쪼개고 검증하는지, 기업 데이터가 안전하게 grounding되는지, 추론 비용을 작업 난이도에 맞게 조절하는지에서 난다. 오늘의 뉴스는 AI가 하나의 답변 모델에서, 상태와 비용과 실행을 관리하는 작업 운영체계로 바뀌고 있음을 보여준다.

## 체크포인트
1. Codex Sites와 직무형 플러그인이 비개발자 업무 자동화에서 실제 사용 시간을 늘리는지.
2. Dreaming memory가 장기 개인화와 사용자 통제 사이의 균형을 유지하는지.
3. Claude Code dynamic workflows가 대규모 코드 변경에서 검증 품질과 비용을 동시에 만족시키는지.
4. Microsoft IQ와 Foundry Agent Service가 기업 데이터 grounding과 실행 격리를 표준화할 수 있는지.
5. SIRI·ALAR·Code2LoRA 같은 연구가 실제 제품의 토큰 비용, 지연시간, repo context 병목을 줄이는지.
