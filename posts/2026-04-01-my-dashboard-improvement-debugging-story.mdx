# 마이대시보드 개선 삽질기: 시장 데이터, 복구 패키지, 그리고 끝없는 재기동

이번 작업은 단순한 UI 손질이 아니었습니다. `~/Projects/my-dashboard`에서 시작한 작은 수정은 곧 **한국 시장 데이터 복구, 미국 관심종목 저장, stale Next.js 번들 문제, 서비스워커 제거, 포트 3000 정리, LaunchAgent 충돌 해소, 그리고 복구 스킬/패키지 제작**까지 번졌습니다. 한마디로 말하면, 기능 하나를 고치려다 대시보드 운영 체계를 다시 정리한 셈입니다.

처음에는 사용자가 체감하는 문제가 비교적 단순했습니다. 한국 시장 카드가 비거나 지수가 안 나오고, 미국 관심종목은 추가는 되는데 새로고침하면 사라지고, 어떤 날은 아예 빈 화면만 보였습니다. 그런데 막상 들어가 보니, 문제는 코드 한 줄이 아니라 **데이터 소스, 런타임, 저장 구조, 실행 방식**이 한꺼번에 얽혀 있었습니다.

## 문제의 시작: API는 200인데 화면은 비어 있었다

가장 먼저 헷갈렸던 부분은 서버와 화면이 서로 다른 얘기를 하고 있다는 점이었습니다. `/api/market`, `/api/us-market`, `/api/settings` 같은 엔드포인트는 200을 반환하는데, 실제 브라우저에서는 한국 시장이 비어 있거나 화면 전체가 텅 비었습니다. 시크릿 모드에서도 같았기 때문에 단순 캐시 이슈는 아니었습니다.

여기서 핵심 교훈이 나왔습니다.

> API가 200이라고 해서 UI가 정상이라는 뜻은 아니다.

실제로는 브라우저가 **예전 `_next/static/chunks/app/page-*.js` 번들**을 계속 받고 있었고, 새로 빌드한 결과물이 아니라 오래된 런타임이 살아 있었습니다. 이걸 확인하기 위해 live root HTML이 가리키는 `app/page-*.js` 해시와 `.next/server/app/index.html`의 최신 해시를 비교했고, 둘이 다르다는 사실을 확인했습니다. 이 경험 때문에 이후 복구 절차에는 반드시 **bundle hash 비교**를 넣게 됐습니다.

## 포트 3000은 살아 있었지만, 내가 고친 서버는 아니었다

이 문제를 더 꼬이게 만든 건 자동 실행이었습니다. `my-dashboard`는 LaunchAgent와 수동 `npm run start`가 섞여 있었고, 그 결과 3000 포트는 열려 있어도 실제로는 내가 방금 빌드한 서버가 아니라 **이전 빌드 산출물을 서빙하는 다른 인스턴스**가 응답하는 상황이 반복됐습니다.

게다가 중간에 프로세스 정리를 넓은 조건으로 해버리면, 정리 스크립트 자신까지 같이 죽는 사고도 났습니다. 그래서 나중에는 “대충 `pkill`”이 아니라, **정확한 PID 확인 → 해당 PID만 종료 → 재기동** 흐름으로 바꿨습니다. 이 과정은 귀찮지만, 자기 자신까지 kill하는 사고를 막아줍니다.

## 서비스워커가 stale UI를 붙잡고 있었다

또 하나의 함정은 PWA/서비스워커였습니다. 서버는 이미 새 코드를 빌드했는데도, 브라우저는 예전 앱 셸을 계속 보여주는 상황이 남아 있었습니다. 결국 `PwaRegister`를 제거하고 `public/sw.js`까지 정리하면서, 브라우저가 오래된 셸을 강제로 유지하지 못하게 바꿨습니다.

이후에는 “코드 고침 → 빌드 성공 → 그래도 화면 그대로”가 보이면, 기능보다 먼저 다음을 의심하게 됐습니다.

- 포트 3000 stale 프로세스
- LaunchAgent 중복 실행
- 서비스워커/앱 셸 잔존
- root HTML bundle hash mismatch

## 한국 시장과 미국 시장은 같은 주식 카드가 아니었다

처음에는 한국과 미국을 비슷한 카드로 생각했지만, 실제로는 구조가 달랐습니다.

- **한국 시장**: 6자리 숫자 코드, KOSPI/KOSDAQ, 네이버 국내 종목 페이지 기반
- **미국 시장**: 알파벳 티커, 미국 지수/개별 종목, 별도 소스와 저장 구조 필요

특히 미국 관심종목이 “추가는 되는데 새로고침하면 사라지는 문제”는 UI 문제가 아니라 저장 구조 문제였습니다. 한국 관심종목과 미국 티커를 한 배열에 섞는 방식은 구조적으로 무리였고, 결국 `marketWatchCodes`와 `usMarketWatchSymbols`를 분리해야 했습니다. 이걸 나누고 나서야 미국 관심종목 저장/복원이 안정화됐습니다.

## 한국 시장 데이터는 네이버를 기준으로 다시 잡았다

이번 작업의 가장 긴 삽질은 한국 시장 데이터 소스였습니다. 초기에는 Yahoo, fallback, 네이버 HTML, 네이버 모바일, world 페이지 등이 뒤섞여 있었고, 그 결과 어떤 때는 종목은 나오는데 지수는 0이고, 어떤 때는 검색은 되는데 가격은 비어 있고, 어떤 때는 이름 검색이 안 됐습니다.

결국 방향은 명확해졌습니다.

- 한국 **종목 가격**은 네이버 종목 페이지를 우선
- 한국 **검색**은 저장된 종목 레지스트리 기반으로 이름/코드 모두 지원
- 한국 **지수(KOSPI/KOSDAQ)** 는 네이버 모바일 데이터 기준으로 맞추기

특히 검색은 처음에 코드만 처리하던 상태라 `삼성전자`, `NAVER` 같은 이름 검색이 제대로 안 됐습니다. 이를 종목명 prefix / includes 검색까지 확장해, 이제는 `삼성`만 입력해도 삼성전자, 삼성전자우, 삼성바이오로직스 같은 결과가 바로 나오도록 바꿨습니다.

## 미국 시장은 저장과 리스트 UX까지 정리했다

미국 시장은 상대적으로 데이터 자체보다 **지속성**과 **표시 UX**가 더 문제였습니다. 저장은 서버에 되는데 새로고침 후 복원이 안 되거나, 관심종목이 많아지면 카드 높이가 짧아서 일부가 보이지 않는 문제가 있었습니다.

그래서 미국 카드에는 다음을 반영했습니다.

- `usMarketWatchSymbols`를 `/api/settings`에 저장
- 새로고침 후 복원
- 관심종목 칩 영역에 스크롤
- 실제 종목 리스트 영역에도 스크롤
- 저장 개수 제한을 기존보다 크게 완화

사용자 입장에서는 “종목을 더 많이 넣었는데 아래가 잘린다”는 아주 단순한 문제처럼 보이지만, 실제로는 UI 높이/overflow와 저장 제한이 함께 연결되어 있었습니다.

## 한국 카드도 결국 ‘보이는 방식’이 중요했다

한국 시장 카드 역시 데이터가 정상이어도, 리스트가 길어지면 적절한 `overflow-y-auto`가 없으면 체감상 “안 보인다”가 됩니다. 이번 작업에서 확인한 것은, 한국 카드 쪽은 이미 스크롤 구조가 어느 정도 살아 있었지만 미국 카드 쪽은 그 부분이 부족했다는 점입니다. 결국 기능 자체뿐 아니라 **사용자가 실제로 체감하는 스크롤/클리핑 UX**까지 봐야 “다 됐다”고 말할 수 있다는 뜻입니다.

## 이번 작업의 결정적 교훈: 복구는 손기술이 아니라 절차다

이번에 가장 큰 성과는 단순히 대시보드가 다시 뜬 것이 아닙니다. 비슷한 문제가 또 생겼을 때를 대비해, 복구 절차를 **스킬과 패키지**로 남긴 것입니다.

기존 실험용 복구 스킬은 아카이브로 돌리고, 새로 `my-dashboard-recovery` 패키지를 만들었습니다. 이 패키지는 다음을 자동으로 수행하도록 설계했습니다.

1. LaunchAgent 중지
2. dashboard 관련 프로세스 종료
3. `.next` 삭제
4. production build
5. 서버 재기동
6. root bundle 확인
7. `/api/market`, `/api/stocks/search`, `/api/us-market`, `/api/settings` 검증

이제 다음번에 비슷한 문제가 생기면, 예전처럼 대화 기록을 더듬으며 “그때 어떻게 했더라”를 반복할 필요가 없습니다. 경험을 **패키지화**했고, 실전 복구 과정을 재현 가능한 형태로 옮겼기 때문입니다.

## 이번 삽질기의 결론

돌아보면 이번 작업은 한 줄짜리 버그 수정이 아니라, **데이터 소스 재정의 + 런타임 복구 + 저장 구조 분리 + 복구 자동화**까지 묶인 작업이었습니다. 중간에는 분명히 삽질도 많았습니다. stale bundle을 서버 문제로 착각했고, 서비스워커를 늦게 의심했고, 한국 지수와 종목을 같은 방식으로 다루려다가 시간을 썼습니다.

하지만 그 과정 덕분에 무엇이 진짜 문제인지가 명확해졌습니다.

- API 200은 복구 완료가 아니다.
- root bundle hash를 반드시 봐야 한다.
- LaunchAgent와 수동 실행은 충돌할 수 있다.
- 저장 구조가 UI보다 먼저 맞아야 한다.
- 데이터가 살아 있어도 스크롤이 없으면 사용자는 “안 된다”고 느낀다.
- 경험은 결국 패키지로 남겨야 다음번에 덜 고생한다.

이번 경험은 이름 그대로 **마이대시보드 개선 삽질기**였습니다. 하지만 결과적으로는 대시보드 하나를 고친 것이 아니라, 앞으로 같은 문제가 다시 왔을 때 더 빠르고 정확하게 복구할 수 있는 기준을 만든 작업이었습니다.
