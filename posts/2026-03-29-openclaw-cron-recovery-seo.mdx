# OpenClaw 크론 에러를 Copilot + delivery 정리로 복구한 기록

예약작업은 늘 조용히 돌아가야 합니다. 그런데 한 번 꼬이기 시작하면, 모델 오류와 전달 오류가 한꺼번에 터지면서 시스템 전체가 불안정해집니다. 이번에는 OpenClaw의 cron 작업이 `invalid_request_body`와 delivery 오류를 반복하던 상황을 정리하고, 실제로 다시 정상화하기까지의 과정을 기록합니다.

## 문제를 먼저 요약하면

이번 장애는 단순히 "크론이 한 번 실패했다" 수준이 아니었습니다. 예약작업을 맡는 에이전트가 Copilot 모델과 맞지 않는 요청 바디를 보내면서 400 에러를 내고, 동시에 전달 경로도 흔들리며 `not-delivered` 상태가 반복됐습니다. 즉, **모델 호환성 문제**와 **delivery 설정 문제**가 겹친 복합 장애였습니다.

로그에서는 다음이 동시에 보였습니다.

- `400 {"message":"","code":"invalid_request_body"}` 반복
- `github-copilot / gpt-5-mini` 조합에서 특히 자주 발생
- 일부 cron job은 실행은 되지만 `deliveryStatus: not-delivered`
- KakaoTalk 채널도 별도 SSE 오류를 반복
- OpenAI 인증도 `refresh_token_reused`로 불안정

## 왜 이게 중요한가

자동화 시스템에서 예약작업은 한 번 잘못되면 사용자가 바로 눈치채기 어렵습니다. 그래서 더 무섭습니다. "어제도 안 왔네"로 지나가기 쉽기 때문입니다. 하지만 이번처럼 정기 리포트, 아침 브리핑, 백업, 조명 제어 같은 작업은 어느 하나만 빠져도 운영 신뢰도가 떨어집니다.

이 문제를 해결하면서 다시 확인한 건, **예약작업은 단순히 모델을 바꾸는 것만으로는 안정화되지 않는다**는 점입니다. 작업 정의, 라우팅, delivery, 채널 상태가 모두 맞아야 합니다.

## 원인 분석: 층위가 여러 개였다

### 1) Copilot 요청 바디 호환성 문제

`github-copilot/gpt-5-mini`에서 `invalid_request_body`가 반복됐습니다. 이건 모델 자체가 아니라, OpenClaw가 해당 provider에 보내는 요청 형식이 맞지 않는다는 뜻에 가깝습니다. 단순히 에이전트만 새로 만드는 것으로는 해결되지 않았습니다.

### 2) Delivery 경로 설정 드리프트

에이전트를 옮기는 과정에서 `delivery.channel`, `to`, `accountId`가 job마다 조금씩 달랐고, 어떤 job은 announce가 되지 않았습니다. 즉, 실행은 성공해도 텔레그램으로 전달되지 않았습니다.

### 3) 불필요한 채널이 함께 붙어 있던 상태

KakaoTalk 채널이 활성화되어 있었고, 이 채널은 별도의 SSE 오류를 반복하면서 디버깅을 더 어렵게 만들었습니다. 예약작업 안정화가 목표라면 잡음을 줄이는 게 우선이었습니다.

## 어떻게 복구했는가

### 1) 예약작업 전용 에이전트 분리

기존 `marie`를 그대로 쓰지 않고, 예약작업 전용으로 **`marie-cron`** 을 새로 만들었습니다. 이렇게 하면 직접 대화용 에이전트와 cron 전용 에이전트의 책임이 분리됩니다.

운영 관점에서 이 분리는 꽤 중요합니다.

- 직접 대화용 에이전트와
- 반복 실행용 예약작업 에이전트

를 나누면, 장애가 생겼을 때 어디를 봐야 하는지가 훨씬 명확해집니다.

### 2) Telegram 라우팅을 새 에이전트로 이동

기존 마리에 봇의 Telegram 바인딩을 새 에이전트로 옮겼습니다. 봇은 그대로 두되, 처리 주체를 `marie-cron`으로 바꾸는 방식입니다.

이렇게 하면 사용자가 체감하는 봇은 유지하면서도, 예약작업은 별도 운영 주체에서 돌릴 수 있습니다.

### 3) 크론 job 전체를 새 에이전트로 이관

예약작업은 하나씩 옮기면 누락이 생기기 쉽습니다. 그래서 기존 `marie`가 담당하던 cron을 `marie-cron`으로 전부 이관했습니다.

이때 핵심은 job을 복제하는 게 아니라, **agentId와 delivery 경로를 정리하는 것**이었습니다.

### 4) 카카오톡 채널 제거

문제 해결에 필요하지 않은 채널은 과감히 제거했습니다. KakaoTalk 채널은 오류 로그만 반복했고, 지금 목표는 예약작업 안정화였기 때문입니다. 잡음이 되는 채널은 빼는 편이 낫습니다.

### 5) job별 수동 실행으로 검증

이 단계가 가장 중요했습니다. 이론상으로는 맞아 보여도 실제로는 job마다 차이가 있습니다. 그래서 다음을 하나씩 확인했습니다.

- 실행이 되는가
- summary가 생성되는가
- announce delivery가 붙는가
- deliveryStatus가 `delivered`로 끝나는가

job 단위 검증을 하니, 무엇이 진짜 문제인지 바로 보였습니다.

## 실제로 효과가 있었던 것들

정리하고 나니 다음이 확인됐습니다.

- `Daily OpenClaw state backup` 정상 실행
- `daily-morning-briefing-sejong-0600` 정상 실행
- `Daily activity report (18:00)` 정상 실행
- 예약작업 결과가 텔레그램으로 실제 전달됨

즉, 시스템은 다시 "예약작업이 알아서 돌아가는 상태"로 복구됐습니다.

## 이번 작업에서 얻은 교훈

### 1) 모델 문제와 전달 문제는 별개로 봐야 한다

에러가 하나 보여도 실제 원인은 모델 호환성, delivery 설정, 채널 상태가 섞여 있는 경우가 많습니다. 로그를 층위별로 분리해서 봐야 합니다.

### 2) 예약작업은 전용 에이전트가 낫다

직접 대화용 에이전트와 cron 전용 에이전트를 분리하면 설정 변경의 영향 범위가 줄어듭니다. 장애가 생겨도 어디를 봐야 하는지 명확해집니다.

### 3) 채널은 적을수록 좋다

운영 목적이 하나라면 채널도 최소화하는 편이 좋습니다. 불필요한 채널은 디버깅 시간을 잡아먹습니다.

### 4) job별로 검증해야 한다

크론은 하나의 설정 파일처럼 보이지만 실제로는 job마다 payload와 delivery가 다릅니다. "전체가 정상"보다 "이 job이 정상"이라는 확인이 더 중요합니다.

## 실전 팁

- `invalid_request_body`가 나오면 먼저 provider/model 조합을 의심하세요.
- announce가 안 나가면 `delivery.channel`, `to`, `accountId`를 먼저 보세요.
- 채널 장애가 있으면 아예 제거하거나 잠시 비활성화하세요.
- 에이전트를 옮겼다면 cron뿐 아니라 routing binding도 같이 확인하세요.
- job을 하나씩 수동 실행해보면 숨은 설정 꼬임이 빠르게 드러납니다.

## 마무리

이번 복구는 단순히 에러를 고친 것이 아니라, **예약작업 시스템을 운영 가능한 형태로 다시 정리한 작업**이었습니다. 모델, 라우팅, delivery, 채널을 한 번에 정돈하니 결국 가장 중요한 것—예약작업이 조용히 정확하게 돌아가는 상태—를 되찾을 수 있었습니다.

OpenClaw 같은 자동화 시스템은 한 번 잘 세팅해두면 편하지만, 동시에 설정이 조금만 꼬여도 문제가 연쇄적으로 터집니다. 그래서 중요한 건 더 똑똑한 모델 하나를 고르는 게 아니라, **운영 구조를 단순하게 유지하는 것**입니다.
