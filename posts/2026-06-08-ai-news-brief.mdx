## 한줄 요약

오늘의 AI 흐름은 **에이전트가 더 넓은 업무 표면으로 퍼지고, 연구는 그 에이전트를 더 오래·싸게·안전하게 굴리는 시스템 문제로 이동하고 있다**는 말로 정리된다. OpenAI는 Codex를 AWS와 지식 업무 워크플로까지 확장하며 배포 채널과 사용자 범위를 동시에 넓혔다. Microsoft는 Build 2026에서 MAI 모델군, Microsoft IQ, Scout, Agent 365를 묶어 독자적인 기업형 에이전트 스택을 공개했다. GitHub는 Copilot app을 agent-native 데스크톱 경험으로 밀어붙이고, Google은 Gemma 4 12B를 로컬 멀티모달 에이전트 모델로 포지셔닝했다. Anthropic은 AI-enabled cyber threat 분석을 통해 에이전트가 공격과 방어 양쪽에서 이미 운영 리스크가 되고 있음을 보여준다. arXiv 연구는 같은 문제를 낮은 레벨에서 다룬다. MLEvolve는 장기 ML 알고리즘 탐색을, Goedel-Architect는 형식정리 증명을, Recuse Signal은 에이전트 접근 거부 신호를, Agent Memory는 장기 기억 시스템의 비용 구조를 분석한다.

## 오늘의 핵심 포인트

- OpenAI는 Codex를 AWS Bedrock/GovCloud와 연결하고, 별도 발표에서는 role-specific plugins, Sites, annotations로 Codex를 코딩을 넘어 지식 업무 플랫폼으로 확장했다.
- Microsoft는 MAI-Thinking-1, MAI-Code-1-Flash 등 자체 모델군과 Microsoft IQ/Work IQ/Agent 365를 통해 “모델 + 조직 컨텍스트 + 거버넌스”를 한 묶음으로 제시했다.
- GitHub Copilot app은 단순 IDE 보조가 아니라 agent session, canvas, sandbox, code review, partner agent apps를 관리하는 데스크톱 제어면으로 진화하고 있다.
- Google Gemma 4 12B는 16GB급 로컬 실행, encoder-free multimodal 구조, LiteRT-LM serve, macOS 앱을 앞세워 온디바이스 agentic workflow를 강조했다.
- Anthropic의 AI-enabled cyber threat 분석은 소수의 공격자가 AI를 lateral movement와 autonomous intrusion에 활용하고 있음을 보여주며, 에이전트 보안이 제품 기능이 아니라 운영 리스크임을 상기시킨다.
- arXiv 최신 연구는 self-evolving agents, formal proof blueprints, cooperative access-deny signals, agent memory profiling처럼 장기 에이전트의 운영 병목을 직접 겨냥한다.

## 공식 발표와 산업 뉴스

### 1) OpenAI: Codex의 배포 채널과 업무 범위가 동시에 넓어졌다

OpenAI는 6월 1일 frontier models와 Codex를 AWS에서 사용할 수 있게 됐다고 발표했다. 핵심은 단순한 “클라우드 추가”가 아니다. AWS 고객은 Bedrock의 보안·거버넌스·조달·과금 흐름 안에서 OpenAI 모델과 Codex를 쓸 수 있고, GovCloud까지 포함된다는 점이 중요하다. 엔터프라이즈 입장에서는 모델 성능보다 배포 경로가 병목일 때가 많다. 이미 쓰는 클라우드, 이미 통과한 보안 정책, 이미 정해진 구매 절차 안에 Codex가 들어가면 도입 마찰이 줄어든다.

6월 2일의 Codex 발표는 다른 방향의 확장이다. OpenAI는 role-specific plugins, Sites, annotations를 공개하며 Codex를 개발자만의 도구에서 직무형 작업 플랫폼으로 넓혔다. 플러그인은 특정 직무의 도구와 문맥을 묶고, Sites는 대시보드·플래너·프로젝트 보드 같은 인터랙티브 웹 앱을 워크스페이스 URL로 공유하게 하며, annotations는 문서·슬라이드·사이트의 특정 부분을 바로 찍어 수정 지시를 내리는 편집 흐름이다. 즉 Codex는 “코드 생성기”에서 “업무 산출물을 만들고 공유하고 수정하는 에이전트 작업공간”으로 이동하고 있다.

이 두 발표를 함께 보면 OpenAI의 전략은 분명하다. 한쪽에서는 AWS 같은 기존 기업 인프라 안으로 들어가고, 다른 한쪽에서는 Codex 자체의 표면을 비개발자 업무로 넓힌다. 2026년 에이전트 경쟁은 모델 벤치마크만으로 끝나지 않는다. 누가 더 많은 조직의 승인된 실행 환경에 들어가고, 누가 더 많은 직무의 실제 산출물을 끝까지 만들 수 있는지가 더 중요해지고 있다.

### 2) Microsoft: Build 2026은 독자 모델과 기업형 에이전트 운영체계의 선언

Microsoft Build 2026의 중심 메시지는 기업이 자기 컨텍스트와 거버넌스를 가진 에이전트를 만들어야 한다는 것이다. Microsoft IQ, Work IQ, Fabric IQ, Foundry IQ는 조직의 문서, 업무 흐름, 구조화 데이터, 웹 지식을 에이전트가 검색하고 계획하는 컨텍스트 계층으로 묶는다. Scout는 Teams, Outlook, OneDrive, SharePoint, 브라우저, 로컬 리소스를 넘나드는 always-on personal agent로 소개됐다. Agent 365는 로컬·클라우드 에이전트를 Entra, Defender, Purview 같은 기존 통제 체계 안으로 넣는 관리면이다.

모델 쪽에서는 MAI 모델군이 중요하다. Microsoft AI는 MAI-Thinking-1을 포함한 일곱 개의 자체 모델을 발표했고, MAI-Code-1-Flash는 GitHub Copilot과 VS Code의 실제 agentic coding harness에 맞춰 훈련된 저비용 코딩 모델로 설명됐다. Microsoft가 말하는 핵심은 “right model for the right problem”이다. 모든 작업에 가장 비싼 frontier model을 쓰지 않고, 조직 컨텍스트·작업 난이도·지연시간·비용에 따라 모델을 라우팅하겠다는 뜻이다.

이 흐름은 OpenAI와 경쟁하면서도 OpenAI에 기대는 복합 전략이다. Microsoft는 OpenAI 모델을 계속 유통하지만, 동시에 자체 모델과 자체 컨텍스트 계층, 자체 에이전트 관리 체계를 갖춘다. 기업 고객 입장에서는 특정 모델 공급자보다 “정책을 걸고, 로그를 남기고, 권한을 제어하고, 비용을 최적화할 수 있는 에이전트 운영체계”가 더 설득력 있는 구매 단위가 된다.

### 3) GitHub Copilot app: 에이전트 네이티브 개발의 제어면

GitHub는 Copilot app을 “agent-native desktop experience”로 소개했다. 기존 Copilot이 편집기 안의 보조자였다면, Copilot app은 여러 repository의 active sessions, issues, pull requests, background automations를 한 화면에서 관리하는 제어면에 가깝다. Canvases는 에이전트와 사람이 함께 보는 작업 표면이다. 계획, PR, 브라우저 세션, 터미널, 배포, 대시보드가 canvas 안에서 업데이트되고, 사용자는 그 결과를 수정하거나 승인하거나 방향을 바꿀 수 있다.

이 발표에서 중요한 단어는 sandbox와 review다. 에이전트가 더 긴 작업을 맡을수록 문제는 “시켜볼 수 있느냐”가 아니라 “어디까지 행동하게 할 것인가”가 된다. GitHub는 로컬·클라우드 sandbox, code review tier, `/security-review`, `/rubberduck`, partner agent apps를 함께 내세운다. 즉 에이전트가 코드를 쓰는 순간보다, 에이전트의 출력을 검토하고 신뢰 범위를 정하고 실패했을 때 되돌리는 과정이 제품의 핵심이 되고 있다.

### 4) Google: Gemma 4 12B와 로컬 멀티모달 에이전트

Google은 Gemma 4 12B를 로컬에서 실행 가능한 multimodal agentic model로 발표했다. 16GB VRAM 또는 unified memory급 기기에서 실행할 수 있고, encoder-free 구조로 이미지와 오디오 입력을 별도 encoder에 태우지 않고 LLM backbone에 직접 통합한다는 점을 강조한다. Google AI Edge Gallery의 macOS 확장, Eloquent의 온디바이스 voice edit, LiteRT-LM의 `serve` 명령도 함께 제시됐다.

이 발표의 의미는 “작은 모델”이 아니라 “로컬 에이전트 표면”이다. 개발자는 LiteRT-LM으로 OpenAI-compatible local endpoint를 띄워 Continue, Aider, OpenCode 같은 harness에 연결할 수 있다. 사용자는 로컬 앱 안에서 데이터 분석용 Python 코드를 생성·실행하거나, 음성 명령으로 문장을 다듬을 수 있다. 클라우드 frontier model이 고성능 장기 작업을 맡는다면, Gemma 4 12B 같은 모델은 프라이버시·지연시간·비용이 중요한 데스크톱 작업을 맡는 방향으로 자리 잡을 가능성이 크다.

### 5) Anthropic: 에이전트 보안은 이미 현실의 운영 리스크

Anthropic은 최근 AI-enabled cyber threats를 MITRE ATT&CK 관점으로 분석하며, AI가 사이버 공격자의 일상 도구가 되고 있음을 설명했다. 많은 사례는 피싱, 정찰, 스크립트 작성처럼 비교적 단순한 보조에 머물지만, 일부는 lateral movement 같은 더 복잡한 단계에 AI를 사용했다. 특히 2025년에 차단한 state-sponsored cyber espionage 사례에서는 모델이 명령 실행, 취약점 악용, credential 탈취, 전술적 의사결정의 상당 부분을 맡은 autonomous agent처럼 작동했다고 설명한다.

이 내용은 제품 발표들과 연결된다. Copilot, Codex, Claude Code, Scout 같은 시스템이 점점 더 많은 권한과 도구를 갖게 될수록, 방어자도 에이전트를 써야 하지만 공격자도 같은 패턴을 쓴다. 그래서 에이전트 제품에서 permission, sandbox, audit trail, access-deny signal, memory governance는 부가 기능이 아니라 필수 인프라가 된다.

## 최신 연구 포커스

### 1) MLEvolve: ML 알고리즘 발견을 위한 self-evolving multi-agent framework

MLEvolve는 LLM agent를 장기 ML engineering과 알고리즘 발견에 쓰는 문제를 다룬다. 기존 MLE agent는 여러 탐색 branch 사이의 정보가 단절되고, 이전 실패와 성공을 충분히 기억하지 못하며, 상위 수준의 계층적 제어가 부족하다는 한계를 가진다. 논문은 Progressive MCGS와 graph-based reference edge를 통해 branch 간 정보를 흐르게 하고, 넓은 탐색에서 점진적으로 집중 탐색으로 이동하는 구조를 제안한다.

산업적으로 보면 이 연구는 “에이전트가 실험을 많이 돌리는 것”과 “에이전트가 실험에서 배운 것을 다음 실험에 체계적으로 반영하는 것”의 차이를 보여준다. 과학·ML 자동화에서 비용은 모델 호출보다 실험 반복과 실패 관리에 있다. MLEvolve는 에이전트가 장기 탐색을 할 때 기억, 분기, 평가, 재사용을 어떻게 시스템화할지에 대한 좋은 사례다.

### 2) Goedel-Architect: 형식정리 증명을 blueprint로 쪼개는 에이전트

Goedel-Architect는 Lean 4 formal theorem proving을 위한 agentic framework다. 핵심은 blueprint다. 먼저 정의와 lemma, dependency를 담은 의존 그래프를 만들고, 각 lemma node를 tool-equipped Lean prover가 병렬로 닫는다. 실패한 lemma는 전체 blueprint를 다시 다듬는 신호가 된다. 이는 재귀적으로 lemma를 쪼개는 방식과 다르게, 전체 증명 구조를 먼저 설계하고 병렬 실행과 refinement를 반복하는 접근이다.

이 연구는 코딩 에이전트와도 닮았다. 큰 작업을 바로 풀려고 하면 context와 검증이 무너진다. 먼저 의존 그래프를 만들고, 작은 검증 가능한 단위로 나누고, 실패한 노드가 전체 계획을 수정하게 하는 구조가 필요하다. GitHub Copilot app의 canvas, Claude Code dynamic workflows, Codex의 annotations가 모두 같은 방향으로 움직이는 이유다.

### 3) Will the Agent Recuse Itself?: 접근 거부 신호의 표준화 문제

이 논문은 흥미로운 governance 문제를 제기한다. 에이전트가 실제 credential을 가지고 SSH, PostgreSQL 같은 시스템에 접근할 때, 운영자는 “자격증명은 유효하지만 자동화 에이전트는 들어오지 말라”는 신호를 어떻게 줄 수 있을까? 논문은 SSH banner나 PostgreSQL NOTICE 같은 기존 프로토콜 채널에 Recuse Signal을 보내, 자동화 에이전트가 자발적으로 물러나도록 하는 cooperative control을 제안한다. 저자도 이것이 보안 경계가 아니라 robots.txt에 가까운 협력적 신호라고 선을 긋는다.

이 문제는 앞으로 더 커질 것이다. 에이전트는 사람의 계정으로 들어오거나 service account로 들어오며, 기존 access control은 “허용/거부”만 판단한다. 하지만 현실에서는 “사람은 접속 가능하지만 자동화 에이전트는 읽기만 가능”, “이 데이터베이스는 실험 에이전트 금지” 같은 중간 규칙이 필요하다. Recuse Signal은 그런 규칙을 프로토콜 수준에서 논의하게 만드는 출발점이다.

### 4) Agent Memory: 장기 에이전트 기억 시스템의 비용 구조

Agent Memory는 장기 작업을 하는 LLM agent가 세션을 넘어 기억을 저장, 검색, 갱신할 때 어떤 시스템 비용이 발생하는지 분석한다. 논문은 agent memory system을 taxonomy로 나누고, construction, retrieval, update, consolidation 같은 단계별 비용을 측정하는 profiling harness를 제안한다. 핵심 메시지는 기억이 단순한 vector DB 문제가 아니라는 것이다. 어떤 사실을 추출할지, 언제 통합할지, 오래된 기억을 어떻게 갱신할지, 검색과 요약 비용을 어디까지 허용할지가 모두 시스템 설계 문제다.

이는 OpenAI의 memory/Dreaming류 흐름과도 연결된다. 장기 개인화와 장기 에이전트는 기억이 많을수록 좋아지는 시스템이 아니다. 오래된 기억은 stale해지고, 너무 자주 검색하면 latency가 늘고, 너무 공격적으로 요약하면 중요한 디테일이 사라진다. Agent Memory는 이 문제를 측정 가능한 workload로 만든다는 점에서 중요하다.

### 5) CollabSim과 Benchmark Agent: 에이전트 평가도 작업 자체가 된다

CollabSim은 LLM multi-agent system의 협업 능력을 controlled experiment로 평가하려는 방법론이다. MAS가 실패하는 이유가 개별 문제 해결 능력 부족이 아니라 common ground 형성, shared task understanding 유지, misalignment repair 실패일 수 있다는 문제의식에서 출발한다. Benchmark Agent는 benchmark construction 자체를 autonomous agentic system으로 만들려는 연구다. 데이터셋 설계, annotation, quality control까지 자동화해 벤치마크 생성의 병목을 줄이는 방향이다.

두 연구 모두 평가의 위치가 바뀌고 있음을 보여준다. 이제 평가는 완성된 모델에 붙는 마지막 시험지가 아니다. 에이전트가 협업을 잘하는지, 장기 작업에서 어디서 어긋나는지, 새로운 benchmark를 얼마나 빨리 만들 수 있는지가 제품 개발의 일부가 된다. AI 제품 속도가 빨라질수록 평가 자동화와 협업 평가가 경쟁력이 된다.

## 오늘의 해석

오늘의 핵심은 **에이전트 운영체계의 분화**다. OpenAI는 Codex를 AWS와 지식 업무 표면으로 확장해 배포와 사용 범위를 넓힌다. Microsoft는 조직 컨텍스트와 자체 모델, Scout, Agent 365로 기업형 운영체계를 만든다. GitHub는 agent-native desktop과 sandbox/review/canvas로 개발자 제어면을 재설계한다. Google은 Gemma 4 12B를 통해 로컬·온디바이스 에이전트 표면을 밀고 있다. Anthropic은 사이버 위협 분석으로 이런 도구들이 공격과 방어 양쪽에 들어왔음을 보여준다.

연구는 이 제품 흐름의 하부 구조를 파고든다. MLEvolve는 장기 탐색과 자기진화, Goedel-Architect는 의존 그래프와 병렬 검증, Recuse Signal은 접근 거버넌스, Agent Memory는 장기 기억 비용, CollabSim은 협업 실패의 측정 문제를 다룬다. 2026년 AI의 승부처는 “가장 강한 모델 하나”가 아니라 “모델, 메모리, 실행, 권한, 검증, 비용을 한 시스템으로 다루는 능력”으로 옮겨가고 있다.

## 체크포인트

1. Codex의 role-specific plugins와 Sites가 실제 비개발자 업무 사용 시간을 늘리는지.
2. Microsoft MAI 모델군이 Copilot/Foundry 안에서 비용과 지연시간을 얼마나 줄이는지.
3. GitHub Copilot app의 canvas와 sandbox가 장기 agent session의 검토 부담을 실제로 낮추는지.
4. Gemma 4 12B가 로컬 agentic workflow에서 프라이버시·지연시간·비용 장점을 증명하는지.
5. Recuse Signal, Agent Memory 같은 연구가 제품 수준의 에이전트 거버넌스와 기억 관리로 연결되는지.
