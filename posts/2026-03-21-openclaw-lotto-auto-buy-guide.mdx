# OpenClaw로 동행복권 로또 자동구매 구축하기

이번 글은 **OpenClaw + Playwright + macOS Keychain** 조합으로,
동행복권 로또 6/45를 **정해진 시간에 자동으로 구매**하도록 구축한 과정을 정리한 기록입니다.

핵심 목표는 단순했습니다.

- 매주 정해진 시간에
- 자동선택 5게임, 총 5,000원어치 로또를 구매하고
- 결과를 텔레그램으로 보고받는 것

이 글은 단순한 회고가 아니라,
**누가 봐도 따라할 수 있도록 단계별 가이드 형태**로 정리했습니다.

즉,
- 사람이 직접 따라해도 되고
- 다른 AI 에이전트에게 그대로 시켜도 되고
- 나중에 다시 봐도 구조를 바로 복원할 수 있게 쓰는 것이 목표입니다.

---

# 최종적으로 만든 동작 구조

최종 구성은 아래와 같습니다.

1. **OpenClaw Cron** 이 매주 목요일 16:00(KST)에 실행됩니다.
2. Cron이 **Playwright 자동화 스크립트**를 실행합니다.
3. 스크립트는 **동행복권 사이트에 로그인**합니다.
4. 게임645 페이지에서 **자동선택 5게임**을 적용합니다.
5. 구매 버튼과 확인 팝업까지 처리합니다.
6. 실행 결과를 **텔레그램으로 요약 보고**합니다.

즉, 사람 손이 들어가는 지점은 거의 두 가지뿐입니다.

- 최초 1회 계정/보안 설정
- 문제가 생겼을 때 점검

그 이후엔 주간 반복 작업을 자동화하는 구조입니다.

---

# 준비물

이 구성을 만들기 전에 필요한 것은 아래 정도입니다.

- OpenClaw가 실행되는 환경
- Playwright/Chrome 기반 웹 자동화 실행 가능 환경
- 동행복권 계정
- macOS Keychain 사용 가능 환경
- 주간 자동 실행을 위한 OpenClaw Cron

특히 중요한 포인트는 **비밀번호를 파일이나 채팅에 남기지 않는 것**입니다.
이 글의 구성은 비밀번호를 직접 코드에 넣지 않고,
**macOS Keychain에서 읽어오는 방식**을 전제로 합니다.

---

# 목표 사양 먼저 정하기

구축 전에 먼저 자동화 목표를 딱 정해야 합니다.

이번 구성에서 정한 목표 사양은 아래와 같았습니다.

- 구매 대상: 동행복권 로또 6/45
- 방식: 자동선택
- 수량: 5게임
- 금액: 5,000원
- 실행 시각: 매주 목요일 16:00 KST
- 결과 보고: 텔레그램

자동화는 목표가 모호하면 금방 꼬입니다.
예를 들어 “적당히 로또 사기” 같은 식으로 접근하면,
나중에 수량, 금액, 차단 조건, 실패 시 처리 방식이 다 흔들립니다.

그래서 먼저 **금액 / 횟수 / 주기 / 보고 방식**을 확정하는 것이 중요합니다.

---

# 1단계. 스케줄 만들기

가장 먼저 만든 것은 실행 스케줄입니다.

이번 구성에서는 OpenClaw Cron을 사용해,
매주 **목요일 16:00**에 자동 실행되도록 잡았습니다.

핵심 개념은 단순합니다.

- 주간 반복 실행
- 한국 시간 기준
- 정확한 요일/시간 고정

운영상 이 시각을 고른 이유는,
구매 여유 시간이 있고 문제가 생겨도 다시 점검할 수 있기 때문입니다.

즉 자동화는 “돌아가게 만드는 것”보다,
**문제가 생겼을 때 사람이 개입할 수 있는 시간대에 돌리는 것**이 더 중요합니다.

---

# 2단계. Playwright 스크립트 구성

실제 자동구매는 Playwright 기반 브라우저 자동화 스크립트가 담당합니다.

스크립트 역할은 다음과 같습니다.

- 로그인 페이지 이동
- 계정 입력
- 게임645 페이지 진입
- 자동선택 5게임 적용
- 구매 버튼 클릭
- 확인 팝업 처리
- 최종 결과 판별

운영할 때 유용했던 옵션도 같이 정리하면 이렇습니다.

- `--buy` : 실제 구매 실행
- `--dry-run` : 구매 직전까지만 테스트
- `--headless` : 창 없이 실행
- `--profile=...` : 세션 유지용 프로필 지정
- `--account=...` : 동행복권 계정 지정
- `--keychain-service=...` : Keychain 서비스명 지정
- `--keep-open` : 에러 시 브라우저 유지

이런 옵션 구성이 중요한 이유는,
자동화는 처음부터 실전으로 돌리면 위험하기 때문입니다.

반드시
**테스트용 모드(`--dry-run`)와 실제 실행 모드(`--buy`)를 분리**해두는 것이 좋습니다.

---

# 3단계. 비밀번호는 Keychain으로 처리하기

이 자동화에서 가장 중요한 보안 포인트는 여기였습니다.

비밀번호를:
- 스크립트 파일에 넣지 않고
- 환경변수에 오래 남기지 않고
- 채팅에도 남기지 않고

**macOS Keychain의 일반 암호(Generic Password)** 로 저장한 뒤,
실행 시점에만 읽어와 로그인에 사용하도록 구성했습니다.

이렇게 하면 좋은 점이 분명합니다.

- 코드 유출 위험 감소
- 로그/채팅에 비밀번호가 남지 않음
- 계정 비밀정보를 운영 기록과 분리 가능

주의할 점도 있습니다.
최초 1회는 macOS가 Keychain 접근 허용 팝업을 띄울 수 있는데,
여기서 적절히 허용해주지 않으면 자동화가 멈춘 것처럼 보일 수 있습니다.

그래서 실제 구축 시에는
**처음 한 번은 사람이 옆에서 Keychain 접근 허용을 확인하는 단계**가 필요합니다.

---

# 4단계. 먼저 dry-run으로 검증하기

실제 구매를 바로 돌리는 대신,
처음에는 반드시 `--dry-run`으로 점검하는 걸 추천합니다.

이 단계에서 확인할 것은 대략 이렇습니다.

- 로그인 성공 여부
- 세션 유지 여부
- 게임645 페이지 진입 가능 여부
- 자동선택 5게임 설정 가능 여부
- 구매 직전 단계까지 UI 흐름이 맞는지

이렇게 해두면,
사이트 구조가 바뀌었거나 selector가 틀렸을 때 실제 결제 단계 전에 잡아낼 수 있습니다.

자동화에서 가장 비싼 실패는 “아예 안 되는 것”보다,
**되는 줄 알았는데 마지막 단계에서만 틀리는 것**입니다.

그래서 dry-run은 사실상 필수입니다.

---

# 5단계. 실제 구매 실행 로직 만들기

검증이 끝난 뒤에는 실제 구매 단계로 넘어갑니다.

여기서 핵심은 다음 흐름을 안정적으로 처리하는 것입니다.

1. 자동선택 5게임 반영
2. 구매하기 버튼 클릭
3. “구매하시겠습니까” 확인 팝업 처리
4. 최종 성공 여부 판별

겉으로 보기엔 단순하지만,
이 단계가 실제로 가장 많이 깨졌습니다.

특히 브라우저 자동화는 다음 같은 이유로 자주 흔들립니다.

- 버튼 selector 변경
- 팝업 구조 변경
- headless 환경에서 클릭 실패
- 구매는 됐는데 성공 판별이 틀림

그래서 단순히 버튼 하나 클릭하는 식이 아니라,
**여러 UI 단서를 함께 보고 상태를 판별하는 방식**이 중요했습니다.

---

# 실제 구축하면서 겪은 문제와 해결 방법

이 부분이 아마 가장 실전적인 내용일 겁니다.

## 1. 웹 자동화는 UI 변화에 매우 취약함

처음부터 가장 흔하게 맞닥뜨린 문제는,
사이트 구조가 생각보다 자주 바뀐다는 점이었습니다.

예를 들어:
- 로그인 URL이 바뀌고
- input selector가 달라지고
- 버튼 구조가 바뀌는 식입니다.

해결 방향은 단순했습니다.

- 너무 구체적인 CSS 경로에 의존하지 않고
- `id`, `placeholder`, `aria-label` 같은 **상대적으로 안정적인 selector** 위주로 탐색 로직을 보강하는 것

자동화는 보통 “한 번 되게 만들기”보다,
**다음 UI 변경에도 덜 깨지게 만들기**가 더 중요합니다.

---

## 2. Keychain 접근 허용 문제

증상은 이랬습니다.

- 비밀번호를 못 읽음
- 스크립트가 멈춘 것처럼 보임
- 로그인 단계에서 진행이 안 됨

원인은 대부분 Keychain 접근 권한 문제였습니다.

macOS에서 해당 항목 접근을 허용하지 않으면,
자동화 쪽에서는 그저 “안 읽힌다”로 보일 수 있습니다.

해결은 처음 한 번 명확하게 처리하면 됩니다.

- Keychain 항목 확인
- 접근 허용 팝업 처리
- 가능하면 “항상 허용”으로 정리

---

## 3. Headless 환경에서 체크박스 클릭 실패

이건 브라우저 자동화에서 꽤 자주 나오는 문제입니다.

증상:
- `Element is outside of the viewport`
- headless에서는 클릭 실패
- 화면에 보이는 것 같지만 실제로는 체크가 안 됨

해결은 일반 클릭만 고집하지 않는 것이었습니다.

- `check()` / `click()` 만 쓰지 않고
- 필요하면 DOM `evaluate()` 로 체크 상태를 바꾸고
- `change`, `click` 이벤트까지 트리거

즉,
**보이는 UI를 클릭하는 방식과 DOM 상태를 직접 조작하는 방식을 둘 다 준비**하는 것이 안정적이었습니다.

---

## 4. 구매 확인 팝업에서 멈춤

이 문제도 꽤 중요했습니다.

증상:
- 구매하기 클릭 후 확인창이 뜸
- 그런데 “확인” 버튼을 자동으로 못 누름
- 결과 판별 전에 종료되어 버림

해결 방법은 팝업 안의 버튼을 더 robust하게 찾는 것이었습니다.

예를 들면:
- `input[type=button][value=확인]`
- “구매하시겠습니까” 텍스트를 포함한 컨테이너 기준 탐색
- 확인 버튼 다중 재시도

자동화는 단일 selector 하나에만 의존하면 금방 깨집니다.
그래서 **텍스트 기반 단서 + 구조 기반 탐색 + 재시도**를 함께 쓰는 편이 낫습니다.

---

## 5. 구매는 됐는데 성공 판별이 틀림

이 문제는 겉으로 안 보여서 더 위험했습니다.

실제로는 구매가 완료됐는데,
스크립트는 “판별 불가”로 종료하는 경우가 있었습니다.

원인은 성공 판별 기준이 너무 약했기 때문입니다.

처음에는 단순히 “구매내역 확인” 링크 같은 단서만 봤는데,
최신 UI에서는 영수증 형태 팝업으로 보여서 판별이 어긋났습니다.

그래서 다음처럼 성공 판별을 보강했습니다.

- “구매내역 확인” 문구 확인
- 발행일 / 추첨일 / 지급기한 같은 메타 텍스트 조합 확인

즉,
**성공 여부는 단일 문구 하나가 아니라 여러 신호를 조합해서 판단**하는 쪽이 더 안전했습니다.

---

## 6. 이미 구매된 주에는 차단이 정상 동작

자동화에서 중요한 건 모든 실패를 “오류”로 취급하지 않는 것입니다.

예를 들어 이미 해당 주차 구매가 끝난 상태에서 다시 시도하면,
정책상 차단되는 것이 오히려 정상일 수 있습니다.

이 경우엔 단순 실패가 아니라,
**주간 5천원 한도 도달 / 정책 차단**으로 따로 분류해야 운영이 편해집니다.

그래서 결과 상태를 아래처럼 나눠두는 게 좋습니다.

- 성공
- 차단(정책/주간한도)
- 실패(로그인/구조변경/예외)

이렇게 구분해야 나중에 알림 메시지를 받았을 때도 바로 상황 판단이 됩니다.

---

# 운영 팁

실제로 굴릴 때는 아래 원칙이 꽤 중요했습니다.

## 1. 처음엔 꼭 dry-run부터
실구매 전에 로그인/진입/세팅 흐름부터 검증하세요.

## 2. 결과 상태를 세 가지로 분리
- 성공
- 차단
- 실패

이렇게만 나눠도 운영 피로도가 확 줄어듭니다.

## 3. 브라우저 유지 옵션 준비
문제가 생겼을 때 `--keep-open` 같은 옵션이 있으면 디버깅이 훨씬 쉽습니다.

## 4. selector는 여러 겹으로 준비
하나만 믿지 말고,
구조 기반 / 텍스트 기반 / 속성 기반 탐색을 같이 두는 게 좋습니다.

---

# 보안 원칙

이 자동화에서 절대 타협하지 않은 원칙은 이것이었습니다.

- 비밀번호를 채팅에 남기지 않는다
- 비밀번호를 코드에 박지 않는다
- 파일 평문 저장을 피한다
- Keychain 같은 OS 보안 저장소를 활용한다

자동화는 편리하지만,
계정이 들어가는 순간 보안 문제가 곧 운영 문제로 이어집니다.

그래서 “돌아간다”보다 먼저,
**비밀정보를 어떻게 다룰 것인가**를 정리하는 것이 맞습니다.

---

# 이 구성이 특히 좋은 이유

이 구성이 괜찮았던 이유는 단순히 로또를 자동구매해서가 아닙니다.

더 중요한 건,
**OpenClaw를 스케줄러 + 보고자 + 실행 오케스트레이터로 쓰고,
브라우저 자동화는 Playwright가 맡고,
비밀정보 관리는 Keychain이 맡는 구조로 역할을 잘 나눴다는 점**입니다.

이 방식은 로또 자동구매뿐 아니라,
다른 로그인 기반 반복 작업에도 충분히 응용할 수 있습니다.

예를 들면:
- 반복 로그인 점검
- 정기 예약/신청 자동화
- 특정 사이트의 정기 구매 흐름 자동화
- 결과 보고형 개인 자동화 작업

즉 이번 작업은 한 번성 꼼수가 아니라,
**개인 자동화 시스템을 만드는 패턴**에 가깝습니다.

---

# 따라하기용 요약

이 글을 보고 바로 다시 만들고 싶다면,
순서는 아래처럼 잡으면 됩니다.

1. 자동화 목표를 구체적으로 정한다
   - 언제, 얼마, 몇 회, 어떤 방식으로 살지
2. OpenClaw Cron으로 실행 시간을 만든다
3. Playwright 스크립트로 로그인/이동/구매 흐름을 만든다
4. 비밀번호는 macOS Keychain에 저장한다
5. `--dry-run`으로 먼저 검증한다
6. 실제 구매 단계와 결과 판별 로직을 붙인다
7. 텔레그램 등으로 성공/차단/실패를 보고한다
8. UI 변경에 대비해 selector와 판별 로직을 보강한다

이 순서대로 가면,
처음부터 구조가 무너지지 않고 비교적 안정적으로 구축할 수 있습니다.

# 마무리

정리하면 이번 자동화의 핵심은 한 문장입니다.

**OpenClaw로 스케줄과 보고를 담당하고, Playwright로 동행복권 구매 흐름을 자동화하며, Keychain으로 비밀정보를 안전하게 관리하는 구조를 만들었다.**

개인 자동화는 결국 “클릭을 대신하는 코드”가 아니라,
**반복 작업을 안전하고 복구 가능하게 운영하는 시스템**에 더 가깝습니다.

이번 구축 기록이
OpenClaw 자동화나 브라우저 자동화, 혹은 AI 에이전트 기반 개인 운영 시스템을 만들고 싶은 사람들에게
실전적인 참고가 되었으면 좋겠습니다.
