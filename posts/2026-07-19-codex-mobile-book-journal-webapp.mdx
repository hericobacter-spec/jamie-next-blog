## 책은 늘어나는데, 어디에 있는지는 모르겠다

집에 책이 한두 권일 때는 기억으로 충분했다. 그런데 책장이 차고, 다른 방에도 책이 놓이기 시작하면 이야기가 달라진다. 분명히 산 책인데 어디에 꽂아뒀는지 모르겠고, 읽었는지 아직 안 읽었는지도 가물가물하다. 같은 책을 또 살 뻔한 적도 있다.

기존 도서관리 앱을 찾아보면 기능은 많았다. 문제는 등록 과정이었다. 제목과 저자를 입력하고, 표지를 찾고, 카테고리를 고르는 순간부터 귀찮아졌다. 책장을 한 번 정리하려면 수십 권을 연속으로 입력해야 하는데, 한 권 등록하는 데 시간이 오래 걸리면 결국 포기하게 된다.

그래서 목표를 아주 단순하게 잡았다.

> 책 뒤의 ISBN 바코드를 비추면 책 정보가 자동으로 채워지고, 상태만 선택해 저장한다.

처음부터 거창한 독서 플랫폼을 만들 생각은 없었다. 내가 가진 책을 빠르게 등록하고, 읽기 전·읽는 중·읽음 상태와 짧은 감상 정도를 남길 수 있는 개인 서재가 필요했다. 이 아이디어를 Codex App의 새 대화에 적으면서 프로젝트가 시작됐다.

## Codex에게 코딩보다 먼저 기획을 맡겼다

처음부터 “앱을 만들어줘”라고 하지는 않았다. 먼저 알라딘 Open API 문서를 함께 확인하고, 어떤 정보를 받을 수 있는지와 모바일에서 ISBN을 어떻게 처리할지를 정리했다.

기획 단계에서 확정한 핵심 기능은 다음과 같았다.

- ISBN-13 바코드 스캔과 체크섬 검증
- 알라딘 API를 통한 제목·저자·출판사·표지 자동 입력
- 카메라를 쓸 수 없을 때 ISBN 직접 입력
- 읽기 전·읽는 중·읽음 상태 관리
- 별점, 감상문, 기억하고 싶은 문장과 페이지 번호
- 개인 태그와 실제 책장 위치
- 이미 등록된 ISBN 중복 안내
- 모바일 홈 화면에 설치할 수 있는 PWA

여기서 중요했던 결정은 API 키를 모바일 브라우저에 넣지 않는 것이었다. 알라딘 TTB 키를 프런트엔드 코드에 넣으면 방문자가 개발자 도구에서 그대로 볼 수 있다. 브라우저가 알라딘을 직접 호출하는 대신, 내가 만든 서버 API가 중간에서 요청하도록 구성했다.

```text
모바일 브라우저
  → 내 웹앱의 ISBN 조회 API
  → 알라딘 Open API
  → 필요한 책 정보만 정리해 반환
```

이렇게 하면 키를 서버 환경변수에 보관할 수 있고, CORS나 오류 형식도 앱 안에서 일관되게 처리할 수 있다. 실제 코드에서는 `ALADIN_TTB_KEY`를 비밀 환경값으로 읽고, ISBN 길이에 따라 `ISBN13` 또는 `ISBN` 조회를 선택했다.

## 처음에는 거대한 기능보다 한 줄짜리 흐름부터

첫 개발 목표는 아래 흐름 하나였다.

```text
ISBN 입력 → 책 정보 조회 → 저장 → 내 서재 목록에 표시
```

카메라부터 붙이면 문제가 생겼을 때 원인을 구분하기 어렵다. 바코드 인식이 실패한 것인지, ISBN이 잘못된 것인지, 알라딘 API가 응답하지 않는 것인지 한꺼번에 확인해야 하기 때문이다. 그래서 ISBN 직접 입력을 먼저 완성한 뒤 카메라 스캔을 연결했다.

ISBN도 단순히 숫자 13자리가 들어왔다고 믿지 않았다. ISBN-13은 자리마다 1과 3을 번갈아 곱한 합이 10의 배수가 되는지 확인할 수 있다. ISBN-10도 마지막 `X`를 포함한 별도의 검증 방식이 있다. 스캔 직후 체크섬을 검사하면 엉뚱한 바코드를 책으로 조회하는 일을 줄일 수 있다.

```ts
const sum = isbn.split('').reduce((total, digit, index) => {
  return total + Number(digit) * (index % 2 === 0 ? 1 : 3)
}, 0)

const valid = sum % 10 === 0
```

카메라 바코드 인식에는 `@zxing/browser`를 사용했다. 모바일 카메라 스트림을 열고 EAN 계열 바코드를 읽은 뒤, 유효한 ISBN이 들어오면 조회 화면으로 넘겼다. 인식이 되지 않거나 카메라 권한을 거부한 경우에도 사용자가 막히지 않도록 직접 입력 버튼을 항상 남겨뒀다.

## 따뜻한 종이책 느낌의 모바일 화면

디자인은 크림색 종이와 짙은 초록색을 중심으로 잡았다. 기능이 많은 관리 화면이지만 차가운 업무용 대시보드처럼 보이고 싶지는 않았다. 책 표지가 주인공이 되고, 읽기 상태는 작은 색상 배지로 빠르게 구분되도록 했다.

하단 메뉴는 홈, 내 서재, 스캔, 통계, 설정으로 구성했다. 처음 기획에서는 네 개였지만 기능이 늘면서 통계를 별도 메뉴로 분리했다. 홈에서는 전체 보유 권수와 현재 읽는 책을 보고, 서재에서는 제목·저자·ISBN 검색과 상태 필터를 사용한다. 스캔 버튼은 한 손으로 누르기 쉬운 위치에 뒀다.

데이터 구조는 처음부터 책 자체의 정보와 나의 독서 기록을 구분하려고 했다. 실제 구현에서는 한 테이블에서 시작했지만, 다음과 같은 필드를 분리해 관리했다.

| 구분 | 저장 내용 |
|---|---|
| 서지 정보 | ISBN, 제목, 저자, 출판사, 출간일, 표지, 설명 |
| 보유 정보 | 수량, 책장 위치, 카테고리, 태그, 위시리스트 |
| 독서 기록 | 상태, 현재 페이지, 독서 시간, 별점, 감상문 |
| 추가 기록 | 기억할 문장, 재독 횟수, 대여자와 반환 예정일 |
| 가족 기록 | 프로필별 상태, 진도, 별점과 감상문 |

ISBN은 숫자형이 아니라 문자열로 저장했다. ISBN에는 앞자리 0이 올 수 있고 ISBN-10의 마지막 자리는 `X`가 될 수도 있기 때문이다.

## Supabase 대신 맥미니의 개인 데이터베이스를 선택했다

초기 기획에서는 Supabase와 PostgreSQL도 고려했다. 하지만 이 앱은 우선 나와 가족이 집의 책을 관리하기 위한 도구였다. 외부 공개 서비스보다 내가 관리하는 맥미니에 데이터를 두는 쪽이 목적에 맞았다.

최종 구성은 Next.js 계열의 `vinext`, TypeScript, Drizzle ORM과 SQLite 호환 데이터베이스, Docker 조합이 됐다. 컨테이너의 `/data`를 맥미니의 폴더에 연결해 이미지를 새로 빌드하거나 Docker를 재시작해도 도서 데이터가 남도록 했다.

```text
휴대폰 Safari
  → Tailscale HTTPS
  → 맥미니 Docker
  → 웹앱과 API
  → 영구 데이터 볼륨
```

모바일 카메라는 보안 연결에서만 정상적으로 사용할 수 있다. 단순히 맥미니의 포트를 열어 `http://` 주소로 접속하는 것으로는 부족했다. Tailscale의 사설 네트워크와 HTTPS를 연결해 외부에 공개하지 않으면서도 휴대폰에서 카메라를 사용할 수 있게 했다.

## 첫 배포에서 알라딘 API가 실패했다

로컬에서는 잘되던 ISBN 조회가 Docker 컨테이너에서는 실패했다. 처음에는 API 키나 요청 주소가 잘못된 줄 알았다. 로그를 확인해보니 원인은 더 단순했다. 실행 이미지로 사용한 `node:22-bookworm-slim`에 표준 인증기관 인증서 묶음이 없어서 알라딘의 HTTPS 인증서를 신뢰하지 못하고 있었다.

결국 Docker 이미지에 `ca-certificates`를 설치했다.

```dockerfile
RUN apt-get update \
    && apt-get install -y --no-install-recommends ca-certificates \
    && rm -rf /var/lib/apt/lists/*
```

이미지를 다시 빌드한 뒤 실제 ISBN으로 조회해봤다. 테스트한 책은 《아몬드》였고, 제목과 저자, 출판사와 표지가 정상적으로 들어왔다. API 키는 프로젝트 파일에 넣지 않고 맥미니의 비공개 환경 파일에 저장했으며 파일 권한도 `600`으로 제한했다.

이 문제를 겪고 나서 “로컬 빌드 성공”과 “실제 운영 환경에서 외부 API 호출 성공”은 전혀 다른 검증이라는 걸 다시 느꼈다.

## 모바일 Safari의 날짜 입력칸과 두 번 싸웠다

배포 후 휴대폰으로 확인해보니 독서 시작일과 완독일 입력칸이 화면 밖으로 삐져나왔다. 데스크톱 브라우저의 반응형 화면에서는 괜찮아 보였는데 실제 모바일 Safari의 `input[type="date"]`가 내부 최소 너비를 유지하면서 2열 레이아웃을 밀어내고 있었다.

처음에는 입력 요소에 `min-width: 0`을 넣고 간격과 패딩을 줄였다. 배포도 정상적으로 됐지만 실제 아이폰에서는 여전히 어색했다. 첫 수정이 충분하지 않았던 것이다.

결국 모바일에서는 두 날짜를 억지로 한 줄에 넣지 않기로 했다.

```text
독서 시작일  [          ]
완독일       [          ]
```

시작일과 완독일을 한 줄씩 세로로 배치하자 문제가 깔끔하게 해결됐다. 모바일 최적화는 화면을 작게 줄이는 일이 아니라, 작은 화면에서는 어떤 정보를 같은 줄에 두지 않을지를 결정하는 작업에 가까웠다.

## MVP가 끝난 뒤 실제 책장 관리 기능을 확장했다

처음 버전이 동작하고 나니 필요한 기능이 더 구체적으로 보였다. 책을 등록하는 것만큼 “어디에 꽂혀 있는지”, “가족 중 누가 읽었는지”, “누구에게 빌려줬는지”가 중요했다.

다음 개발에서 추가한 기능은 꽤 많았다.

- 같은 ISBN의 중복 감지와 보유 수량
- 거실 책장·서재처럼 실제 보관 위치 지정
- 여러 개의 개인 태그
- 카메라를 닫지 않고 다음 책을 등록하는 연속 스캔
- ISBN이 없는 책의 제목·저자 검색 등록
- 현재 페이지와 전체 페이지를 이용한 진행률
- 독서 시간과 재독 횟수
- 문장과 페이지 번호 기록
- 대여자, 대여일과 반환 예정일
- 위시리스트
- 가족별 독서 프로필
- 월간·연간 독서 목표와 분야별 통계
- JSON 전체 백업과 복원
- PWA 홈 화면 설치와 오프라인 등록 대기열

기능을 추가할 때 가장 걱정한 것은 이미 저장된 데이터였다. 당시 실제로 등록해둔 책이 5권 있었고, 새 필드를 추가한다고 기존 데이터를 날리고 싶지는 않았다. 기존 레코드를 읽을 때 빠진 값에 기본값을 채우는 방식으로 자동 확장했고, 배포 후 5권이 그대로 남아 있는지 확인했다. 임시 도서를 생성하고 삭제하는 API 테스트도 실제 운영 환경에서 진행했다.

## Codex가 만들었지만, 결정과 검증은 내가 했다

이번 프로젝트에서 Codex App은 단순한 코드 자동완성 도구가 아니었다. API 문서를 읽고 MVP 범위를 정리하는 단계부터 화면 구현, 데이터 구조 확장, 테스트, Docker 배포와 오류 분석까지 같은 대화 안에서 이어갔다.

그렇다고 “AI에게 한 줄 시켰더니 앱이 완성됐다”는 이야기는 아니다. 어떤 문제를 풀 것인지, 가족용 개인 서재로 운영할 것인지, 데이터를 어디에 둘 것인지, 실제 아이폰에서 어떤 부분이 불편한지는 내가 결정하고 확인해야 했다.

특히 다음 작업은 사람이 직접 하지 않으면 끝났다고 말하기 어려웠다.

1. 실제 책의 바코드를 휴대폰 카메라로 읽어보기
2. 실제 알라딘 API 키로 책 정보가 들어오는지 확인하기
3. Docker를 다시 띄운 뒤 기존 책이 남아 있는지 확인하기
4. 모바일 Safari에서 날짜 입력과 하단 메뉴를 직접 눌러보기
5. 임시 데이터를 만들어 수정·삭제한 뒤 원상 복구하기

AI가 구현 속도를 크게 높여준 것은 분명하다. 하지만 무엇을 검증해야 하는지 판단하고, 실패한 결과를 다시 확인하고, 개인 데이터와 비밀키의 경계를 정하는 일은 여전히 내 몫이었다.

## 이번에 얻은 교훈

### 가장 짧은 핵심 흐름부터 완성하는 게 빠르다

카메라, 데이터베이스, 외부 API를 한꺼번에 붙이는 것보다 ISBN 직접 입력부터 완성한 것이 도움이 됐다. 문제가 생길 때 원인을 한 단계씩 분리할 수 있었다.

### 모바일은 실제 기기에서 봐야 한다

개발자 도구의 작은 화면과 모바일 Safari는 같지 않았다. 특히 날짜 입력, 카메라 권한, 설치형 PWA와 캐시는 실제 기기 확인이 필요했다.

### 비밀키는 편의보다 경계가 중요하다

API 키를 프런트엔드에 넣지 않고 서버를 통해 호출한 덕분에 코드 저장소와 브라우저에서 키가 노출되지 않았다. 로그에도 실제 키를 출력하지 않았다.

### 개인용 도구일수록 백업이 중요하다

서비스 규모가 작다고 데이터 가치까지 작은 것은 아니다. 내가 읽은 책과 감상문은 다시 만들기 어렵다. Docker 볼륨과 JSON 백업·복원을 둘 다 둔 이유다.

## 마무리

책장일기는 거대한 서비스가 아니다. 지금도 우리 집 책을 더 편하게 찾고 기록하기 위해 만든 개인 웹앱에 가깝다. 하지만 아이디어 한 줄에서 시작해 모바일 카메라, 외부 API, 데이터베이스, Docker와 사설 HTTPS까지 실제로 연결해봤다는 점에서 의미가 있었다.

무엇보다 재미있었던 건 개발의 출발점이 기술이 아니었다는 점이다.

“무슨 프레임워크를 써볼까?”가 아니라 “책을 덜 귀찮게 등록하려면 어떻게 해야 할까?”에서 시작했다. Codex는 그 생각을 구조화하고 빠르게 실험할 수 있게 도와줬고, 나는 실제 책과 휴대폰을 들고 결과를 확인했다.

다음에는 서재 데이터를 바탕으로 자연어로 책을 찾는 기능과 월간 독서 회고를 더 다듬어보고 싶다. 다만 추천 기능을 늘리는 것보다 먼저, 책을 등록하고 찾고 기록하는 기본 흐름이 계속 편한지를 확인할 생각이다. 결국 매일 쓰게 되는 도구는 기능이 많은 도구가 아니라, 시작하기 귀찮지 않은 도구이기 때문이다.
