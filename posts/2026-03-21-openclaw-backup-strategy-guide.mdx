# OpenClaw 백업 전략 정리

OpenClaw를 계속 운영하다 보면 백업은 언젠가 한 번 만드는 작업이 아니라,
**계속 실패 없이 돌아가야 하는 운영 구조**에 가깝습니다.

처음에는 보통 이렇게 생각하기 쉽습니다.

- 가능하면 전체를 다 백업하자
- 자주 돌릴수록 더 안전하겠지
- 많이 남길수록 더 든든하겠지

그런데 실제 운영에 들어가면 꼭 그렇지만은 않습니다.

백업은 “많이 하는 것”보다,
**자동으로 오래 살아남는 것**과
**문제가 생겼을 때 바로 복구할 수 있는 것**이 더 중요합니다.

이번 글은 바로 그 지점에서 시작합니다.

기존 OpenClaw 백업 구조를 다시 보면서,
무거운 full backup을 매일 돌리는 구조 대신,
**매일은 경량 config backup, 매주는 전체 스냅샷**으로 역할을 분리해 운영 전략을 재설계한 과정을 정리했습니다.

이 글은 단순 작업 기록이 아니라,
왜 구조를 바꿨는지,
무엇을 기준으로 분리했는지,
다른 사람이나 AI도 그대로 따라할 수 있도록 자세히 풀어쓴 실전 가이드입니다.

---

# 왜 백업 구조를 다시 손봐야 했나

운영 중인 시스템의 백업은 단순히 “파일 하나를 잘 남겼다”로 끝나지 않습니다.
실제로 중요한 것은 대체로 세 가지입니다.

1. **정기적으로 자동 실행되는가**
2. **실패 확률이 낮은가**
3. **복구할 때 동선이 명확한가**

기존에는 full backup 중심 사고가 강했습니다.
즉 가능한 범위를 한 번에 다 묶어서 남기는 방향입니다.

겉보기에는 든든합니다.
하지만 이 방식은 운영 단계로 가면 문제를 드러냅니다.

- 데이터가 쌓일수록 용량이 커짐
- 실행 시간이 길어짐
- verify까지 붙으면 더 무거워짐
- 새벽 자동화에서 timeout 위험이 커짐
- 출력 경로를 잘못 잡으면 source path 안으로 다시 백업을 쓰는 구조적 실수도 가능함

즉,
**백업이 있다는 사실보다 백업 자동화가 실패 없이 계속 유지되는가**가 더 중요해지는 시점이 옵니다.

이때 필요한 건 단순 최적화가 아니라,
백업 목적 자체를 다시 나누는 일입니다.

---

# 핵심 판단: daily와 weekly는 목적이 다르다

이번 정리의 출발점은 아주 단순했습니다.

**매일 해야 하는 백업과, 주간 단위로 남기면 되는 백업은 역할이 다르다.**

이걸 먼저 분리해서 보면 구조가 선명해집니다.

## 매일 필요한 백업
매일 필요한 백업은 이렇게 생겨야 합니다.

- 가벼워야 함
- 빠르게 끝나야 함
- 실패 확률이 낮아야 함
- 설정이 깨졌을 때 빠르게 복구 가능해야 함

즉 daily는 “최대한 자주, 안정적으로 도는 구조”가 핵심입니다.

---

## 매주 필요한 백업
반면 주간 백업은 목적이 다릅니다.

- 전체 상태를 한 번 통째로 남김
- 특정 시점의 복구 기준점 역할
- 용량이 커도 괜찮음
- 실행 시간이 길어도 daily만큼 민감하지 않음

즉 weekly는 “무겁더라도 복원력 있게” 가는 구조가 더 맞습니다.

이 둘을 같은 작업으로 묶으면,
결국 daily가 너무 무거워지거나 weekly가 너무 약해집니다.

그래서 이번 재정리는
**daily는 경량화, weekly는 전체 보존**
이라는 두 층 구조로 나누는 방향으로 갔습니다.

---

# 재정리 전 구조의 문제

## 1. full backup을 매일 돌리기엔 너무 무거웠다
OpenClaw를 계속 쓰면 환경 안에는 이런 것들이 쌓입니다.

- 세션 데이터
- 상태 파일
- 로그
- 각종 캐시와 작업 흔적

이걸 통째로 매일 백업하면,
시간이 갈수록 아카이브는 커지고 실행 시간은 길어집니다.

문제는 백업이 느리다는 것 자체가 아니라,
**자동화 작업이 실패하기 쉬워진다**는 데 있습니다.

대표적으로 생기는 문제가 이렇습니다.

- cron 실행 시간이 길어짐
- timeout 위험 증가
- verify 비용 증가
- 저장소 사용량 빠르게 증가

즉,
full backup을 매일 돌리는 구조는
“언뜻 가장 안전해 보이지만 실제론 가장 지치기 쉬운 구조”일 수 있습니다.

---

## 2. source path 안에 백업 파일을 쓰는 위험
백업에서 자주 놓치는 문제 중 하나가,
**백업 대상 안에 백업 결과물을 다시 써버리는 구조**입니다.

예를 들어 `~/.openclaw`를 백업하면서,
출력도 `~/.openclaw` 아래에 쓰면 구조가 복잡해질 수 있습니다.

이건 단순 미관 문제가 아니라,
나중에 반복 백업, 아카이브 중첩, 용량 증가, 관리 혼선을 일으킬 수 있습니다.

그래서 이번 정리에서는 원칙을 명확하게 잡았습니다.

- 백업 산출물은 source path 밖으로 뺀다
- daily와 weekly의 저장 위치를 분리한다
- 어떤 백업이 어떤 용도인지 폴더 구조만 보고도 알 수 있게 한다

---

## 3. 복구 동선이 백업 생성보다 중요했다
운영에서 중요한 질문은 이것입니다.

- 문제가 생겼을 때 가장 먼저 무엇을 복구할 것인가?
- 설정만 복구하면 해결되는가?
- 전체 시점 복구가 필요한가?

이 관점에서 보면,
모든 백업이 같은 역할일 필요는 없습니다.

오히려:
- 설정 복구용
- 전체 상태 복구용

이렇게 나누는 편이 훨씬 현실적입니다.

---

# 이번에 바꾼 핵심 구조

이번 재정리의 핵심은 딱 두 줄입니다.

- **Daily backup = 경량 config backup**
- **Weekly backup = 전체 스냅샷**

이 구조를 기준으로 전체 자동화와 저장 경로를 다시 설계했습니다.

---

# 1. Daily backup: 경량 config backup

매일 새벽 03:00에는 설정 중심의 경량 백업만 수행하도록 바꿨습니다.

핵심 방식은 다음과 같습니다.

```bash
openclaw backup create --only-config --verify
```

그리고 출력 경로는 별도 백업 폴더로 분리했습니다.

```text
~/Backups/openclaw
```

이 구조가 좋은 이유는 명확합니다.

- 빠름
- verify까지 포함해도 부담이 적음
- 매일 자동화에 적합함
- 설정 복구 목적에 정확히 맞음

즉 daily는 “가능한 걸 많이 담는 백업”이 아니라,
**매일 실패 없이 돌아가는 백업**으로 재정의한 것입니다.

이게 실제 운영에서는 훨씬 중요합니다.

---

# 2. Weekly backup: 전체 스냅샷

매주 일요일 04:30에는 전체 상태 보존용 주간 스냅샷을 따로 남기도록 만들었습니다.

이 작업은 단순 backup command 하나로 끝내지 않고,
별도 스크립트 기반으로 구성했습니다.

핵심 설계는 이렇습니다.

- `~/.openclaw` 전체를 tar.gz로 스냅샷
- 단, `~/.openclaw/backups`는 제외
- 저장 경로는 별도 주간 백업 폴더 사용
- 완료 후 크기와 SHA256 출력
- 결과는 Telegram으로 요약 보고

저장 위치는 이렇게 분리했습니다.

```text
~/Backups/openclaw-weekly
```

이 구조는 daily와 다르게,
“빠른 복구”보다
**특정 시점의 전체 상태를 안전하게 남겨두는 것**에 더 초점을 둡니다.

즉 weekly는 복원력과 이력 보존용입니다.

---

# 왜 이 구조가 더 운영 친화적인가

이 구조의 장점은 단순히 daily 1개, weekly 1개가 있다는 데 있지 않습니다.
진짜 장점은 **운영 목적별로 백업을 분리한 것**에 있습니다.

## 1. daily는 성공률 중심
매일 도는 작업은 일단 가벼워야 합니다.
무거운 작업은 언젠가 실패합니다.
자동화에서 제일 중요한 건 화려함보다 생존성입니다.

## 2. weekly는 복원력 중심
전체 스냅샷은 무겁더라도 괜찮습니다.
대신 주간 단위로 확실하게 남겨,
정말 문제가 생겼을 때 시점 복구 기준점 역할을 하도록 하면 됩니다.

## 3. 저장 경로만 봐도 역할이 보임
- `~/Backups/openclaw` → daily config backup
- `~/Backups/openclaw-weekly` → weekly full snapshot

이렇게 나누면 사람이 봐도 바로 이해되고,
AI에게 다시 시켜도 복구 동선을 설명하기 쉬워집니다.

## 4. 자동화 실패 리스크가 크게 줄어듦
기존 full backup daily 구조에서 생기던
- timeout 위험
- 대용량 생성 부담
- verify 비용
을 많이 덜어낼 수 있습니다.

---

# 실제로 반영한 작업

이번 재정리에서 실제로 한 일은 다음과 같습니다.

## 1. daily 03:00 백업 cron 수정
기존 daily 백업을 경량 구조로 바꾸었습니다.

방향은 분명합니다.

- full → config-only
- verify 유지
- 출력 경로는 `~/Backups/openclaw`

즉 daily는 “많이 담는 백업”이 아니라,
**안전하게 계속 도는 백업**으로 재설계했습니다.

---

## 2. 구조 변경 후 수동 검증
자동화에 넣기 전에,
구조 변경 직후 수동으로 1회 실행해 verify까지 확인했습니다.

이 단계는 생각보다 중요합니다.

설계는 좋아 보여도 실제 명령이 도는지,
산출물이 원하는 위치에 생기는지,
verify가 통과하는지는 반드시 직접 확인해야 합니다.

즉 변경 직후에는 최소한 아래 세 가지는 직접 보는 게 맞습니다.

- 수동 실행 성공 여부
- verify 통과 여부
- 결과 파일 위치

---

## 3. weekly 전체 스냅샷 스크립트 작성
주간 보존용으로 `scripts/openclaw_weekly_snapshot.sh` 를 작성했습니다.

이 스크립트의 역할은 다음과 같습니다.

- `~/.openclaw` 전체를 tar.gz 스냅샷으로 저장
- 내부 백업 폴더는 제외
- 결과 크기 출력
- SHA256 출력

이렇게 하면 주간 백업은 단순 보관이 아니라,
나중에 파일이 손상되지 않았는지 확인할 수 있는 기준까지 같이 남길 수 있습니다.

---

## 4. weekly cron 추가
새 cron `Weekly OpenClaw snapshot` 을 추가했습니다.

설정 개념은 다음과 같습니다.

- 매주 일요일 04:30 (Asia/Seoul)
- isolated 실행
- timeout 없이 수행
- Telegram으로 결과 보고

daily는 가볍고 빨라야 하지만,
weekly는 시간이 더 걸려도 전체 아카이브 완성이 우선입니다.
그래서 주간 작업은 더 여유 있게 설계하는 편이 낫습니다.

---

## 5. 운영 문서 보강
이번 작업은 단순히 스크립트 하나 바꾸고 끝난 게 아닙니다.
운영 문서도 같이 보강했습니다.

관련 문서:
- `operations/model-handover-checklist.md`
- `operations/scheduled-work-registry.md`

이 문서들이 중요한 이유는,
백업 구조는 결국 **누가 이어받아도 이해 가능해야 하기 때문**입니다.

백업 파일만 있어서는 부족합니다.
- 어떤 cron이 살아 있는지
- 왜 daily/weekly가 분리됐는지
- 어떤 점을 주의해야 하는지
까지 기록되어야,
운영 체계가 사람 교체나 세션 전환에도 버틸 수 있습니다.

# 블로그 글로 가져갈 핵심 메시지

이번 작업을 블로그 포스트로 정리할 때 핵심 메시지는 세 가지입니다.

## 메시지 1
**OpenClaw 백업은 “다 싸잡아 한 번에”보다 daily/weekly 역할 분리가 더 운영 친화적이다.**

## 메시지 2
**매일은 설정 복구용 경량 백업, 매주는 전체 상태 복원용 스냅샷으로 나누는 것이 실패율과 복구성을 동시에 잡는다.**

## 메시지 3
**백업 설계에서 더 중요한 것은 백업 파일 생성 자체보다, 자동화 성공률과 복구 동선의 명확성이다.**

이 세 가지가 이 작업의 본질입니다.

---

# 다른 사람이나 AI가 따라하려면

이 구조를 다시 만들고 싶다면 순서는 아래처럼 잡으면 됩니다.

1. 현재 백업 구조와 cron 작업을 먼저 점검한다
2. daily 백업은 full이 아니라 config-only로 경량화한다
3. 출력 경로를 source path 밖 전용 폴더로 분리한다
4. verify 옵션을 붙여 수동 검증한다
5. weekly 전체 스냅샷 스크립트를 따로 만든다
6. `~/.openclaw/backups` 같은 내부 백업 폴더는 제외한다
7. weekly 결과에 크기와 SHA256을 남긴다
8. daily와 weekly 저장 경로를 분리한다
9. cron/registry/운영 문서에 구조를 명확히 기록한다
10. 결과를 Telegram 같은 채널로 요약 보고하도록 붙인다

이렇게 가면 단순한 백업이 아니라,
**오래 버티는 운영용 백업 체계**가 됩니다.

---

# 개인적 판단

이번 정리에서 가장 크게 다시 확인한 건,
백업을 많이 하는 것과 잘 복구되는 것은 다르다는 사실입니다.

매일 전체를 무겁게 백업하는 방식은 얼핏 가장 안전해 보입니다.
하지만 실제로는 자동화 실패 가능성을 높이고,
운영 부담을 키울 수 있습니다.

반대로:
- 매일은 가볍고 확실하게
- 매주는 무겁더라도 복원력 있게

이렇게 역할을 분리하면,

# 마무리

정리하면 이번 백업 구조 개편의 핵심은 한 문장입니다.

**OpenClaw 백업은 매일 full backup을 무겁게 돌리는 방식보다, daily config backup과 weekly full snapshot으로 역할을 분리하는 쪽이 더 안정적이고 운영 친화적이었다.**

백업은 결국 파일을 얼마나 많이 남기느냐보다,
얼마나 오래 실패 없이 자동으로 돌고,
문제가 생겼을 때 얼마나 빨리 복구할 수 있느냐가 더 중요합니다.

OpenClaw를 장기 운영할 생각이라면,
백업도 기능이 아니라 **운영 전략**으로 설계하는 편이 훨씬 낫습니다.
