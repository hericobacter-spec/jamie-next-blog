# OpenClaw 봇 분리로 토큰 비용 줄이기 | 메인 봇과 예약 작업 봇을 분리한 실전 구축기

AI 자동화를 오래 돌려보면,
생각보다 빨리 부딪히는 문제가 하나 있습니다.

바로 **토큰 비용**입니다.

처음에는 “자동화가 잘 돌아가면 됐지” 싶습니다.
하지만 예약 작업이 늘어나고,
아침 브리핑·일정 리마인드·백업 보고·주식 마감 브리핑 같은 작업이 계속 쌓이면 이야기가 달라집니다.

특히 메인 대화용 봇 하나에 모든 걸 몰아넣으면,
다음 문제가 한 번에 생깁니다.

- 메인 대화 컨텍스트가 불필요하게 커짐
- 예약 작업까지 고성능 모델이 계속 사용됨
- 간단한 정기 보고에도 과한 비용이 들어감
- 같은 채널에서 대화와 자동화가 섞여 운영 가독성이 떨어짐
- 디버깅할 때 “누가 보낸 메시지인지” 구분이 어려워짐

그래서 이번에는 OpenClaw를 **메인 봇과 예약 작업 봇으로 분리하는 방식**으로 다시 구성했습니다.

결론부터 말하면,
이 방식은 단순히 봇을 하나 더 만드는 수준이 아닙니다.

**토큰 비용 절감, 운영 분리, 자동화 안정성 확보**를 동시에 가져가는 구조입니다.

이번 글에서는 실제로 제가 적용한 내용을 기준으로,
다음 내용을 자세히 정리해보겠습니다.

- 왜 OpenClaw 봇 분리가 토큰 비용 절감에 유리한지
- 메인 봇과 예약 작업 봇을 어떻게 나눠야 하는지
- Marie 에이전트를 반복 작업 전용으로 분리한 이유
- Telegram 라우팅과 cron delivery를 어떻게 분리했는지
- 분리 후 생긴 문제와 어떻게 안정화했는지
- 이 구조가 어떤 사람에게 특히 잘 맞는지

# 왜 AI 자동화는 시간이 갈수록 비용이 커질까

AI 자동화에서 비용이 커지는 이유는 단순합니다.

**“작업 수는 늘어나는데, 구조는 그대로이기 때문”**입니다.

예를 들어 다음 자동화가 있다고 가정해보겠습니다.

- 매일 06:00 아침 브리핑
- 일정 15분 전 리마인드
- 매일 18:00 활동 보고
- 매일 03:00 설정 백업
- 평일 15:45 주식 마감 브리핑
- 주간 스냅샷 보고

이런 작업은 하나하나 보면 길지 않습니다.

하지만 문제는,
이 작업들이 전부 **메인 대화 세션 감각**으로 돌아가면
비용 구조가 급격히 비효율적이 된다는 점입니다.

왜냐하면 예약 작업은 보통 다음이 필요하지 않습니다.

- 긴 인간 대화 맥락
- 과거 수십 턴의 대화 기록
- 고도의 자유 대화 능력
- 고비용 모델의 강한 추론

대부분 필요한 건 오히려 이쪽입니다.

- 정해진 시각에 실행
- 정해진 데이터 소스 조회
- 정해진 형식으로 출력
- 실패 시 규칙적인 fallback

즉,
예약 작업은 **대화형 AI라기보다 작업형 AI**에 가깝습니다.

그런데 메인 봇 하나에 이걸 다 몰아버리면,
반복 작업도 메인 봇의 무거운 컨텍스트와 모델 비용을 계속 따라가게 됩니다.

이게 장기적으로 토큰 낭비를 부릅니다.

---

# 그래서 왜 봇을 분리해야 하나

핵심은 역할 분리입니다.

저는 이번에 OpenClaw를 아래처럼 나눴습니다.

## 1) main

역할:
- 사람과 직접 대화
- 요청 해석
- 긴 문맥이 필요한 작업
- 파일 정리, 설계, 기획, 복잡한 의사결정

즉,
**메인 봇은 생각하고, 판단하고, 긴 맥락을 유지하는 역할**입니다.

## 2) marie

역할:
- cron 기반 반복 작업 실행
- 아침 브리핑
- 일정 리마인드
- 백업 결과 보고
- 주식 마감 브리핑
- 정형화된 자동 메시지 발송

즉,
**Marie는 예약 작업 전담 운영 봇**입니다.

이 분리를 해두면
자동화 트래픽이 main에 쌓이지 않습니다.

결과적으로,
메인 대화는 가볍고 선명하게 유지되고,
반복 작업은 더 싼 비용 구조와 규칙적인 운영 방식으로 흘러갑니다.

---

# 실제 분리 구조는 어떻게 잡았나

이번 구성의 핵심은 세 가지였습니다.

## 1. 에이전트 분리

OpenClaw 설정에 `marie` 에이전트를 별도로 추가했습니다.

이렇게 하면 메인 대화용 `main`과
반복 작업용 `marie`가 논리적으로 분리됩니다.

여기서 중요한 건,
단순히 이름만 따로 만든 게 아니라 **모델 선택까지 분리**했다는 점입니다.

메인 대화는 더 강한 모델이 필요할 수 있지만,
반복 작업은 상대적으로 가벼운 모델로도 충분한 경우가 많습니다.

이 차이가 토큰 비용 최적화에서 꽤 중요합니다.

---

## 2. Telegram 계정(봇) 분리

에이전트만 분리하면 끝나는 것 같지만,
실제로는 메시지 전달 경로도 분리해야 운영상 이점이 생깁니다.

그래서 Telegram도 이렇게 나눴습니다.

- `default` → 메인 봇
- `marie` → 예약 작업 전용 봇

이렇게 해야 자동 알림이 메인 대화창에 뒤섞이지 않습니다.

특히 실제로 써보면,
이 차이가 생각보다 큽니다.

예전에는 다음이 한 채팅방에 다 들어왔습니다.

- 사용자가 한 질문
- 제가 한 답변
- 갑자기 날아오는 자동 백업 보고
- 아침 브리핑
- 주식 마감 브리핑

이러면 인간 대화 흐름이 깨집니다.

반면 Marie로 자동화 발송 채널을 분리하면,
메인 봇은 정말 “대화”에 집중할 수 있고,
Marie는 “정기 알림” 채널로 깔끔하게 남습니다.

---

## 3. cron delivery 분리

여기서 가장 중요한 실무 포인트가 하나 있습니다.

많은 사람이 놓치는 부분인데,
**agentId를 바꾸는 것만으로는 충분하지 않습니다.**

왜냐하면 cron은 보통 두 가지가 따로 있기 때문입니다.

- 누가 실행하느냐 (`agentId`)
- 어디로 보내느냐 (`delivery`)

즉,
작업을 Marie가 실행하더라도
delivery가 여전히 기존 메인 채팅을 가리키고 있으면,
사용자 체감상 “아직 메인 봇에서 오는 것처럼” 보일 수 있습니다.

실제로 저는 이 단계에서 한 번 꼬였습니다.

겉으로는 Marie로 작업을 넘겼는데,
알림은 계속 기존 쪽으로 가거나,
일부는 Marie로 가고 일부는 main으로 가는 혼합 상태가 발생했습니다.

이 문제를 해결하려면
**cron의 실행 주체와 delivery accountId를 함께 정리해야 합니다.**

즉 정리 포인트는 다음과 같습니다.

- `agentId = marie`
- `delivery.channel = telegram`
- `delivery.to = telegram:사용자ID`
- `delivery.accountId = marie`

이렇게 맞춰야
비로소 “Marie가 실행하고 Marie bot으로 보낸다”가 완성됩니다.

# 토큰 비용 절감에 실제로 어떤 효과가 있나

이 구조의 장점은 단순한 기분 문제가 아닙니다.

실제로 비용 측면에서 유리한 이유는 명확합니다.

## 1. 반복 작업에 메인 맥락이 안 붙는다

반복 작업은 대개 짧고 구조화되어 있습니다.

예를 들어 아침 브리핑은 이런 흐름입니다.

- 날씨 조회
- 오늘 일정 조회
- 뉴스 조회
- 정해진 형식으로 출력

이 작업은 긴 대화 맥락이 필요 없습니다.

메인 세션에 붙여두면 불필요하게 긴 컨텍스트를 같이 들고 갈 가능성이 커집니다.

반면 Marie isolated cron으로 돌리면,
그때그때 필요한 작업만 수행하고 끝납니다.

이게 곧 토큰 절감으로 이어집니다.

---

## 2. 반복 작업에는 더 가벼운 모델을 배정할 수 있다

모든 작업에 같은 모델이 필요하지는 않습니다.

예를 들어 다음 작업은 비교적 가벼운 모델로도 충분합니다.

- 오늘의 날씨 요약
- 일정 알림
- 백업 성공 여부 보고
- 주식 마감 5줄 브리핑

반면 이런 작업은 더 강한 모델이 유리합니다.

- 구조 설계
- 장문의 글 작성
- 복잡한 디버깅
- 사람 의도를 섬세하게 해석해야 하는 대화

즉,
**메인 봇은 고성능 / 반복 봇은 경량화**가 가능해집니다.

이건 장기 운영에서 매우 큰 차이를 만듭니다.

---

## 3. 실패 패턴이 분리돼 디버깅 비용도 줄어든다

토큰 비용은 단순 호출량만의 문제가 아닙니다.

운영이 불안정하면,
실패를 잡기 위해 더 많은 재시도와 더 많은 조사 턴이 들어갑니다.

이번 분리 작업에서도 실제로 이런 일이 있었습니다.

- cron은 Marie로 넘겼는데 delivery는 main으로 남아 있던 문제
- 중복 LaunchAgent 때문에 main/Marie 둘 다 자동 메시지를 보내던 문제
- 아침 브리핑에서 오늘 일정 대신 내일 일정이 섞이던 문제
- Marie 주식 마감 브리핑이 “못 가져오니 어떻게 할까요?” 같은 식으로 흐르던 문제

이걸 하나씩 정리하면서 느낀 건,
**자동화는 모델보다 운영 규칙이 더 중요할 때가 많다**는 점이었습니다.

그래서 단순히 agent를 나누는 것만이 아니라,
각 자동화에 대해 **공식 데이터 소스, 형식, fallback, 금지 규칙**까지 문서화해 두는 것이 필요했습니다.

이렇게 해야 불필요한 재시도와 이상 출력이 줄고,
결국 비용도 줄어듭니다.

# 실제로 무엇을 Marie로 넘겼나

반복 작업 전용 봇으로 넘긴 대표 작업은 이런 것들입니다.

- Apple Calendar 일정 15분 전 리마인드
- 매일 06:00 아침 브리핑
- 매일 03:00 OpenClaw 경량 백업 보고
- 평일 15:45 KOSPI/KOSDAQ 마감 브리핑
- 매일 18:00 활동 보고
- 주간 OpenClaw 스냅샷
- 평일 특정 시각 Home Assistant 스위치 제어

이 작업들의 공통점은 명확합니다.

- 정해진 시간에 실행됨
- 입력 구조가 예측 가능함
- 출력 형식도 어느 정도 고정 가능함
- 자유 대화보다 작업 수행이 핵심임

즉,
**사람처럼 생각하는 메인 봇보다 작업자처럼 움직이는 봇이 더 잘 맞는 일**입니다.

# 분리만 해서는 안 되고, 운영 규칙까지 고정해야 한다

이 부분이 정말 중요합니다.

봇을 분리해도,
반복 작업이 매번 이렇게 말하면 자동화는 실패입니다.

- “현재 권한이 없습니다”
- “어떤 방식으로 진행할까요?”
- “데이터를 제공해 주시면 작성해 드립니다”

대화에서는 괜찮은 응답일 수 있습니다.

하지만 cron 자동화에서는 치명적입니다.

자동화는 물어보면 안 됩니다.
자동화는 **정해진 범위 안에서 완성된 결과를 내야 합니다.**

그래서 저는 Marie 쪽 자동화에 대해 다음 구조를 적용했습니다.

## 아침 브리핑

공식 기준 문서:
- `operations/marie-morning-briefing-spec.md`

공식 생성 스크립트:
- `daily-schedule/morning_briefing.py --print`

규칙:
- 날씨 / 오늘 일정 / 주요 뉴스만 포함
- 오늘 일정은 Apple Calendar를 오늘 00:00~내일 00:00 미만으로 조회
- 뉴스는 Google News RSS 1차 + Naver RSS 2차 보강
- 임의 추론 브리핑 금지

## 주식 마감 브리핑

공식 기준 문서:
- `operations/marie-market-close-brief-spec.md`

공식 생성 스크립트:
- `scripts/market_close_brief.py`

규칙:
- 사용자에게 되묻지 않기
- 5줄 형식 고정
- 지수는 Naver Finance 지수 페이지 기준
- 시장폭은 Naver Finance 상승/하락/보합 목록 기준
- 수급 숫자가 일부 부족해도 방향성 중심으로 브리핑 완성

이런 규칙 고정이 있어야
반복 작업 봇이 진짜로 “운영 가능한 자동화 봇”이 됩니다.

# 봇 분리 후 실제로 느낀 장점

실제로 적용해보니 장점은 생각보다 더 명확했습니다.

## 1. 메인 대화가 깨끗해진다

자동화 메시지가 main 대화에 계속 섞이면,
중요한 대화가 묻히기 쉽습니다.

Marie를 따로 두면,
main은 정말로 사람과 이야기하는 공간으로 남습니다.

이게 체감 품질을 크게 올려줍니다.

---

## 2. 자동화 실패를 더 빨리 잡을 수 있다

메인 대화와 반복 작업이 분리되면,
문제가 생겨도 어느 쪽 문제인지 빠르게 판단할 수 있습니다.

- 메인 대화 이상 → main 점검
- 정기 브리핑 이상 → Marie 점검

운영이 단순해지면,
결국 유지 비용도 줄어듭니다.

---

## 3. 작업별 최적화를 적용하기 쉬워진다

메인 봇은 메인 봇대로,
Marie는 Marie대로 규칙을 최적화할 수 있습니다.

예를 들어 Marie 쪽은
아예 “질문 금지, 정해진 출력만 전달”로 고정할 수 있습니다.

이런 최적화는 대화형 메인 봇에서는 오히려 어색하지만,
반복 작업 봇에는 매우 잘 맞습니다.

---

## 4. 비용 구조를 설명 가능하게 만든다

운영하면서 중요한 건 “왜 비용이 이렇게 나왔는지” 설명 가능한 구조를 만드는 것입니다.

봇 분리 후에는 다음처럼 해석이 쉬워집니다.

- main 비용 = 사람과의 대화 / 복잡한 작업
- Marie 비용 = cron / 정기 자동화 / 보고

이렇게 구분되면,
앞으로 어떤 자동화를 추가할 때도
“이건 어느 봇에 넣어야 하는가?”를 더 명확히 판단할 수 있습니다.

---

# 이런 구조가 특히 잘 맞는 사람

다음에 해당한다면,
OpenClaw 봇 분리 구조는 꽤 강력합니다.

## 1. 정기 자동화가 많은 사람

- 아침 브리핑
- 일정 알림
- 뉴스 요약
- 시스템 점검
- 백업 보고
- 투자/주식 브리핑

이런 작업이 많다면,
반복 작업 전용 봇을 두는 편이 훨씬 낫습니다.

## 2. Telegram이나 메신저에서 대화와 자동화가 섞여 불편한 사람

이건 정말 체감 차이가 큽니다.

메인 대화와 자동화 채널을 분리하면,
AI를 “비서”와 “자동화 데몬”으로 나눠 쓰는 느낌이 됩니다.

## 3. 토큰 비용을 체계적으로 줄이고 싶은 사람

무작정 모델만 낮추는 방식보다,
**구조를 나눠서 비용을 줄이는 방식**이 더 건강합니다.

왜냐하면 메인 대화 품질은 유지하면서,
반복 작업만 효율화할 수 있기 때문입니다.

---

# 구축하면서 얻은 가장 큰 교훈

이번 작업에서 가장 크게 느낀 건 이것입니다.

**토큰 비용 절감은 모델 선택보다 구조 설계가 먼저다.**

많은 경우 사람들은 이렇게 접근합니다.

- 더 싼 모델로 바꾸자
- 응답을 짧게 하자
- 호출 수를 줄이자

물론 이것도 의미 있습니다.

하지만 실제 운영에서는,
어떤 작업을 어떤 세션에서,
어떤 봇이,
어떤 데이터 소스로,
어떤 형식으로 처리할지를 먼저 나누는 편이 훨씬 효과적입니다.

특히 OpenClaw처럼
에이전트, 라우팅, cron, 채널 delivery를 세밀하게 나눌 수 있는 구조라면,
이 장점이 더 크게 드러납니다.

---

# OpenClaw 봇 분리를 고민하는 사람에게 추천하는 순서

만약 비슷한 구조를 만들고 싶다면,
저는 다음 순서를 추천합니다.

1. **메인 봇과 반복 작업을 구분한다**
2. 반복 작업 전용 에이전트를 만든다
3. 반복 작업 전용 Telegram bot 또는 채널을 분리한다
4. cron의 `agentId`와 `delivery`를 동시에 정리한다
5. 각 자동화에 대해 공식 스크립트와 운영 문서를 만든다
6. “질문 금지 / 형식 고정 / fallback 고정” 규칙을 넣는다

이 순서로 가면,
그냥 봇을 하나 더 만드는 것이 아니라
**운영 가능한 자동화 시스템**으로 발전시킬 수 있습니다.

---

# 마무리

처음에는 단순히 “메인 봇이 너무 바쁘다”는 느낌에서 시작했습니다.

하지만 실제로 OpenClaw 봇을 분리해보니,
이건 단순한 편의 기능이 아니라
**토큰 비용 절감과 운영 안정화를 동시에 잡는 아키텍처 개선**에 가까웠습니다.

특히 반복 작업이 많아질수록,
메인 봇 하나에 모든 걸 몰아넣는 방식은 금방 한계가 옵니다.

그럴 때 필요한 건
더 센 모델이 아니라,
더 좋은 구조일 수 있습니다.

OpenClaw를 진지하게 자동화 도구로 쓰고 있다면,
메인 봇과 예약 작업 봇 분리는 한 번쯤 꼭 검토해볼 만한 전략입니다.

그리고 직접 해보니,
이건 생각보다 훨씬 실용적이었습니다.

---

# 자주 묻는 질문

## OpenClaw 봇을 분리하면 무조건 토큰 비용이 줄어들까?

무조건이라고 말할 수는 없지만,
반복 작업이 많을수록 절감 효과가 커질 가능성이 높습니다.

핵심은 단순히 봇 수를 늘리는 것이 아니라,
**무거운 메인 대화와 가벼운 반복 작업을 서로 다른 구조로 분리하는 것**입니다.

## 메인 봇 하나로도 운영은 가능한데 굳이 나눠야 할까?

가능은 합니다.

하지만 아침 브리핑, 리마인드, 백업, 마감 보고처럼
정기 작업이 계속 쌓이면 결국 메인 대화 흐름과 비용 구조가 함께 무거워집니다.

그래서 일정 규모 이상부터는 분리하는 편이 운영상 훨씬 편합니다.

## OpenClaw 봇 분리에서 가장 중요한 설정은 무엇일까?

실제로는 `agentId` 하나보다,
**delivery까지 같이 분리하는 것**이 훨씬 중요합니다.

즉 누가 실행하느냐와 어디로 보내느냐를 동시에 정리해야,
사용자 체감상 제대로 분리된 구조가 됩니다.

## 반복 작업 봇에는 어떤 일을 넣는 게 좋을까?

다음처럼 구조화된 작업이 잘 맞습니다.

- 아침 브리핑
- 일정 리마인드
- 백업 결과 보고
- 시스템 상태 점검
- 마감 브리핑
- 반복 알림

반대로 긴 맥락 해석이나 복잡한 기획은 메인 봇 쪽이 더 잘 맞습니다.

# 마무리 한 줄 요약

OpenClaw에서 토큰 비용을 줄이는 가장 좋은 방법은,
무작정 더 싼 모델로 내리는 것보다
**메인 대화와 예약 작업을 구조적으로 분리하는 것**이었습니다.
