## 한줄 요약
오늘의 AI 뉴스는 **모델 성능 경쟁보다 에이전트 실행 신뢰성과 하니스(harness) 설계가 더 중요한 변수로 떠올랐다**는 점이다. 연구 쪽에서는 장기 절차를 정확히 따라가는 능력이 여전히 취약했고, 제품 쪽에서는 OpenAI, Google, Anthropic 모두 샌드박스, 상태 유지, 도구 오케스트레이션, 거버넌스 같은 운영 레이어를 전면에 올리고 있다.

## 오늘의 핵심 포인트
- 에이전트의 병목은 더 이상 “답을 아느냐”가 아니라 **“절차를 끝까지, 정확히, 반복 가능하게 수행하느냐”**다.
- 멀티모달 에이전트는 단순한 반응형 도구를 넘어, **상태 기반 경험 저장과 미래 상태 예측**으로 진화 중이다.
- OpenAI는 Agents SDK와 Responses API를 통해 **샌드박스·메모리·웹소켓·filesystem 툴**을 묶어 하니스를 강화하고 있다.
- Google과 Anthropic도 각각 **Managed Agents / Enterprise Agent Platform / Opus 4.7 / long-running agent harness**를 내세우며 엔터프라이즈 운영 문제를 정면으로 다룬다.

## 오늘의 논문/연구 4선

### 1) When LLMs Stop Following Steps: A Diagnostic Study of Procedural Execution in Language Models
이 논문은 “그럴듯한 추론”과 “명시된 절차를 정확히 실행하는 능력”이 전혀 다르다는 점을 정면으로 보여준다. 14개 모델과 55개 데이터셋을 비교한 결과, 5단계 절차에서는 평균 첫 답 정확도가 63%였지만 95단계에서는 20%까지 떨어졌다. 실패 양상도 단순 오답이 아니라 누락, 조기 응답, 자기수정, 과잉 단계처럼 실행 그 자체가 무너지는 형태로 나타난다.

**왜 중요하나:** 장기 에이전트는 지능보다 절차 준수가 먼저다. 코딩, 업무 자동화, 과학 워크플로 모두 여기서 막힌다.

### 2) Beyond State Machines: Executing Network Procedures with Agentic Tool-Calling Sequences
네트워크 절차를 LLM 기반 에이전트가 도구 호출 순서로 수행할 수 있는지 평가한 연구다. 핵심은 “에이전트가 절차를 직접 생각하게 둘 것인가, 아니면 도구 내부에 절차를 캡슐화할 것인가”다. 결과는 후자가 더 안정적이었다. 반복적인 에이전트 추론은 지연을 키우고 오류도 늘렸으며, 절차 길이가 길어질수록 신뢰성은 계속 떨어졌다.

**왜 중요하나:** 장기 멀티스텝 작업에서 하니스 설계가 모델 성능만큼 중요하다는 점을 다시 확인시킨다.

### 3) MuSEAgent: A Multimodal Reasoning Agent with Stateful Experiences
멀티모달 에이전트의 경험을 trajectory 단위가 아니라 **상태 단위 경험**으로 추상화해 재사용하는 접근이다. hindsight reasoning으로 고품질의 상태-행동 쌍을 뽑아 경험 은행을 만들고, 현재 의사결정 상태에 맞춰 이를 검색해 의사결정을 돕는다. 단순히 과거 로그를 뒤지는 것이 아니라, 현재 시점에 유효한 지식을 꺼내 쓰는 방식이라는 점이 핵심이다.

**왜 중요하나:** 멀티모달 에이전트의 기억은 “많이 저장하는 것”보다 “현재 상태에 맞게 재구성하는 것”이 더 중요하다.

### 4) Anticipatory Planning for Multimodal AI Agents
TraceR1이라는 2단계 강화학습 프레임워크를 통해, 멀티모달 에이전트가 미래 상태를 예측하고 그에 맞춰 행동하도록 훈련하는 논문이다. 먼저 짧은 미래 궤적을 forecast하고, 그 다음 grounded execution을 강화하는 구조라서, 반응형 실행보다 계획 일관성이 높아진다. 장기 목표를 가진 에이전트가 왜 자주 흔들리는지를 설명하는 데도 유용하다.

**왜 중요하나:** 멀티모달 에이전트의 다음 싸움은 단발성 정확도가 아니라 **예측 가능한 장기 계획 능력**이다.

## 오늘의 공식 발표 3선

### OpenAI: The next evolution of the Agents SDK
OpenAI는 Agents SDK를 더 강한 하니스로 밀고 있다. configurable memory, sandbox-aware orchestration, Codex처럼 동작하는 filesystem tools, 표준화된 통합 primitives를 넣어, 에이전트의 실행 계층을 제품의 중심으로 끌어올렸다. 메시지는 분명하다. 에이전트는 모델 하나가 아니라, 주변 실행 환경 전체로 정의돼야 한다.

### OpenAI: Speeding up agentic workflows with WebSockets in the Responses API
Responses API에 WebSocket mode를 넣어 에이전트 워크플로의 지연을 줄였다. 단순한 API 기능 추가가 아니라, 모델 추론이 빨라질수록 주변 서비스와 시스템도 같이 빨라져야 한다는 문제의식이 드러난다. 장기 실행 에이전트에서는 연결 지속성과 상태 전달 비용이 실제 사용자 경험을 좌우한다.

### Google: Introducing Managed Agents in the Gemini API
Google은 Gemini API에 managed agents를 넣고, 보안 클라우드 샌드박스에서 실행되는 에이전트를 versionable files인 AGENTS.md와 SKILL.md로 정의할 수 있게 했다. 기업 입장에서는 “모델을 부르는 API”보다 **배포, 거버넌스, 재현성, 샌드박스**가 더 중요한 주제가 됐다.

## 오늘의 해석
- 에이전트 시장의 중심은 이제 모델 벤치마크가 아니라 **하니스 설계와 운영 안정성**으로 이동했다.
- 장기 멀티스텝 작업에서는 “한 번 잘 맞추는 모델”보다 “오래 일해도 무너지지 않는 시스템”이 더 가치 있다.
- 멀티모달 에이전트는 단순한 입력-출력 변환기를 넘어, 상태 기반 메모리와 미래 예측을 결합하는 방향으로 진화 중이다.
- OpenAI, Google, Anthropic 모두 공통적으로 **sandbox + memory + tool orchestration + enterprise governance**를 밀고 있다.

## 체크포인트
1. 장기 절차형 작업에서 실제 실행 신뢰도를 어떻게 정량화할지.
2. 상태 기반 메모리가 멀티모달 에이전트의 정확도와 안정성에 얼마나 기여하는지.
3. 엔터프라이즈 에이전트에서 샌드박스와 거버넌스가 제품 차별점이 될지.
