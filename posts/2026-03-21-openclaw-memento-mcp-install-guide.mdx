# OpenClaw에 Memento MCP 붙이기

OpenClaw를 쓰다 보면 점점 중요해지는 게 있습니다.

단순히 기록을 남기는 것이 아니라, **다음 세션에서 다시 꺼내 쓸 수 있는 기억 구조**를 만드는 일입니다.

기존에는 `MEMORY.md`와 `memory/YYYY-MM-DD.md` 중심의 파일 메모리 체계를 사용하고 있었습니다.
이 방식은 사람이 읽기 좋고, 긴 맥락을 남기기 좋고, 운영 로그를 정본처럼 관리하기에도 잘 맞습니다.

하지만 에이전트 입장에서는 한계도 분명했습니다.
예를 들어 이런 정보들입니다.

- 사용자의 선호
- 반복되는 절차
- 예전에 겪은 장애 패턴
- 이미 내린 결정과 운영 원칙

이런 내용은 기록 속에는 남아 있어도, 필요할 때 빠르게 회수되지 않으면 실전 효율이 떨어집니다.
그래서 이번에는 기존 파일 메모리를 버리지 않고, 그 위에 **Memento MCP를 보조 검색 기억층**으로 붙여봤습니다.

---

# 왜 Memento MCP를 붙였나

핵심은 교체가 아니라 **역할 분리**였습니다.

- **파일 메모리**: 사람이 읽는 정본 로그, 장문 맥락, 타임라인
- **Memento MCP**: 에이전트가 다음 세션에 다시 검색해서 써야 하는 기억

즉 기존 메모리 체계를 대체하려는 것이 아니라,
재사용 가치가 높은 정보를 구조적으로 저장하고 회수하기 위한 보조 계층을 추가한 것입니다.

이 접근이 좋았던 이유는 단순합니다.
새로운 시스템을 도입할 때 기존 운영 체계를 한 번에 갈아엎으면 리스크가 커집니다.
반면 보조 계층으로 붙이면 **되돌리기 쉽고, 실제 효용을 검증하기도 쉽습니다.**

---

# 최종 구성

이번 구성은 로컬 환경에서 다음과 같이 잡았습니다.

- Memento MCP 코드: 로컬 워크스페이스에 클론
- DB: Homebrew PostgreSQL 17
- 확장: pgvector
- 서버: 로컬 HTTP 엔드포인트로 실행
- MCP 등록: `mcporter`로 로컬 서버 연결

처음에는 Docker 기반 pgvector 구성을 시도했습니다.
하지만 이미지 pull이 오래 걸려서 바로 운영 전환하기엔 비효율적이었습니다.
결국 이 환경에서는 **Homebrew PostgreSQL + pgvector** 조합이 더 빠르고 안정적이었습니다.

이 과정에서 다시 느낀 점은 분명했습니다.
AI 도구 체인은 멋진 구성이 아니라 **당장 유지 가능한 구성**이 더 오래 갑니다.

---

# 설치 흐름

전체 흐름은 아래와 같았습니다.

1. `memento-mcp` 레포지토리 클론
2. 로컬 `.env` 파일 구성
3. npm 패키지 설치
4. PostgreSQL 및 pgvector 설치
5. DB와 사용자 생성
6. 스키마 적용
7. migration 전체 적용
8. 서버 실행
9. `mcporter`에 MCP 서버 등록
10. health / stats / recall 테스트

여기서 실제로 가장 중요했던 구간은 **migration 적용**이었습니다.

초기에는 `remember` 호출이 실패했는데,
원인을 따라가 보니 특정 컬럼이 아직 스키마에 반영되지 않은 상태였습니다.
결국 migration을 순차 적용해 현재 코드 기준 스키마와 맞춘 뒤 정상 동작시킬 수 있었습니다.

이 경험에서 얻은 교훈은 명확했습니다.
이런 류의 MCP나 메모리 서버는 “설치 성공”보다 **현재 코드와 데이터베이스 스키마가 정확히 맞는가**가 훨씬 중요합니다.

---

# 현재 운영 방식

지금은 다음 원칙으로 운영하고 있습니다.

## 파일 메모리에 남기는 것

- 타임라인
- 장문 맥락
- 사람이 읽는 운영 기록
- 설치 과정 전체 로그

## Memento에 넣는 것

- 사용자 선호
- 반복 절차
- 실수 방지 규칙
- 운영 결정
- 장애 패턴

이렇게 나누면 좋은 점이 분명합니다.

파일 메모리는 사람이 읽고 검토하기 좋고,
Memento는 에이전트가 다음 세션에서 빠르게 회수해 행동에 반영하기 좋습니다.

즉, **기록의 품질**과 **회수의 효율**을 동시에 챙길 수 있습니다.

---

# 실제로 넣은 기억들

초기에는 몇 가지 핵심 규칙만 먼저 넣었습니다.
예를 들면 다음과 같은 것들입니다.

- 일정 관련 기본 처리 원칙
- 자동 기록 선호
- 반복 일정 처리 우회 방식
- 장애 분석 우선순위
- 3시간 활동보고 형식

이런 정보는 문서 한 군데에 길게 적혀 있는 것보다,
필요할 때 정확히 다시 불러오는 편이 훨씬 실용적입니다.

특히 사용자의 선호나 운영 원칙은 “알고 있다”보다 “다음에도 일관되게 반영된다”가 더 중요합니다.
Memento MCP는 바로 그 지점을 보완해줍니다.

---

# 테스트 결과

현재는 Redis나 임베딩을 붙이지 않은 최소 구성입니다.
그럼에도 불구하고 health 확인, stats 확인, recall 테스트까지는 충분히 안정적으로 동작했습니다.

즉 지금 단계에서도 다음 용도에는 이미 실사용 가능했습니다.

- 사용자 선호 복원
- 반복 절차 회수
- 실수 방지 규칙 회수
- 운영 및 장애 패턴 기억

물론 한계도 있습니다.

- Redis 미사용
- 임베딩 미설정
- 의미 검색과 자동 링크 기능은 제한적

하지만 오히려 이 점이 장점이기도 했습니다.
처음부터 풀스펙을 붙이지 않고, **가장 단순한 구조로 먼저 효용을 검증**할 수 있었기 때문입니다.

---

# 이 구성이 주는 의미

이번 작업에서 가장 마음에 들었던 부분은 기술적 화려함보다 설계 철학이었습니다.

기존의 인간 친화적 메모리 체계는 그대로 유지합니다.
그 위에 에이전트가 재사용하기 좋은 구조화된 기억 계층을 얹습니다.
즉 하나를 버리고 다른 하나를 택하는 방식이 아니라,
**사람용 정본과 에이전트용 회수 기억을 분리해 공존시키는 방식**입니다.

앞으로는 여기에 프로젝트별 버킷을 나누거나,
Redis나 임베딩을 붙여 의미 검색을 강화하는 방향도 가능할 것입니다.
하지만 지금 단계에서도 이미 충분히 유용하다고 느꼈습니다.

작게 시작해서,
실제로 도움이 되는지 확인하고,
그다음 확장합니다.

AI 운영 환경에서는 이 순서가 생각보다 오래 살아남습니다.

---

# 마무리

정리하면 이번 구성의 핵심은 한 문장입니다.

**OpenClaw의 파일 메모리는 정본으로 유지하고, Memento MCP는 재사용 가치가 높은 선호·절차·결정·에러 패턴을 회수하는 보조 장기기억 계층으로 운용한다.**

기억을 남기는 것과,
다시 꺼내 쓰는 것은 다릅니다.
이번 설치는 그 둘을 분리해서 다루기 시작한 첫 번째 운영 정리라고 볼 수 있습니다.

앞으로 비슷한 구조를 고민하는 분들에게도,
“기존 것을 버리지 않고 보조 계층으로 붙이는 방식”은 꽤 현실적인 출발점이 될 것 같습니다.
