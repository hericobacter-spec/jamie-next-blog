# 대시보드 시장 데이터 개선기: 네이버 전환, 실패, 복구, 그리고 성공

이번 작업은 단순히 “주식 카드 하나 추가” 수준이 아니었습니다.  
처음에는 미국 시장 카드를 붙이는 정도로 시작했지만, 실제로 진행해 보니 **데이터 소스 선택, 저장 구조, 서버 운영 방식, 브리지 안정성, UI 표시 방식**이 전부 연결되어 있었습니다.

결론부터 말하면, 이번 작업은 꽤 많이 돌아갔습니다.  
하지만 그 과정 덕분에 무엇이 불안정했고, 왜 구조를 바꿔야 했는지 명확하게 드러났습니다. 그리고 최종적으로는 **한국 시장과 미국 시장을 모두 네이버 기반으로 정리하는 방향**까지 도달했습니다.

이 글은 그 과정을 “무엇을 만들었다”보다 **어디서 실패했고, 무엇이 잘못됐고, 어떻게 바로잡았는지** 중심으로 정리한 기록입니다.

---

## 시작점: Yahoo와 Alpha Vantage의 한계

처음 한국 시장 데이터는 Yahoo 계열 소스가 섞여 있었고, 미국 시장은 Alpha Vantage로 붙이려 했습니다.  
겉보기에는 둘 다 빨리 붙일 수 있는 선택처럼 보였습니다.

하지만 실제 운영에서는 문제가 바로 드러났습니다.

### 한국 시장

한국 시장은 Yahoo 기반으로 갈수록 신뢰도가 떨어졌습니다.

- 종목명 검색 품질이 만족스럽지 않았고
- KOSPI / KOSDAQ 지수 값도 일관성이 떨어졌으며
- 일부 응답은 fallback으로만 떨어졌습니다.

그 결과, 한국 시장은 빠르게 **네이버 기반으로 전환**하는 쪽이 더 맞다는 결론이 났습니다.

### 미국 시장

미국 시장은 처음 Alpha Vantage를 붙였습니다.  
문제는 무료 플랜 제한이 생각보다 훨씬 강했다는 점입니다.

실제로 확인된 응답은 아래와 같은 형태였습니다.

- `Thank you for using Alpha Vantage!`
- 요청 수 제한 초과
- 지수 값 0 처리

즉, 개발 중 잠깐 테스트하는 수준은 가능했지만, **개인 대시보드에서 반복 호출되는 실사용 구조로는 부적합**했습니다.

이 시점에서 방향이 바뀌었습니다.

- 한국 시장: 네이버로 정리
- 미국 시장: 가능하면 네이버로 정리
- fallback은 유지하되, 주 소스는 네이버로 통일

---

## 중간에 겪은 가장 큰 문제: 서버가 살아 있지 않았다

사용자 입장에서는 “한국 주가도 안 나오고, 미국 주식도 안 나오고, 할 일 저장도 안 된다”는 현상이었습니다.  
처음엔 브라우저 캐시나 API 문제처럼 보였지만, 실제로 확인해 보니 핵심 원인은 훨씬 단순했습니다.

**로컬 3000 서버가 안 떠 있었습니다.**

즉:

- 대시보드가 Offline으로 보이고
- `/api/market`도 안 되고
- `/api/us-market`도 안 되고
- `/api/settings`도 죽어 있었던 것

이 경험은 꽤 중요했습니다.  
기능 문제처럼 보이는 많은 증상이 사실은 **실행 환경 문제**일 수 있다는 걸 다시 확인한 셈입니다.

이후에는 단순히 기능 수정만 하지 않고,

1. 서버가 실제로 살아 있는지
2. 3000 포트에 단일 프로세스만 있는지
3. 응답이 실제 JSON으로 오는지

를 먼저 확인하는 흐름으로 바꿨습니다.

---

## 미국 관심종목이 저장되지 않던 이유

미국 종목 입력이 안 붙는 문제는 한동안 UI 문제처럼 보였습니다.  
하지만 실제 원인은 **설정 저장 구조가 한국 주식 코드 전용으로 묶여 있었기 때문**이었습니다.

기존 구조는 사실상 이런 식이었습니다.

- 한국 관심종목: `005930`, `000660`
- 미국 관심종목: `AAPL`, `RGTI`
- 둘 다 `marketWatchCodes` 한 배열에 저장

이건 구조적으로 틀렸습니다.

왜냐하면:

- 한국은 6자리 숫자 코드
- 미국은 알파벳 티커
- 검증 규칙도 다르고
- 검색/렌더 방식도 다르기 때문입니다.

그래서 구조를 분리했습니다.

- `marketWatchCodes` → 한국 전용
- `usMarketWatchSymbols` → 미국 전용

이 수정 이후에야 미국 종목이 “추가는 되는데 사라지는 문제”를 제대로 잡을 수 있었습니다.

즉, 이번 작업에서 중요한 교훈 하나는 이겁니다.

> UI를 고치기 전에 저장 구조가 맞는지 먼저 봐야 한다.

## 네이버로 미국 지수를 옮기는 과정

처음에는 “미국 지수도 네이버에서 될까?”가 불확실했습니다.  
그런데 확인해 보니 네이버에는 해외 지수용 페이지가 따로 있었습니다.

예를 들어:

- `SPI@SPX` → S&P 500
- `DJI@DJI` → Dow Jones
- `NAS@IXIC` → Nasdaq

이 페이지들은 단순 HTML만으로도 어느 정도 값을 읽을 수 있었습니다.

하지만 여기서 또 함정이 있었습니다.

숫자를 통째로 긁으면 페이지 안의 다른 숫자들까지 같이 잡혀서,

- 값이 비정상적으로 커지거나
- 등락률이 틀리거나
- 현재값이 깨지는 문제

가 생겼습니다.

그래서 느슨한 숫자 파싱을 버리고,

- `dayTable`의 첫 행
- 상단 현재값 블록
- 방향성(`up/down`)

같이 **의미 있는 구조 단위**로 바꿔 읽는 방식으로 수정했습니다.

그 결과 미국 지수 3종은 네이버 기반으로 안정화할 수 있었습니다.

---

## 미국 개별 종목은 왜 더 어려웠는가

미국 지수는 HTML만으로도 가능했지만, 미국 개별 종목은 달랐습니다.

예를 들어:

- `AAPL.O`
- `TSLA.O`
- `RGTI.O`

같은 페이지는 존재했지만, 서버가 받아오는 HTML에는 실제 가격 데이터가 거의 없었습니다.  
페이지 본문은 사실상 `div id="root"`만 있는 구조였고, 값은 클라이언트 JS가 채우고 있었습니다.

이 시점에서 단순 HTML 파싱은 포기하고, JS 번들을 뒤졌습니다.

그 결과 발견한 것이 바로 네이버 내부 API입니다.

실제로 확인된 경로는 다음과 같습니다.

- `https://api.stock.naver.com/stock/AAPL.O/price`
- `https://api.stock.naver.com/stock/AAPL.O/overview`
- `https://api.stock.naver.com/stock/RGTI.O/price`
- `https://api.stock.naver.com/stock/RGTI.O/overview`

이 응답에는 실제로 필요한 값이 들어 있었습니다.

- `closePrice`
- `compareToPreviousClosePrice`
- `fluctuationsRatio`
- `localTradedAt`
- `openPrice`
- `highPrice`
- `lowPrice`

즉, 미국 관심종목이 “추가는 되는데 가격이 안 나오는 문제”는 여기서 해결됐습니다.

결국 미국 시장은 이렇게 정리됐습니다.

- 미국 지수 → 네이버 해외지수 페이지
- 미국 개별 종목 → 네이버 내부 stock API

이건 꽤 만족스러운 결과였습니다.  
중간에 Alpha Vantage, HTML 파싱, watchlist 구조 꼬임을 다 거쳤지만, 최종적으로는 **미국도 네이버-only 방향**으로 정리할 수 있었기 때문입니다.

## OpenClaw 브리지와 음성 입력 문제

시장 데이터만 손본 게 아니었습니다.  
대시보드 안의 OpenClaw 입력창도 같이 개선했습니다.

처음에는 이런 메시지가 보였습니다.

- `Unexpected end of JSON input`
- 브리지 폴백 응답

즉, 실시간 브리지가 불안정해서 fallback 응답으로 빠지고 있었습니다.

원인은 대체로 이랬습니다.

- `openclaw agent --local --json` 출력이 깔끔한 JSON만 주지 않음
- stdout이 비거나 잘릴 수 있음
- 전체 stdout을 무조건 `JSON.parse` 하는 건 너무 약함

그래서 브리지 파서를 강화했습니다.

- 전체 JSON 파싱 실패 시
- 줄 단위 JSON 시도
- 마지막 JSON 블록 추출 시도
- 그래도 안 되면 plain text fallback

이렇게 바꾸니 “브리지 자체가 바로 죽는 문제”는 상당 부분 줄었습니다.

그리고 음성 입력도 실험했습니다.

처음엔 Whisper 업로드형으로 넣었지만,
사용자 요구는 “버튼 누르면 대화 시작, 다시 누르면 종료”에 더 가까웠습니다.  
그래서 브라우저 음성인식 + 대화 모드 + TTS를 붙이는 쪽으로 확장했습니다.

물론 이 부분은 아직 1차 구현 수준이고, 브라우저 호환성과 대화 흐름 정리가 더 필요합니다.  
하지만 단순 업로드형보다 훨씬 “대화형 UI”에 가까운 방향으로 옮겨졌다는 점은 분명합니다.

## 이번 작업에서 가장 많이 반복한 실수

이번에 뼈아프게 반복된 실수도 있습니다.

### 1. 서버가 떠 있다고 가정한 것

실제로는 안 떠 있었는데, UI 문제처럼 계속 보던 순간들이 있었습니다.

### 2. 부분 패치를 많이 한 것

미국 시장을 처음부터 네이버-only로 설계하지 않고,

- Alpha Vantage 붙이고
- 일부만 네이버로 바꾸고
- 카드 구조를 나중에 다시 손보는 식으로 진행하면서

혼선이 커졌습니다.

### 3. 설정 구조를 늦게 분리한 것

미국과 한국 watchlist를 처음부터 나눴어야 했습니다.

### 4. 백그라운드 실행을 여러 번 겹쳐 띄운 것

한 번 살아난 서버가 다시 죽은 것처럼 보였던 이유 중 하나는,
실제로는 여러 개의 `next start`가 겹치면서 `EADDRINUSE`가 반복됐기 때문입니다.

즉, 기능 문제와 운영 문제가 서로 섞여 있었습니다.

---

## 그래도 결국 얻은 것

이번 작업은 돌아가는 길이 많았지만, 대신 얻은 것도 분명합니다.

### 1. 네이버-only 방향이 현실적으로 가능하다는 확신

- 한국 시장: 가능
- 미국 지수: 가능
- 미국 종목: 내부 API 발견으로 가능

### 2. 대시보드 데이터 모델이 더 명확해짐

- 한국 배열과 미국 배열 분리
- API 응답 구조 통일
- 카드별 책임 분리

### 3. 운영 체크리스트가 생김

이제는 기능보다 먼저 아래를 확인하게 됐습니다.

- 서버 살아 있나
- 포트 충돌 없나
- `/api/settings` 응답 오나
- `/api/market` / `/api/us-market` 값이 맞나
- 저장 구조가 UI와 일치하나

이건 단순한 디버깅 습관이 아니라, 앞으로 유사한 대시보드 작업을 훨씬 빠르게 만드는 기준이 됩니다.

---

## 이번 작업의 결론

이번 대시보드 개선 작업은 결국 단순 UI 수정이 아니었습니다.  
**데이터 소스 재설계 + 저장 구조 정리 + 실행 안정화 + 브리지 개선**까지 한 번에 건드린 작업이었습니다.

중간에는 서버가 죽고, API가 제한에 걸리고, 미국 watchlist가 저장되지 않고, 한국 지수를 숨겨놓고, 음성 버튼도 안 보이는 등 온갖 문제가 이어졌습니다.  
하지만 그 덕분에 “왜 안 되는지”가 꽤 선명하게 정리됐고, 지금은 훨씬 더 맞는 구조로 가고 있습니다.

요약하면 이렇습니다.

- Alpha Vantage는 실사용에 약했다.
- 네이버는 한국 시장에 강했고, 미국도 생각보다 깊은 데이터 경로가 있었다.
- 미국/한국 watchlist는 반드시 분리해야 한다.
- 브리지와 서버는 기능보다 먼저 살아 있어야 한다.
- 부분 패치보다 구조 정리가 결국 더 빠르다.

개발 작업에서 가장 좋은 순간은 “처음부터 한 번에 잘되는 순간”이 아니라,  
**왜 틀렸는지 이해한 뒤 구조를 바로잡아 다시 안정적으로 만드는 순간**이라고 생각합니다.

이번 작업이 딱 그런 케이스였습니다.
